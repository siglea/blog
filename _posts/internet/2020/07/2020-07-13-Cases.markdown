## 一、电商秒杀系统

秒杀是电商场景中最具挑战性的业务之一，其核心矛盾在于：极短时间内的海量请求与极少量的库存之间的冲突。一个优秀的秒杀系统需要在保证数据一致性的前提下，尽可能多地拦截无效请求。

### 1.1 流量削峰前置

秒杀的本质是"漏斗模型"——尽早在链路上游过滤掉大部分请求，避免压力传导到数据库层。

- **库存数量判断，直接走缓存，挡掉大部分请求**：可以用 Lua 脚本在 Nginx 层做库存预检，对于库存已售罄的商品直接返回，无需进入后端服务。
- **只接受拥有合法 token 的、等于库存数量的用户**：通过令牌桶机制控制进入下单流程的用户数，需要额外考虑超时未支付后的库存回补逻辑。

### 1.2 异步化与状态轮询

用户获取到秒杀资格后，下单操作通过消息队列异步处理，前端通过轮询接口查询秒杀状态及订单结果，避免同步阻塞带来的超时风险。

### 1.3 Redis 压力转移

当所有库存判断集中在单个 Redis 实例时，该实例会成为瓶颈。可以通过以下方式分散压力：

- **本地缓存**：应用服务器在本地内存中缓存一份库存标记，售罄后直接在本地拦截。
- **分片存储**：采用"商品 ID + 分片号"的方式，将同一商品的库存拆分到多个 Redis Key 中，实现水平扩展。

## 二、今日头条系统架构

今日头条是国内信息流领域的代表性产品，其系统架构需要支撑海量的内容分发与实时推荐。

### 2.1 业务规模（2015 年数据）

| 指标 | 数值 |
|------|------|
| 日活用户 | 3000 万 |
| 日 PV | 30 亿 |
| 日视频播放量（VV） | 1 亿 |
| 服务器规模 | 7000 台 |

### 2.2 技术栈

日志采集和流式处理方面使用了 Scribe、Flume、Kafka 进行数据管道建设，离线计算基于 Hadoop，实时计算基于 Storm。

### 2.3 在线存储 Abase

Abase 是头条自研的在线 KV 存储系统，关键性能指标如下：

- 读 QPS：360 万
- 写 QPS：40 万
- 平均延时：1ms
- P99 延时：4ms（99% 的请求在 4ms 内完成）

这样的性能指标充分说明了自研存储在满足业务定制化需求方面的优势。

参考：<https://blog.csdn.net/mucaoyx/article/details/84498468>

## 三、春晚摇一摇系统——100 亿次的挑战

2015 年微信红包与春晚的合作是移动互联网历史上最大规模的实时互动活动之一。

### 3.1 核心指标

| 指标 | 数值 |
|------|------|
| 峰值 QPS | 1400 万 |
| 全程互动次数 | 110 亿次 |
| 接入服务器 | 638 台 |
| 同时在线 | 14.6 亿 |
| 单机同时在线 | 200 万 |

这种规模的系统设计关键在于：接入层的水平扩展能力、核心逻辑的极致简化、以及完善的容灾降级方案。每一台接入机器能支撑 200 万同时在线，体现了协议层和网络层的深度优化。

## 四、基于 Token 的多平台身份认证架构

在多端（Web、iOS、Android、小程序）共存的场景下，统一的身份认证体系至关重要。基于 Token 的认证架构可以实现无状态的跨平台身份验证，降低服务端的 Session 管理压力。

<img src="{{ site.baseurl }}/img/system-cases/token.png" width="600px" />

核心思路是：用户登录成功后，服务端签发包含用户身份信息的 Token（通常采用 JWT 格式），客户端在后续请求中携带该 Token，服务端通过验证签名来确认用户身份，无需维护集中式的会话存储。

参考：<https://www.cnblogs.com/beer/p/6029861.html>

## 五、架构可视化——框架图的绘制方法

好的架构设计离不开清晰的可视化表达。以下是两种主流的架构图绘制方法论：

### 5.1 C4 模型

C4 模型由 Simon Brown 提出，将架构图分为四个层次，由粗到细逐级展开：

- **语境图（Context）**：系统与外部用户、系统的关系
- **容器图（Container）**：系统内的应用、数据库、消息队列等
- **组件图（Component）**：容器内部的模块划分
- **类图（Code）**：代码级别的细节

此外还有系统景观图、部署图、动态图等补充视图。

参考：<https://www.jianshu.com/p/33c6a7ed126f>

### 5.2 UML

UML 是软件工程中的经典建模语言，常用的图包括：

- **用例图**：描述系统功能与角色
- **活动图**：描述业务流程
- **序列图**：描述对象间的交互顺序
- **类图**：描述类的结构与关系
- **状态机图**：描述对象的状态迁移

参考：<https://blog.csdn.net/qq_35495763/article/details/80764914>

## 六、IM 系统的存储设计

即时通讯（IM）系统的存储设计需要在写扩散和读扩散之间做权衡，不同的业务场景适用不同的模式。

### 6.1 写扩散（推模式）—— 适用于单聊和小群

写扩散的核心思想是：消息产生后，立即将消息副本写入每一个接收者的收件箱。

**优点：**
- 消除了读模式下的 IO 集中点，每个用户都读自己的数据，高并发下锁竞争少
- 拉取消息列表（如朋友圈 Feed 流）的业务流程异常简单，速度很快
- 不需要进行大量的内存计算和网络传输，读取性能很高

**缺点：**
- 极大地消耗存储资源，消息数据会冗余存储多份。例如杨幂拥有 5000 万粉丝，她每发一条博文，消息就会冗余 5000 万份
- 新增关注、取消关注、发布 Feed 等写操作的业务流程更加复杂

### 6.2 读扩散（拉模式）—— 适用于群聊和大 V Feed

读扩散的核心思想是：消息只存一份，读取时再聚合。可以先拉取有更新的索引，然后触发服务器推送。

**优点：**
- 存储结构简单，数据存储量较小，关系数据与 Feed 数据都只存一份
- 取消关注、发布 Feed 的业务流程非常简单
- 非常适合项目早期用户量、数据量、并发量不大时的快速实现

**缺点：**
- 拉取 Feed 流列表的业务流程非常复杂
- 需要多次数据访问，进行大量的内存计算和网络传输，性能较低
- 瓶颈容易出现在"用户所发布 Feed 列表"的读取上。由于每个用户发布 Feed 的频率其实很低，架构优化的核心是通过缓存降低数据存储磁盘 IO

## 七、亿级用户中心设计

用户中心是典型的"单 KEY"类业务——以用户 ID（uid）为核心的 CRUD 操作。这一类业务都可以使用下述的分库分表架构方案。

### 7.1 数据库水平切分

常见的水平切分方式有两种：

1. **范围法**：按 uid 的范围划分，例如 0~1000 万在库 1，1000 万~2000 万在库 2
2. **哈希法**：对 uid 取模，例如 uid % 8 决定落入哪个库

### 7.2 水平切分后的查询问题

水平切分后，通过 uid 属性查询能直接定位到库，但通过非 uid 属性（如 login_name、phone）查询则无法直接定位。

非 uid 属性查询可以分为两类场景：

| 场景 | 特征 | 要求 |
|------|------|------|
| **用户侧（前台）** | 单条记录查询，访问量大 | 高可用，强一致性 |
| **运营侧（后台）** | 批量分页查询，访问量低 | 可用性要求不高，一致性要求宽松 |

### 7.3 用户侧：建立非 uid 到 uid 的映射

针对前台高频查询，有四种常见实践：

1. **索引表法**：数据库中记录 login_name 与 uid 的映射关系
2. **缓存映射法**：缓存中记录 login_name 与 uid 的映射关系
3. **生成 uid 法**：通过 login_name 生成 uid
4. **基因法**：将 login_name 基因融入 uid 的生成规则中

### 7.4 运营侧：前台与后台分离

后台运营侧的最佳实践是"前台与后台分离"：

1. 前台、后台系统 Web/Service/DB 分离解耦，避免后台低效查询引发前台查询抖动
2. 可以采用数据冗余的设计方式
3. 可以采用"外置索引"（如 Elasticsearch）或"大数据处理"（如 Hive）来满足后台复杂的查询需求

## 总结

上述案例覆盖了互联网行业中最常见的几类高并发、大数据量系统。尽管它们的业务形态各不相同，但在架构设计上有着共通的原则：

- **分层过滤**：尽早在链路上游拦截无效请求（秒杀系统）
- **读写分离**：根据读写比例选择扩散策略（IM 系统）
- **水平拆分**：通过分片突破单机瓶颈（用户中心）
- **异步解耦**：通过消息队列削峰填谷（秒杀系统）
- **缓存为王**：用多级缓存降低后端压力（几乎所有系统）

理解这些案例背后的设计思想，比记住具体的技术选型更为重要。在实际工作中，架构设计永远是在性能、一致性、可用性和开发成本之间寻找最佳平衡点。
