树是计算机科学中最重要的非线性数据结构之一。从操作系统的文件系统到数据库的索引结构，从编译器的语法树到人工智能的决策树，树无处不在。对于后端开发者而言，理解各种树结构的特性及其在数据库索引中的应用，是写出高性能系统的基础。

本文将系统介绍 AVL 树、二叉搜索树、红黑树、B 树和 B+ 树的核心特性，并重点分析为什么 MySQL 选择 B+ 树作为索引结构。

## 二叉搜索树（BST）

二叉搜索树（Binary Search Tree）是二叉树的一种，又称为二叉查找树、二叉排序树。

**核心特性**：任何一个结点的值都大于其左子树的所有结点的值，任何一个结点的值都小于其右子树的所有结点的值。

这个性质使得查找操作非常高效——每次比较都能排除一半的数据，类似二分查找。但 BST 有一个致命弱点：如果数据是有序插入的，BST 会退化为链表，查找的时间复杂度从 O(log n) 退化为 O(n)。为了解决这个问题，人们发明了各种自平衡二叉搜索树。

---

## AVL 树

AVL 树是最早被发明的自平衡二叉搜索树，以其发明者 Adelson-Velsky 和 Landis 的名字命名。

**核心特性**：

- 每个结点的**平衡因子**（左子树高度 - 右子树高度）只可能是 -1、0、1
- 如果平衡因子的绝对值超过 1，则认为是失衡，需要通过旋转操作重新平衡
- 每个结点的左右子树高度差不超过 1
- 搜索、插入、删除的时间复杂度均为 **O(log n)**

AVL 树对平衡性的要求非常严格，这保证了查找效率始终是 O(log n)，但频繁的旋转操作使得插入和删除的常数因子较大。在读多写少的场景下 AVL 树表现优异，但在需要频繁修改的场景下，红黑树是更好的选择。

---

## 红黑树（Red-Black Tree）

红黑树是一种含有红黑结点并能自平衡的二叉搜索树。它放宽了 AVL 树的平衡条件，用"近似平衡"换取更少的旋转操作，在综合场景下性能更优。

**红黑树必须满足以下五条性质**：

1. **每个结点要么是红色，要么是黑色**
2. **根结点必须是黑色**
3. **叶结点（外部结点、空结点 NIL）是黑色**
4. **红色结点不能连续**——红色结点的孩子和父亲都是黑色
5. **对于每个结点，从该点至 NIL 的任何路径都包含相同个数的黑色结点**（黑色完美平衡）

红黑树在实际工程中的应用极为广泛：

- Java 中的 `TreeMap`、`TreeSet`
- Linux 内核的 CFS 进程调度
- epoll / select 中的句柄管理
- Java 8+ 中 `HashMap` 的长链表优化

红黑树的最大优势在于**插入和删除操作最多只需 3 次旋转**，而 AVL 树在最坏情况下需要从失衡结点旋转到根结点。

---

## B 树（B-Tree）

B 树（Balanced Tree）是一种平衡的**多路搜索树**，多用于文件系统和数据库的实现。

**核心特性**：

- 一个结点可以存储**超过 2 个元素**，可以拥有**超过 2 个子结点**
- 拥有二叉搜索树的排序性质
- **完美平衡**——每个结点的所有子树高度一致
- 树的高度非常矮，极大减少了磁盘 IO 次数

B 树的设计初衷就是为了减少磁盘 IO。在内存中，二叉树的 O(log n) 查找非常快，但在磁盘存储场景下，每次访问一个结点就意味着一次磁盘 IO，而磁盘 IO 的耗时是内存访问的数万倍。B 树通过增加每个结点的元素数量来降低树的高度，从而减少 IO 次数。

---

## 数据库为什么用 B 树而不用二叉树？

这个问题的本质在于**磁盘 IO 的代价**。

二叉树（包括 AVL 树、红黑树）每个结点只存储一个元素，对于百万级的数据量，树的高度可达 20 层左右，意味着查找一条数据最多需要 20 次磁盘 IO。更关键的是，二叉树的构建过程决定了其数据在磁盘上的存储是**分散的**，缺乏局部性，每次 IO 只能读取少量有效数据。

而 B 树每个结点可以存储数百甚至上千个元素，树的高度通常只有 3-4 层，查找任意数据最多只需 3-4 次磁盘 IO。这个差距在高并发场景下是巨大的。

---

## B+ 树相对 B 树的优势

MySQL 的 InnoDB 引擎使用的是 B+ 树而非 B 树。B+ 树是 B 树的变种，主要区别在于：

- **B+ 树只有叶子结点存放数据**，其余结点仅用来索引
- **B 树的每个结点都有 Data 域**（指向实际数据的指针）

这个设计差异带来了以下优势：

### 1. B+ 树的层级更少

由于非叶子结点不存储数据，每个结点可以容纳更多的索引键值，相同数据量下 B+ 树比 B 树更矮。InnoDB 中一个数据页默认 16KB，假设索引键为 bigint（8 字节）+ 指针（6 字节），一个非叶子结点可以存放约 1170 个索引项，三层 B+ 树可以索引的数据量为 1170 × 1170 × 16 ≈ 2000 万行。

### 2. B+ 树查询速度更稳定

B+ 树的所有数据都在叶子结点上，每次查找都需要遍历到叶子层，所以每次查询的路径长度相同，查询性能非常稳定。而 B 树的数据分布在各层结点中，查询性能波动较大。

### 3. B+ 树天然具备排序功能

B+ 树的所有叶子结点构成了一个**有序链表**，在查询范围数据（如 `BETWEEN`、`ORDER BY`、`GROUP BY`）时非常高效。数据紧密排列，缓存命中率远高于 B 树。

### 4. B+ 树全表扫描更快

遍历 B+ 树只需要扫描叶子结点链表即可，而 B 树需要对每一层进行遍历。这使得 B+ 树在全表扫描场景下效率更高。

### 对比总结

| 特性 | B 树 | B+ 树 |
|------|------|-------|
| 数据存储位置 | 所有结点 | 仅叶子结点 |
| 非叶结点容量 | 较小（含数据） | 较大（仅索引） |
| 查询稳定性 | 不稳定 | 稳定 |
| 范围查询 | 需回溯 | 叶子链表直接遍历 |
| 全表扫描 | 需遍历所有层 | 只遍历叶子链表 |

---

## 总结

树形数据结构的演进路线可以总结为：

- **BST** 提供了基本的搜索树框架，但可能退化
- **AVL 树**解决了退化问题，但旋转开销大
- **红黑树**在平衡性和修改效率之间取得了最佳折中，成为内存数据结构的首选
- **B 树**将多路搜索引入，大幅降低了磁盘 IO 次数
- **B+ 树**在 B 树基础上进一步优化，通过数据下沉到叶子结点和叶子链表，完美适配了数据库索引的需求

理解这条演进路线，不仅能帮你在面试中从容应对"为什么 MySQL 用 B+ 树做索引"这类高频问题，更能让你在实际工作中做出合理的数据结构选型决策。
