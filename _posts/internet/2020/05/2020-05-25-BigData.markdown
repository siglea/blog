### 一、Hadoop 基础知识

Hadoop 是大数据生态的基石，主要由分布式文件系统 HDFS 和分布式计算框架 MapReduce 两大核心组件构成。

#### 1.1 HDFS 架构

- **NameNode**：管理文件系统元数据，核心数据结构包括 FsImage（文件系统镜像）和 EditLog（HDFS 日志文件）
- **Secondary NameNode**：辅助 NameNode 进行元数据合并，并非 NameNode 的热备
- **DataNode**：实际存储数据块的工作节点

#### 1.2 MapReduce 架构

- **JobTracker**：负责任务调度和资源管理
- **TaskTracker**：执行具体的计算任务
  - **MapTask**：执行 Map 阶段的数据处理
  - **ReduceTask**：执行 Reduce 阶段的数据汇总

#### 1.3 NameNode 高可用方案

HDFS 的 NameNode 是单点，其高可用方案经历了多个发展阶段：

1. **Secondary NameNode**：定期从 NameNode 读取 EditLog，与自己存储的 Image 合并形成新的 metadata image。这是最基础的方案，但不能做到自动故障切换。
2. **Backup NameNode**：实时获取 EditLog，当 NameNode 宕掉后需手动切换到 Backup NN。
3. **Avatar NameNode**：Facebook 提供的 HA 方案，将 client 访问 Hadoop 的 EditLog 放在 NFS 中，Standby NN 能够实时拿到 EditLog；DataNode 需要同时与 Active NN 和 Standby NN 汇报 block 信息。
4. **Hadoop HA（2.x）**：Hadoop 2.x 新增的标准特性，包括 NameNode HA 和 ResourceManager HA，基于 ZooKeeper 实现自动故障切换。

### 二、Storm 实时计算框架

Storm 是一个分布式实时计算系统，能够可靠地处理无界数据流。

#### 2.1 核心组件

- **Nimbus**：全局唯一的主控节点（相当于 Master），负责在集群里发送代码、分配工作给机器并监控状态。
- **Supervisor**：部署在每台工作机器上，监听分配给它的工作，根据需要启动/关闭工作进程 Worker。按照机器配置设定上面分配的槽位数。
- **ZooKeeper**：Storm 重点依赖的外部资源。Nimbus 和 Supervisor 甚至实际运行的 Worker 都把心跳保存在 ZooKeeper 上。Nimbus 根据 ZooKeeper 上的心跳和任务运行状况进行调度和任务分配。
- **Spout**：拓扑中的数据源组件，通常从外部数据源读取数据并转换为拓扑内部的源数据流。Spout 是一个主动角色，其 `nextTuple()` 函数会被 Storm 框架不停调用。
- **Bolt**：接收数据并执行处理的组件，可以执行过滤、函数操作、合并、写数据库等任何操作。Bolt 是一个被动角色，在接收到消息后调用 `execute(Tuple input)` 函数。
- **Topology**：Storm 中运行的实时应用程序，各组件间的消息流动形成逻辑上的拓扑结构。
- **Worker**：具体处理组件逻辑的进程。
- **Task**：处理任务的线程，不再与物理进程一一对应。
- **Stream**：源源不断传递的 Tuple 组成 Stream。
- **Tuple**：消息传递的基本单元，本质上是一个 value list（因为字段名称已事先定义好）。

#### 2.2 Storm 拓扑示例

一个 Spout 向拓扑提供数据，多个 Bolt 完成统计任务：

```java
package storm.analytics;
...
public class TopologyStarter {
    public static void main(String[] args) {
        Logger.getRootLogger().removeAllAppenders();
        TopologyBuilder builder = new TopologyBuilder();
        builder.setSpout("read-feed", new UsersNavigationSpout(),3);
        builder.setBolt("get-categ", new GetCategoryBolt(),3)
               .shuffleGrouping("read-feed");
        builder.setBolt("user-history", new UserHistoryBolt(),5)
               .fieldsGrouping("get-categ", new Fields("user"));
        builder.setBolt("product-categ-counter", new ProductCategoriesCounterBolt(),5)
               .fieldsGrouping("user-history", new Fields("product"));
        builder.setBolt("news-notifier", new NewsNotifierBolt(),5)
               .shuffleGrouping("product-categ-counter");

        Config conf = new Config();
        conf.setDebug(true);
        conf.put("redis-host",REDIS_HOST);
        conf.put("redis-port",REDIS_PORT);
        conf.put("webserver", WEBSERVER);

        LocalCluster cluster = new LocalCluster();
        cluster.submitTopology("analytics", conf, builder.createTopology());
    }
}  
```

**参考：** Storm 教程 <https://www.w3cschool.cn/storm/qk761jzb.html>

#### 2.3 Storm 命名由来

Storm（暴风雨）的组件大多以气象名词命名，这种命名方式形象而有趣：

- **Spout（龙卷）**：形象地理解为把原始数据"卷进"Storm 流式计算中
- **Bolt（雷电）**：从 Spout 或其他 Bolt 中接收数据进行处理或输出
- **Nimbus（雨云）**：主控节点，存在单点问题，但可以用 watchdog 保证可用性，fast-fail 后马上启动
- **Topology（拓扑）**：Storm 的任务单元，Spout/Bolt 是点，数据流是线，整体构成拓扑

#### 2.4 为什么 Storm 要接 Kafka

- **系统解耦**：Storm 专注计算，Kafka 负责消息存储和缓冲
- **Storm 没有自己的存储**：只负责计算，需要外部系统提供数据源
- **流量控制**：可以通过 Kafka 控制进入 Storm 的数据流量

#### 2.5 Storm 应用场景

Storm 在以下场景中表现出色：

- **日志分析**：应用系统产生大量业务日志（如网关 API 调用日志），量大且不适合直接入库，通过 Storm 进行实时加工分析。
- **大数据实时统计**：海量数据无法在数据库层面直接 SQL 统计，需要对产生的数据进行二次加工后产出结果。
- **实时分析系统**：一淘的 PORA 系统——实时分析用户属性并反馈给搜索引擎，替代了之前每天定时运行的 MR Job。
- **网站性能监控**：携程利用 HTML5 performance 标准获取指标，Storm 集群实时分析日志并入库，通过 DRPC 聚合报表并触发预警。
- **游戏数据分析**：收集游戏中的数据，运营者可在上线后几秒钟内得到持续更新的监控报告，大幅缩短游戏迭代周期。
- **推荐系统**：结合用户当前点击和历史数据生成实时推荐，如电影推荐系统中结合社交关系和点播记录进行智能推荐。

### 三、HDFS+Hive 与 HBase+Phoenix 的区别

理解 Hive 和 HBase 的区别是掌握大数据存储方案选型的关键。Hive 中的表是**纯逻辑表**（元数据），本身不存储数据，完全依赖 HDFS 和 MapReduce；而 HBase 表是**物理表**，适合存放非结构化数据。

#### 3.1 它们分别是什么

- **Apache Hive**：数据仓库，通过 HQL（类 SQL 语言）查询存放在 HDFS 上的数据，HQL 最终被转化为 MapReduce 任务。Hive 不能进行交互式查询，因为它基于 MapReduce。
- **Apache HBase**：Key/Value 数据库，基础单元是 Cell，运行在 HDFS 之上。与 Hive 不同，HBase 能够在数据库上实时运行查询。

#### 3.2 各自的特点

- **Hive**：帮助熟悉 SQL 的人运行 MapReduce 任务（JDBC 兼容）。查询会花费较长时间（默认遍历全表），但可以通过分区来优化。
- **HBase**：通过存储 Key/Value 工作，支持版本控制。适合作为数据存储层，存放会动态变化的数据。

#### 3.3 限制

- **Hive**：不支持更新操作；基于 Hadoop 批量操作，查询通常需要几分钟到几个小时；必须提供预定义 schema；与 ACID 不兼容。
- **HBase**：查询语言需要重新学习（类 SQL 功能可通过 Apache Phoenix 实现，但以提供 schema 为代价）；不兼容所有 ACID 特性；运行 HBase 必须部署 ZooKeeper。

#### 3.4 应用场景

- **Hive**：适合对一段时间内的数据进行分析查询，如计算趋势或网站日志分析。不适合实时查询。
- **HBase**：适合大数据的实时查询。Facebook 用 HBase 进行消息处理和实时分析。

#### 3.5 两者关系

Hive 和 Pig 都可以与 HBase 组合使用，为 HBase 提供高层语言支持。两者都在 Hadoop 体系中使用，数据可以从 Hive 写到 HBase，甚至再从 HBase 写回 Hive。

#### 3.6 其他

- **Pig** 是接近脚本方式去描述 MapReduce，Hive 则用 SQL。近似理解为 SQL ON Hadoop。
- **Hive 与 MySQL 数据互导**：使用 Sqoop 工具。

### 四、Hadoop、Spark 与 Storm 的对比

这三者定位不同，适用于不同的数据处理场景：

- **Hadoop**：实现了 MapReduce 思想，将数据切片计算来处理大量的**离线数据**。数据必须已存放在 HDFS 或 HBase 中，通过移动计算到数据所在的机器来提高效率。适合对实时性要求极低的场景。

- **Storm**：处理源源不断流进来的消息，处理后将结果写入存储。在**实时性**方面做得极好。可以脱离 Hadoop 体系单独使用。

- **Spark**：基于内存计算的开源集群计算系统，由 UC Berkeley AMP 实验室使用 Scala 开发。类似于 Hadoop MapReduce 的通用并行计算框架，但 Job 中间输出和结果可以保存在内存中，不再需要读写 HDFS。因此 Spark 能更好地适用于数据挖掘与机器学习等需要迭代的算法。可以简单理解为"另一种形式的 MapReduce"或第二代"引擎"，需要在 Hadoop 体系使用。

### 六、搭建单机 Hadoop

以下是在单机环境下快速搭建 Hadoop 的步骤：

```shell
# 1. 配置环境变量
export JAVA_HOME=/home/java/jdk1.8
export JRE_HOME=/home/java/jdk1.8/jre
export CLASSPATH=.:$JAVA_HOME/lib/dt.jar:$JAVA_HOME/lib/tools.jar:$JRE_HOME/lib

export HADOOP_HOME=/home/hadoop/hadoop2.8
export HADOOP_COMMON_LIB_NATIVE_DIR=$HADOOP_HOME/lib/native
export HADOOP_OPTS="-Djava.library.path=$HADOOP_HOME/lib"
export PATH=.:${JAVA_HOME}/bin:${HADOOP_HOME}/bin:$PATH

# 2. 创建目录
mkdir  /root/hadoop  
mkdir  /root/hadoop/tmp  
mkdir  /root/hadoop/var  
mkdir  /root/hadoop/dfs  
mkdir  /root/hadoop/dfs/name  
mkdir  /root/hadoop/dfs/data

# 3. 修改配置文件
vim core-site.xml

# 4. 启动
bin/hadoop  namenode  -format
start-dfs.sh
start-yarn.sh
```

```shell
disable 'test'
drop 'test'
describe 't_user'

create '<table name>','<column family>'   # 其中column family 就是列族的意思
create 'Student','Num','Name','Sex','Age' # 列族可以指定 Version 及 TTL

put <table>,<rowkey>,<family:column>,<value>,<timestamp>
put 'Student','1001','ZhangSan',male','23'
put 't1','rowkey001','f1:col1','value01' #可以指定列族及列
delete 'Student','1001','Age' # 删除1001的年龄列
delete 'Student','1001' # 删除1001所有列

scan 'Student' # 查询该表所有数据
scan 't1',{LIMIT=>5} # 扫描表t1的前5条数据
scan  'stu2',{COLUMNS => 'cf1:age', LIMMIT 10, STARTROW => 'xx'} # 分页查询

count 't1', {INTERVAL => 100, CACHE => 500} # 查询表t1中的行数，每100条显示一次，缓存区为500

get 'Student','1001' # 查看1001的数据
get 't1','rowkey001', 'f1:col1' # 查询表t1，rowkey001中的f1下的col1的值

create 'Student',{NAME=>'username',VERSIONS=>5} # 创建表的时候指定版本数
get 'Student','1001',{COLUMN=>'username',VERSIONS=>5} # 查询指定版本的数据

alter 'test1',{NAME=>'body',TTL=>'15552000'},{NAME=>'meta', TTL=>'15552000'} #指定TTL

```

### 八、HBase 深入解析

#### 8.1 二级索引方案

HBase 原生只支持基于 RowKey 的查询，二级索引是扩展查询能力的关键。主要方案包括：

**基于 Coprocessor 的方案：**

- **华为 HIndex**
- **Apache Phoenix**，其二级索引特点：
  - **Covered Indexes（覆盖索引）**：把关注的数据字段也附在索引表上，只需通过索引表就能返回所要查询的数据
  - **Functional Indexes（函数索引）**：索引不局限于列，支持任意表达式创建索引
  - **Global Indexes（全局索引）**：适用于读多写少场景，所有写操作都会引起索引更新
  - **Local Indexes（本地索引）**：适用于写多读少场景，索引数据和表数据都存储在本地
- **Lily HBase Indexer**
- **Solr/Elasticsearch**

#### 8.2 RowKey 查询设计

HBase 只能基于 RowKey 查询，不支持 WHERE 条件查询，关键看如何设计 RowKey。可以使用 RowFilter 进行过滤。

#### 8.3 HBase 高可用（CP）

HBase 在 CAP 理论中属于 CP 系统，优先保证一致性和分区容忍性。

**参考：** <https://mp.weixin.qq.com/s/Mw-r2AhuIsHwArd4xJ-_fQ>

#### 8.4 HMaster 的作用

HMaster 在功能上主要负责 Table 表和 HRegion 的管理工作：

1. Master 负责 DDL 操作（建表、删表），RegionServer 负责 DML 操作（数据读写）
2. 管理 HRegion 服务器的负载均衡，调整 HRegion 分布
3. 在 HRegion 分裂后，负责新 HRegion 的分配
4. 在 HRegion 服务器停机后，负责失效 HRegion 服务器上的 HRegion 迁移

#### 8.5 Region 寻址方式（通过 ZooKeeper + .META.）

- **第 1 步**：Client 请求 ZK 获取 .META. 所在的 RegionServer 的地址
- **第 2 步**：Client 请求 .META. 所在的 RegionServer 获取访问数据所在的 RegionServer 地址，Client 会将 .META. 的相关信息 cache 下来以便下次快速访问
- **第 3 步**：Client 请求数据所在的 RegionServer，获取所需要的数据

#### 8.6 HBase 的写入流程

1. Client 获取数据写入的 Region 所在的 RegionServer
2. 请求写 HLog（HLog 存储在 HDFS，当 RegionServer 出现异常时用于恢复数据）
3. 请求写 MemStore——只有当写 HLog 和写 MemStore 都成功了才算请求写入完成。MemStore 后续会逐渐刷到 HDFS 中。
      
#### 8.7 HBase 全局内存控制

HBase 的内存刷盘策略分为多个触发条件：

1. **全局 MemStore 上限**：当所有 MemStore 占整个 Heap 的比例达到 `hbase.regionserver.global.memstore.upperLimit`（默认 40%）时触发刷盘，flush 到 `lowerLimit`（默认 35%）时停止，以平滑系统负载。
2. **单个 MemStore 上限**：当 MemStore 大小达到 `hbase.hregion.memstore.flush.size`（默认 128MB）时触发刷盘。
3. **HLog 数量上限**：当达到 `hbase.regionserver.max.logs`（默认 32 个）时强制刷盘，避免故障恢复时间过长。
4. **手工触发**：通过 HBase Shell 或 Java API 手工触发 flush 操作。
5. **关闭 RegionServer**：正常关闭时会触发全部数据刷盘。
6. **Region 恢复后**：RegionServer 故障后，Region 迁移到其他 RegionServer 并恢复完数据后触发刷盘。

#### 8.8 HBase 中 ZooKeeper 的作用

- ZooKeeper 是 HBase HA 的解决方案和整个集群的协调器
- 通过 ZooKeeper 保证至少有一个 HMaster 处于 Active 状态
- 使用 HBase API 读取数据时，不需要知道 HMaster 或 RegionServer 的地址，只需知道 ZooKeeper 集群的地址即可
- HMaster 启动时将系统信息加载到 ZooKeeper，ZooKeeper 保存了 HBase 集群 Region 的信息、META 的信息等
- 维护着 RegionServer 的状态信息，知道哪些数据需要从哪些 RegionServer 读取
