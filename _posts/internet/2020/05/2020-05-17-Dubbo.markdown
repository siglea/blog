### 一、Dubbo 是什么

Dubbo 是一个分布式框架，其核心是远程服务调用。它的核心能力包含三大部分：

- **自动发现**：基于注册中心目录服务，使服务消费方能动态查找服务提供方，使地址透明，使服务提供方可以平滑增加或减少机器。
- **远程通讯**：提供对多种基于长连接的 NIO 框架的抽象封装，包括多种线程模型、序列化方案，以及"请求-响应"模式的信息交换方式。
- **集群容错**：提供基于接口方法的透明远程过程调用，包括多协议支持，以及软负载均衡、失败容错、地址路由、动态配置等集群支持。

---

### 二、Dubbo 的五大核心组件

- **Provider**：暴露服务的服务提供方
- **Consumer**：调用远程服务的服务消费方
- **Registry**：服务注册与发现的注册中心
- **Monitor**：统计服务调用次数和调用时间的监控中心
- **Container**：服务运行容器

### 三、Dubbo 架构分层：4 大层 10 小层

Dubbo 采用了清晰的分层架构设计，从上到下分为服务接口层、配置代理层、路由监控层和传输层。

<img src="{{ site.baseurl }}/img/dubbo1.jpg" width="600px" />

**四大层概述：**

1. **服务接口层（Service）**
2. **配置代理层**（配置层、服务代理层、服务注册层）
3. **路由监控层**（集群层、监控层）
4. **传输层**（协议层由 Invoker 转变为 Exporter、信息交换异步/同步转换、网络传输、序列化）

<img src="{{ site.baseurl }}/img/dubbo2.jpg" width="600px" />

**十小层详解：**

1. **服务接口层（Service）**：与实际业务逻辑相关，根据服务提供方和消费方的业务设计对应的接口和实现。
2. **配置层（Config）**：对外配置接口，以 `ServiceConfig` 和 `ReferenceConfig` 为中心，可以直接 new 配置类，也可以通过 Spring 解析配置生成配置类。
3. **服务代理层（Proxy）**：服务接口透明代理，生成服务的客户端 Stub 和服务器端 Skeleton，以 ServiceProxy 为中心，扩展接口为 ProxyFactory。
4. **服务注册层（Registry）**：封装服务地址的注册与发现，以服务 URL 为中心，扩展接口为 RegistryFactory、Registry 和 RegistryService。
5. **集群层（Cluster）**：封装多个提供者的路由及负载均衡，并桥接注册中心，以 Invoker 为中心，将多个服务提供方组合为一个服务提供方，对消费方透明。
6. **监控层（Monitor）**：RPC 调用次数和调用时间监控，以 Statistics 为中心。
7. **远程调用层（Protocol）**：封装 RPC 调用，以 Invocation 和 Result 为中心。Protocol 是服务域，负责 Invoker 的生命周期管理。Invoker 是 Dubbo 的核心模型，代表一个可执行体。
8. **信息交换层（Exchange）**：封装请求响应模式，同步转异步，以 Request 和 Response 为中心。
9. **网络传输层（Transport）**：抽象 Mina 和 Netty 为统一接口，以 Message 为中心。
10. **数据序列化层（Serialize）**：可复用的一些工具，扩展接口为 Serialization、ObjectInput、ObjectOutput 和 ThreadPool。

### 四、Dubbo Exporter——真正的服务发布

Invoker 经过协议层变成真正的服务暴露 Exporter：

<img src="{{ site.baseurl }}/img/exporter.jpg" width="600px" />

```shell
Exporter<?> exporter = protocol.export(wrapperInvoker)
```

Dubbo 会在 Spring 实例化完 bean 之后，在刷新容器最后一步发布 `ContextRefreshEvent` 事件时，通知实现了 `ApplicationListener` 的 `ServiceBean` 类进行回调 `onApplicationEvent` 事件方法。Dubbo 会在这个方法中调用 `ServiceBean` 父类 `ServiceConfig` 的 `export` 方法，而该方法真正实现了服务的（异步或同步）发布。

### 五、Dubbo 2.7 新特性

#### 5.1 异步化改造

Dubbo 支持多种请求调用方式：

- **oneway**：客户端发送消息后不需要接受响应，适合不关心服务端响应的场景
- **sync**：最常用的通信方式，也是默认通信方法
- **异步 future**：接收响应时 `future.get()` 会导致线程阻塞
- **异步 callback**：设置回调线程，收到响应时自动执行，不阻塞当前线程

```shell
  AsyncService.sayHello("Han Meimei");
  Future<String> fooFuture = RpcContext.getContext().getFuture();
  fooFuture.get();
  # 2.7 前后
  CompletableFuture<String> future = asyncService.sayHiAsync("Han MeiMei");
  future.whenComplete((retValue, exception) -> {
      if (exception == null) {
          System.out.println(retValue);
      } else {
          exception.printStackTrace();
      }
  });
  # Dubbo 2.7 中使用了 JDK1.8 提供的 CompletableFuture 原生接口对自身的异步化做了改进。
```

#### 5.2 三大中心改造

- **注册中心**：简化到只做服务注册
- **元数据中心**：之前生产者端注册 30+ 参数，接近一半不需要传递给注册中心。2.7 只将真正属于服务治理的数据发布到注册中心，全量元数据发布到元数据中心（支持 Redis、ZooKeeper）
- **配置中心**：2.7 正式支持配置中心（ZooKeeper、Apollo、Nacos），主要承担外部化配置的集中式存储和服务治理规则的存储与通知

#### 5.3 服务治理增强

Dubbo 不仅仅是一个 RPC 框架，更是一个服务治理框架。2.7 增加了标签路由的能力，并抽象出了应用路由和服务路由的概念。

### 六、Dubbo 常见的坑

使用 Dubbo 时需要注意以下已知问题：

- **超时重试**：默认的 Failover 策略会在超时后自动重试，对于非幂等操作可能造成重复执行
- **子类继承父类的属性，反序列化丢失**
- **自定义 Exception 的序列化问题**：如果自定义 Exception 定义在单独的 jar 包中（而非 api.jar），可能出现反序列化异常

### 十、多实现与版本管理

#### 10.1 一个服务接口有多种实现

当一个接口有多种实现时，可以用 `group` 属性来分组，服务提供方和消费方都指定同一个 group：

```xml
<dubbo:reference interface="com.xxx.MenuService" group="*" merger="true">
    <dubbo:method name="getMenuItems" merger="false" />
</dubbo:reference>
```

#### 10.2 服务上线兼容旧版本

可以用版本号（version）过渡，多个不同版本的服务注册到注册中心，版本号不同的服务相互间不引用：

```xml
<dubbo:service interface="com.foo.BarService" version="1.0.0" />
<dubbo:service interface="com.foo.BarService" version="2.0.0" />

<dubbo:reference id="barService" interface="com.foo.BarService" version="1.0.0" />
<dubbo:reference id="barService" interface="com.foo.BarService" version="2.0.0" />
<dubbo:reference id="barService" interface="com.foo.BarService" version="*" />
```

### 十二、Dubbo 四种负载均衡策略

#### 12.1 RandomLoadBalance（随机负载均衡）

Dubbo **默认**的负载均衡策略。不是完全随机，而是按照权重设置随机概率。例如两个提供者权重分别为 100 和 300，则被分配的概率分别为 25% 和 75%。

#### 12.2 RoundRobinLoadBalance（轮询负载均衡）

轮询每一个服务提供者。也有权重概念，可以严格按比例分配。缺点是较慢的机器会累积请求，可能拖慢整个系统。

```java
@Component
public class ChangeServiceImpl implements ChangeService {

    @Reference(loadbalance = "roundrobin")
    NameService nameService;

    @Override
    public String change(String name) {
        return nameService.updateName(name);
    }
}
```

#### 12.3 LeastActiveLoadBalance（最少活跃调用数负载均衡）

将请求转发至活跃调用数最少的机器。每个服务维护一个活跃数计数器，存放未处理完的请求数。活跃数相同时采取随机策略。这种方式能让慢机器收到更少请求。

#### 12.4 ConsistentHashLoadBalance（一致性哈希负载均衡）

保证相同参数的请求一定发送到同一台机器上。即使有机器崩溃，由于虚拟节点的存在，发往该机器的请求会被转发到其它机器，不会引发剧烈变动。

```xml
缺省只对第一个参数Hash，如果要修改，请配置
<dubbo:parameter key="hash.arguments" value="0,1" />
 
缺省用160份虚拟节点，如果要修改，请配置
<dubbo:parameter key="hash.nodes" value="320" />
```

**参考：** <http://dubbo.apache.org/zh-cn/blog/dubbo-loadbalance.html>

### 十三、Dubbo 集群容错

Dubbo 内置了以下容错策略：

- **Failover（失败自动切换）**：默认策略。调用失败时根据重试次数从其他地址重试。副作用是增加开销和响应时间，某些情况下可能造成资源浪费。
- **Failsafe（失败安全）**：调用失败时忽略错误并记录日志，返回空结果。适用于旁路系统，如写入审计日志。
- **Failfast（快速失败）**：调用失败立即报错。适用于非幂等操作。
- **Failback（失败自动恢复）**：调用失败时返回空结果（类似 Failsafe），但会将失败调用加入内存列表，在另一个线程中异步重试。适合实时性要求不高且不需要返回值的异步操作。
- **Forking（并行调用）**：第一次调用就同时发起多个，只要其中一个成功就认为成功。适合资源充足且对失败容忍度低的场景。
- **Broadcast（广播调用）**：逐个调用所有提供者，任意一个出错则认为此次调用出错。适用于通知所有提供者更新缓存或日志等场景。

### 十四、Dubbo 异步转同步

Dubbo 默认的同步调用本质上是基于异步通信实现的：

1. 调用方请求远程服务之后阻塞等待结果
2. 远程服务返回结果后唤醒请求线程

```java
public Object get(int timeout) throws RemotingException {
        if (timeout <= 0) {
            timeout = Constants.DEFAULT_TIMEOUT;
        }
        // isDone()方法就是判断Response是否有值(即是否有返回结果)
        if (!isDone()) {
            long start = System.currentTimeMillis();
            lock.lock();
            try {
                while (!isDone()) {
                    // 超时等待
                    done.await(timeout, TimeUnit.MILLISECONDS);
                    // 如果有返回结果了，或者，超时了，就退出循环
                    if (isDone() || System.currentTimeMillis() - start > timeout) {
                        break;
                    }
                }
            } catch (InterruptedException e) {
                throw new RuntimeException(e);
            } finally {
                lock.unlock();
            }
            // 如果是超时了，就抛出异常
            if (!isDone()) {
                throw new TimeoutException(sent > 0, channel, getTimeoutMessage(false));
            }
        }
        // 远程服务正常返回结果，则返回给调用方
        return returnFromResponse();
}
private void doReceived(Response res) {
        lock.lock();
        try {
            response = res;
            if (done != null) {
                // 唤醒阻塞的线程
                done.signal();
            }
        } finally {
            lock.unlock();
        }
        if (callback != null) {
            invokeCallback(callback);
        }
    }
```

这段代码清晰地展示了 Dubbo 如何通过 `Condition` 的 `await/signal` 机制将异步通信转化为同步调用。

### 十六、路由、负载均衡与集群容错的区别

这三个概念容易混淆，它们都描述了如何从多个 Provider 中选择一个进行调用，但发生在 RPC 调用的不同阶段。

以一个实际例子说明：一个 Dubbo 用户服务在北京部署了 10 个，在上海部署了 20 个。一个杭州的消费方发起调用：

1. **路由**：根据配置的路由规则，杭州发起的调用路由到较近的上海的 20 个 Provider
2. **负载均衡**：根据配置的随机负载均衡策略，在 20 个 Provider 中随机选择第 7 个
3. 调用第 7 个 Provider **失败**
4. **集群容错**：根据配置的 Failover 模式，重试其他服务器
5. 重试第 13 个 Provider，调用成功

**调用顺序**：先路由（从全集中筛选子集），再负载均衡（从子集中选一个），最后在调用失败时触发集群容错。
