Web 安全是每个后端工程师都必须重视的领域。随着互联网应用的普及，攻击手段也在不断演进，从暴力的流量攻击到精巧的逻辑漏洞利用，每一种攻击都可能给业务带来严重损失。本文将系统梳理几种最常见的 Web 安全攻击类型，剖析其原理并探讨对应的防御策略。

---

## DDoS：分布式拒绝服务攻击

**DDoS（Distributed Denial of Service，分布式拒绝服务）** 是最具破坏力的网络攻击之一。其核心原理是利用大量合法的分布式服务器对目标发送请求，从而导致正常合法用户无法获得服务。通俗地讲，就是利用网络节点资源（如 IDC 服务器、个人 PC、手机、智能设备、打印机、摄像头等）对目标发起大量攻击请求，导致服务器拥塞而无法对外提供正常服务。

### 攻击分类

DDoS 并非单一的攻击方式，按照攻击目标和手法可以分为以下几类：

- **资源消耗类攻击**：SYN Flood、ACK Flood、UDP Flood 等。这类攻击通过发送大量伪造的网络协议包，消耗服务器的连接资源和带宽。防御上通常采用限流和唯一标识限制。

- **服务消耗性攻击**：针对服务的特点进行精确定点打击，如 Web 的 CC 攻击、数据服务的复杂检索、文件服务的大文件下载等。这类攻击不是为了拥塞流量通道或协议处理通道，而是让服务端始终处于高消耗型业务的忙碌状态。

- **反射类攻击**：利用第三方服务的放大效应，一般响应流量远远大于请求本身流量。攻击者伪造源 IP 向第三方服务发送请求，放大后的响应被导向受害者。

- **CC 攻击（Challenge Collapsar，挑战黑洞）**：通过代理服务器或者大量肉鸡模拟多个用户访问目标网站的动态页面，制造大量后台数据库查询动作，消耗目标 CPU 资源，造成拒绝服务。CC 攻击不像传统 DDoS 可以用硬件防火墙过滤，因为其请求本身是正常的 HTTP 请求，难以通过协议层面区分。

### 防御手段

- **资源隔离**：将核心业务与非核心业务分离部署，避免攻击影响全局
- **应用层用户规则**：基于请求频率、行为特征等维度进行识别和拦截
- **数据分析**：通过流量基线分析异常模式，结合机器学习进行智能识别
- **资源对抗与流量清洗**：隐藏源站 IP 后将攻击流量引流到高防 IP，由专业的流量清洗设备过滤恶意流量

## XSS：跨站脚本攻击

**XSS（Cross Site Scripting，跨站脚本）** 是一种通过向网页注入恶意脚本来窃取用户数据的攻击方式。之所以缩写为 XSS 而非 CSS，是为了避免与层叠样式表混淆。

### 攻击原理

攻击者通过某种手段（如评论区、URL 参数、表单输入等）将恶意 JavaScript 代码注入到正常网站的页面中。当其他用户访问该页面时，恶意脚本在用户浏览器中执行，可以通过 `document.cookie` 等 API 读取用户的敏感信息，然后通过动态创建 `<img>` 或 `<script>` 标签，将数据以 `.src` 的方式发送到攻击者的服务器。

XSS 攻击通常分为三种类型：
- **存储型 XSS**：恶意脚本被持久化到数据库，所有访问该页面的用户都会中招
- **反射型 XSS**：恶意脚本在 URL 参数中，通过诱导用户点击特制链接触发
- **DOM 型 XSS**：通过修改页面 DOM 结构触发，不经过服务端

### 防御手段

- **Cookie 加 HttpOnly 标志**：禁止 JavaScript 读取 Cookie，即使存在 XSS 漏洞也无法窃取会话信息
- **服务端输入过滤**：对用户输入进行严格过滤，移除或转义 `<script>` 等危险标签
- **前端输出编码**：在页面渲染时对动态内容进行 HTML 实体编码，防止脚本执行
- **Content Security Policy（CSP）**：通过 HTTP 响应头限制页面可以加载的资源来源

## CSRF：跨站请求伪造

**CSRF（Cross Site Request Forgery，跨站请求伪造）** 是一种利用用户已认证的会话发起非法操作的攻击。与 XSS 窃取数据不同，CSRF 的核心是伪造合法请求。

### 攻击原理

假设用户在 A 网站登录成功并保持了本地 Cookie，此时用户在浏览器中访问了攻击者的 B 网站。B 网站读取 A 站的 Cookie 并伪造一个指向 A 站的请求（如转账、修改密码等），由于浏览器会自动携带目标域的 Cookie，A 站服务器无法区分这个请求是用户主动发起还是被诱导发起的。

这种攻击之所以能生效，涉及浏览器安全策略的几个特点：

- 同源策略阻止了 JavaScript 读取其他网站的 Cookie 等内容，但没有防止 JavaScript 向其他网站发出请求
- 同源策略限制了 AJAX 跨域请求，但没有限制 `<form>` 表单提交和 `<img>` 标签加载
- 浏览器在向目标网站发起请求时会自动携带该网站下发的 Cookie

### 防御手段

- **Referer 验证**：检查 HTTP 请求头中的 Referer 字段，判断请求来源页面是否合法。但由于某些浏览器版本中 Referer 头存在被篡改的可能性，这个方案并不完善。

- **CSRF Token**：目标网站通过 Cookie 生成 Hash 值（Token），在表单提交时必须带上该 Token，服务端验证 Token 的合法性。由于攻击者无法读取目标站的 Cookie 内容，也就无法构造出正确的 Token。

- **验证码**：表单提交时要求输入图片验证码等人机校验手段，增加伪造请求的难度。

- **双重 Cookie 验证**：用户登录时服务端生成 Token 写入 Cookie，之后用户请求必须从 Cookie 中读出 Token 并放到请求参数中，服务端比对两者是否一致。

> 参考：[CSRF 攻击详解](https://www.jianshu.com/p/2de97b609a44)

## SQL 注入

SQL 注入是最经典的 Web 安全漏洞之一。攻击者通过在输入字段中构造恶意 SQL 语句，使其被拼接到后端的数据库查询中执行，从而实现非授权的数据读取、修改甚至删除。

### 典型场景

假设后端查询语句为：

```sql
SELECT * FROM users WHERE username = '输入值' AND password = '输入值';
```

如果攻击者在用户名字段输入 `' OR 1=1 --`，最终执行的 SQL 变为：

```sql
SELECT * FROM users WHERE username = '' OR 1=1 --' AND password = '';
```

`OR 1=1` 使条件永远为真，`--` 注释掉后面的条件，攻击者无需密码即可登录。

### 防御手段

- **参数化查询（PreparedStatement）**：这是最根本的防御手段，将 SQL 语句和参数分离，数据库引擎会将参数作为值处理而非 SQL 片段
- **ORM 框架**：使用 MyBatis、Hibernate 等 ORM 框架自带的参数绑定机制
- **输入校验**：对用户输入进行白名单校验，限制特殊字符
- **最小权限原则**：数据库连接账号只授予业务所需的最小权限

> 参考：[SQL 注入详解](https://www.cnblogs.com/myseries/p/10821372.html)

## 其他安全攻防

除了上述四种常见攻击外，Web 安全领域还有许多值得关注的攻击方式，包括但不限于：

- **点击劫持（Clickjacking）**：通过透明 iframe 覆盖在正常页面上，诱导用户点击
- **文件上传漏洞**：上传恶意脚本文件并执行
- **URL 跳转漏洞**：利用开放重定向进行钓鱼

> 参考：[常见六大 Web 安全攻防解析](https://www.cnblogs.com/fundebug/p/details-about-6-web-security.html)

## 总结

Web 安全是一个攻防不断博弈的领域。作为技术人员，我们需要：

1. **深入理解攻击原理**：只有知道攻击如何发生，才能设计出有效的防御方案
2. **纵深防御**：不依赖单一防御手段，而是在网络层、应用层、数据层构建多层防护
3. **安全意识前置**：在系统设计和编码阶段就考虑安全性，而不是事后补救
4. **持续关注**：安全领域日新月异，定期了解最新的漏洞和攻击手法，及时更新防御策略

安全无小事，每一个看似微不足道的漏洞都可能成为攻击者的突破口。
