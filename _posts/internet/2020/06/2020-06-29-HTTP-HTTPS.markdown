HTTP 和 HTTPS 是互联网通信的基石。理解它们的工作原理、演进历史和安全机制，对于每一个后端工程师都至关重要。本文将从 HTTP 协议的版本演进讲起，深入剖析 SSL/TLS 握手流程、HTTPS 的加密原理，以及中间人攻击的攻防之道。

---

## 一、HTTP 协议版本演进

HTTP 协议经历了 30 年的演进，从最初只能传输简单文本，到如今支持多路复用和服务端推送：

### 1.1 HTTP/0.9（1991）

HTTP 的最初版本，功能极为简陋：
- 只支持 GET 方法
- 没有协议头（Header）
- 只能传输 HTML 文本，不支持图片等其他资源
- 连接用完立即关闭
- 没有错误码机制

### 1.2 HTTP/1.0（1992-1996）

引入了现代 HTTP 协议的基本框架：
- 请求与响应支持头域（Header）
- 响应以状态行（Status Line）开始
- 响应内容不再限于超文本，支持多种媒体类型
- 支持 GET、HEAD、POST 方法
- 支持长连接（但默认使用短连接）、缓存机制和身份认证

### 1.3 HTTP/1.1（1997-1999）

至今仍是使用最广泛的版本，引入了多项重要特性：
- **Keep-Alive 持久连接**：默认开启，复用 TCP 连接
- **Chunked 分块传输编码**：支持流式传输
- **字节范围请求**：支持断点续传
- **请求流水线（Pipelining）**：可同时发送多个请求
- **Host 头域**：支持虚拟主机
- 新增 OPTIONS、PUT、DELETE、TRACE、CONNECT 方法
- 引入实体标签（ETag）和更强大的 Cache-Control 头

### 1.4 HTTP/2.0（2012-2014）

基于 Google 的 SPDY 协议发展而来，性能大幅提升：
- **多路复用**：基于二进制分帧，在单一连接上并行处理多个请求/响应
- **头部压缩（HPACK）**：减少头部传输的数据量
- **随时复位**：可以在任意时间点取消某个流
- **Server Push**：服务端可以主动推送资源
- **优先级和依赖**：可以设置请求的优先级

### 1.5 SPDY 协议

2009 年，Google 公开了自研的 SPDY 协议，主要解决 HTTP/1.1 效率不高的问题。该协议在 Chrome 浏览器上验证可行后，成为 HTTP/2 的设计基础。

## 二、SSL/TLS 协议

### 2.1 SSL/TLS 发展历史

| 时间 | 事件 |
|------|------|
| 1994 | NetScape 设计 SSL 1.0，未发布 |
| 1995 | SSL 2.0 发布，很快发现严重漏洞 |
| 1996 | SSL 3.0 问世，得到大规模应用 |
| 1999 | ISOC 发布 SSL 升级版 TLS 1.0 |
| 2006 | TLS 1.1 |
| 2008 | TLS 1.2 |

TLS 1.0 通常被标示为 SSL 3.1，TLS 1.1 为 SSL 3.2，TLS 1.2 为 SSL 3.3。目前应用最广泛的是 TLS 1.2+。

### 2.2 SSL/TLS 简介

- **SSL（Secure Socket Layer，安全套接字层）**：位于传输层和应用层之间，通过互相认证、数字签名确保完整性、加密确保私密性，实现安全通讯。由 SSL 记录协议和 SSL 握手协议两层组成。
- **TLS（Transport Layer Security，传输层安全协议）**：SSL 3.0 的后续版本，建立在 SSL 3.0 规范之上。两者差别极小，可以理解为 SSL 3.1。

## 三、SSL/TLS 四次握手详解

SSL/TLS 握手的核心目的是生成**"对话密钥"**，整个过程包含 4 次握手：

### 第一次握手：客户端 → 服务器

客户端发送以下信息：
- 随机数 1（Client Random）
- 支持的 TLS 版本号
- 支持的加密方式（如 RSA）
- 支持的压缩方式

### 第二次握手：服务器 → 客户端

服务器回应以下信息：
- 随机数 2（Server Random）
- 确认的 TLS 版本号
- 确认的加密方式
- 确认的压缩方式
- 服务器证书（包含公钥）

### 第三次握手：客户端 → 服务器

客户端完成以下操作：
- 生成随机数 3（Pre-Master Secret），用服务器公钥加密后发送
- 通知服务器编码方式改变，后续通信使用加密方式
- 握手结束通知

### 第四次握手：服务器 → 客户端

服务器收到随机数 3 后：
- 通知客户端编码方式改变
- 握手结束通知

**关键要点**：通信阶段的会话密钥由 3 个随机数共同生成——客户端产生 2 个（随机数 1 和随机数 3），服务端产生 1 个（随机数 2）。之后双方使用这个"对话密钥"进行对称加密通信。

## 四、公钥与私钥

理解 HTTPS 加密机制的基础是非对称加密的特性：

- 公钥加密的数据，**只有私钥能解密**
- 私钥加密的数据，**只有公钥能解密**
- 公钥加密的数据，**公钥自身无法解密**
- 私钥加密的数据，**私钥自身也无法解密**

这种特性保证了：即使攻击者截获了公钥和密文，也无法解密数据。

## 五、HTTPS 加密原理

HTTPS 的安全性依赖 CA（Certificate Authority）证书体系。完整流程如下：

1. **预置信任根**：操作系统或浏览器提前内置了合法 CA 机构的公钥
2. **服务器申请证书**：服务器向 CA 机构提交相关信息（域名、组织信息等）
3. **CA 签发证书**：CA 生成服务器对应的私钥 + 证书（包含服务器公钥、数字签名等）
4. **客户端请求**：客户端发起 HTTPS 请求，服务器返回 CA 签发的证书
5. **验证证书**：客户端用内置的 CA 公钥解密证书，验证合法性
6. **建立安全通道**：验证通过后，从证书中提取服务器公钥，开始 SSL/TLS 握手

### 5.1 证书验证的具体过程

客户端收到证书后：
- 使用本地 CA 公钥解密证书，得到服务器公钥和数字签名
- 数字签名经 CA 公钥解密得到证书信息摘要
- 用证书签名的方法重新计算当前证书的信息摘要，与解密得到的摘要对比
- 如果一致，证明证书确实是服务器下发的，没有被篡改

### 5.2 为什么中间人无法篡改证书

中间人虽然拥有 CA 的公钥（公钥是公开的），能够解析证书内容，但**篡改后无法重新加密**——因为中间人没有 CA 的私钥。强行修改只会导致证书内容和数字签名不匹配，客户端验证失败。

### 5.3 客户端验证项目

- 证书是否过期
- 签发证书的 CA 是否可靠
- 本地 CA 公钥能否正确解开证书
- 证书上的域名与实际访问的域名是否匹配

### 5.4 攻击者能否伪造证书

答案是不能。当攻击者去 CA 申请证书时，CA 会要求其提供域名 WHOIS 信息、域名管理邮箱等证明，以验证申请者是域名的合法拥有者。攻击者无法提供这些信息，因此无法获得针对目标域名的合法证书。

## 六、中间人攻击（MITM）

中间人攻击（Man-in-the-Middle Attack）是指攻击者拦截并篡改网络中的通信数据。**理解清楚中间人攻击，也就理解了 HTTPS 存在的意义。**

中间人攻击分为两种类型：
- **被动 MITM**：只窃听通信数据而不修改
- **主动 MITM**：不但窃听数据，还会篡改通信内容

最常见的中间人攻击发生在公共 WiFi 或公共路由器上。

### 6.1 SSLStrip 降级攻击

SSLStrip 的原理是将 HTTPS 降级为 HTTP。这种攻击利用了很多网站并没有关闭 HTTP 访问，而是在服务端将首页从 HTTP 跳转到 HTTPS，给了攻击者一个 HTTP 响应作为突破口。

**防御方式**：启用 **HSTS（HTTP Strict Transport Security）** 和 **Preload List**，让浏览器强制使用 HTTPS，攻击者找不到任何 HTTP 的机会。

### 6.2 SSLSplit 解密攻击

SSLSplit 利用的是某些 SDK 未进行证书校验的漏洞。攻击者可以用自签名证书冒充服务器。

**防御方式**：
- 修复 SDK，正确校验证书并给出安全提示
- App 内置服务器证书（Certificate Pinning）
- 远离不可信的公共网络

### 6.3 如何避免中间人攻击

- 客户端不要轻易信任不受信的证书
- App 可以提前预埋证书在本地（证书锁定）
- 始终使用 HTTPS，启用 HSTS
- 避免在公共 WiFi 下进行敏感操作
