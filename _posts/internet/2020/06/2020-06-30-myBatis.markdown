### 引言

在 Java 持久层框架的世界里，MyBatis 凭借其灵活的 SQL 映射和轻量级的设计，成为国内互联网公司最广泛使用的 ORM 框架之一。与 Hibernate 的全自动化不同，MyBatis 被称为"半自动化"的 ORM 框架——它不会帮你生成 SQL，但会帮你优雅地管理 SQL 的输入输出。这种设计哲学让开发者既能享受框架带来的便利，又能完全掌控 SQL 的执行。

### MyBatis 框架概述

MyBatis 是一个支持定制化 SQL、存储过程以及高级映射的持久层框架。它主要完成两件核心工作：

1. **封装 JDBC 操作**：消除了几乎所有的 JDBC 样板代码——手动设置参数、获取结果集等繁琐操作
2. **打通 Java 类与 SQL 语句之间的映射**：利用反射和 XML/注解配置，实现 Java 对象与数据库记录的自动转换

MyBatis 的核心竞争力在于：**让你方便地写出 SQL，方便地获取 SQL 的执行结果。**

### 核心架构与执行流程

MyBatis 的执行流程如下：

```
应用代码 → SqlSession → Executor → StatementHandler → JDBC
    ↑                                                    │
    └────────── ResultSetHandler ← ParameterHandler ←────┘
```

**关键组件说明**：

| 组件 | 职责 |
|------|------|
| **SqlSessionFactory** | 创建 SqlSession 的工厂，应用全局唯一 |
| **SqlSession** | 与数据库交互的会话，提供增删改查方法，非线程安全 |
| **Executor** | 执行器，负责 SQL 的生成和查询缓存的维护 |
| **MappedStatement** | 封装了一条 SQL 语句的所有信息（SQL、参数映射、结果映射） |
| **StatementHandler** | 负责与 JDBC Statement 交互 |
| **ParameterHandler** | 负责将 Java 参数设置到 SQL 语句中 |
| **ResultSetHandler** | 负责将 JDBC ResultSet 映射为 Java 对象 |

### #{} 和 ${} 的区别

这是 MyBatis 中最常被问到的问题，也是最容易引发安全漏洞的知识点。

#### #{} — 预编译参数（推荐）

MyBatis 在处理 `#{}` 时，会将 SQL 中的 `#{}` 替换为 `?` 占位符，然后调用 `PreparedStatement` 的 `set` 方法来赋值。

```xml
<select id="findUser" resultType="User">
    SELECT * FROM user WHERE id = #{id} AND name = #{name}
</select>
```

实际执行的 SQL：
```sql
SELECT * FROM user WHERE id = ? AND name = ?
-- 参数通过 PreparedStatement 安全地绑定
```

**优点**：能有效防止 SQL 注入，是绝大多数场景下的首选。

#### ${} — 字符串替换（慎用）

MyBatis 在处理 `${}` 时，直接将变量的值拼接到 SQL 字符串中。

```xml
<select id="findUser" resultType="User">
    SELECT * FROM user ORDER BY ${columnName}
</select>
```

如果 `columnName = "name"`，实际执行的 SQL：
```sql
SELECT * FROM user ORDER BY name
```

**风险**：如果输入未经校验，可能导致 SQL 注入。例如 `columnName = "name; DROP TABLE user;--"` 将产生灾难性后果。

**使用场景**：只在需要动态拼接表名、列名、排序字段等**不能使用占位符**的场景下使用 `${}`，且必须对输入进行严格校验。

```xml
<!-- 动态表名，只能用 ${} -->
<select id="findAll" resultType="Map">
    SELECT * FROM ${tableName} WHERE status = #{status}
</select>
```

### IBatis 与 MyBatis 的演进

MyBatis 的前身是 Apache iBATIS，两者的主要差异：

| 对比项 | iBATIS | MyBatis |
|--------|--------|---------|
| 核心处理类 | SqlMapClient | **SqlSession** |
| 变量语法 | `#变量#`、`$变量$` | `#{变量}`、`${变量}` |
| 节点属性 | class | **type** |
| 查询方法 | queryForObject、queryForList | **selectOne、selectList** |
| 别名配置 | 映射文件中 | **核心配置文件中** |

MyBatis 3.x 在 iBATIS 的基础上进行了大量改进，API 更简洁，支持注解配置，插件机制更灵活。

### 延迟加载机制

MyBatis 支持对关联对象的延迟加载（Lazy Loading），避免在查询主对象时就执行所有关联查询。

#### 支持的关联类型

- **association**：一对一关联（如 User 的 Department）
- **collection**：一对多关联（如 User 的 OrderList）

#### 配置方式

```xml
<settings>
    <setting name="lazyLoadingEnabled" value="true"/>
    <setting name="aggressiveLazyLoading" value="false"/>
</settings>
```

#### 实现原理

延迟加载的底层通过 **CGLIB 动态代理** 实现：

1. MyBatis 创建目标对象的代理对象
2. 当调用 `a.getB()` 时，代理拦截器发现关联属性 `b` 为 null
3. 拦截器自动执行事先保存的关联查询 SQL，获取对象 B
4. 调用 `a.setB(b)` 设置关联属性
5. 后续对 `a.getB().getName()` 的调用就能正常返回

```java
User user = sqlSession.selectOne("getUser", 1);
// 此时只执行了查询 User 的 SQL

Department dept = user.getDepartment();
// 此时才执行查询 Department 的 SQL（延迟加载触发）
```

#### 注意事项

- 延迟加载依赖 SqlSession，如果 Session 已关闭再访问延迟属性会报 `LazyInitializationException`
- 序列化场景下需要特别处理，因为代理对象可能无法正常序列化
- 在 Spring 中可以通过 `OpenSessionInViewFilter` 延长 Session 生命周期

### MyBatis 缓存机制

MyBatis 提供两级缓存来提升查询性能：

#### 一级缓存（本地缓存）

- 作用域：**SqlSession 级别**，默认开启
- 同一个 SqlSession 中，相同的查询只会执行一次 SQL
- 执行 insert/update/delete 操作后自动清空

```java
User user1 = sqlSession.selectOne("getUser", 1);  // 执行 SQL
User user2 = sqlSession.selectOne("getUser", 1);  // 命中缓存，不执行 SQL
// user1 == user2  → true
```

#### 二级缓存（全局缓存）

- 作用域：**Mapper（namespace）级别**，需手动开启
- 多个 SqlSession 可以共享同一个 Mapper 的二级缓存
- 需要实体类实现 Serializable 接口

```xml
<!-- 在 Mapper XML 中开启二级缓存 -->
<cache
    eviction="LRU"
    flushInterval="60000"
    size="512"
    readOnly="true"/>
```

**生产建议**：二级缓存在多表关联查询、分布式环境下容易出现数据不一致问题，建议谨慎使用。对于需要缓存的场景，推荐使用 Redis 等外部缓存。

### MyBatis 插件机制

MyBatis 允许在 SQL 执行的四个阶段进行拦截：

```java
@Intercepts({
    @Signature(type = Executor.class, method = "query", 
               args = {MappedStatement.class, Object.class, 
                       RowBounds.class, ResultHandler.class})
})
public class SlowQueryPlugin implements Interceptor {
    @Override
    public Object intercept(Invocation invocation) throws Throwable {
        long start = System.currentTimeMillis();
        Object result = invocation.proceed();
        long elapsed = System.currentTimeMillis() - start;
        if (elapsed > 1000) {
            log.warn("慢查询：{}ms, SQL: {}", elapsed,
                ((MappedStatement) invocation.getArgs()[0]).getId());
        }
        return result;
    }
}
```

常见的插件应用：
- **分页插件**：PageHelper，自动在 SQL 中添加分页语句
- **慢查询监控**：记录执行时间超过阈值的 SQL
- **SQL 审计**：记录所有执行的 SQL 和参数

### 总结

MyBatis 的设计哲学是"SQL 的事交给开发者，其他的框架来做"。它不像 Hibernate 那样隐藏 SQL，而是让开发者完全掌控 SQL 的编写和优化，同时提供了强大的映射和插件机制来减少样板代码。理解 `#{}` 与 `${}` 的区别是安全编码的基本要求，掌握缓存和延迟加载机制能帮助你写出更高效的数据访问代码。在实际项目中，MyBatis 搭配 PageHelper 分页插件和 MyBatis-Plus 增强工具，能够覆盖绝大多数的持久层开发需求。
