Java 的类加载机制是 JVM 运行时体系中最核心的环节之一。每一个 Java 类在被使用之前，都必须经过类加载器（ClassLoader）加载到 JVM 中。理解类加载的过程和原理，不仅有助于排查 `ClassNotFoundException`、`NoClassDefFoundError` 等常见问题，更是理解 Java 模块化、热部署、SPI 机制等高级特性的基础。

---

## Java 类加载的完整生命周期

一个 Java 类从被加载到 JVM 内存开始，到被卸载出内存为止，经历以下七个阶段：

### 1. 加载（Loading）

加载是类加载的第一个阶段，在这个阶段完成三件事情：

- 通过一个类的全限定名获取该类的二进制字节流（可以来自 .class 文件、JAR 包、网络、动态代理生成等）
- 将该二进制流中的静态存储结构转化为方法区的运行时数据结构
- 在内存中生成该类的 `java.lang.Class` 对象，作为该类的数据访问入口

### 2. 验证（Verification）

验证的目的是确保 Class 文件的字节流中的信息不会危害到虚拟机。这个阶段主要完成四种验证：

- **文件格式验证**：验证字节流是否符合 Class 文件的规范，如主次版本号是否在当前虚拟机范围内，常量池中的常量是否有不被支持的类型
- **元数据验证**：对字节码描述的信息进行语义分析，如这个类是否有父类、是否继承了不允许被继承的 final 类等
- **字节码验证**：是整个验证过程中最复杂的阶段，通过验证数据流和控制流的分析确定程序语义是否正确，主要针对方法体验证，如方法中的类型转换是否正确、跳转指令是否合法等
- **符号引用验证**：发生在解析阶段，确保解析动作能正确执行，如引用的类、字段、方法是否存在且具有合适的访问权限

### 3. 准备（Preparation）

准备阶段为类的**静态变量**分配内存并设置**默认初始值**（零值），这些内存都在方法区中分配。需要注意：

- 这里设置的是默认值而非代码中指定的值（如 `static int value = 123`，准备阶段 value 的值是 0 而非 123）
- 实例变量不会在此阶段分配内存，它随着对象的创建而分配在 Java 堆中
- `static final` 修饰的常量在准备阶段就会被赋予代码中指定的值

### 4. 解析（Resolution）

解析阶段将常量池中的**符号引用**替换为**直接引用**。符号引用是用一组符号来描述引用的目标（如类的全限定名），而直接引用是可以直接指向目标的指针、偏移量或句柄。

解析动作并不一定在初始化之前完成，JVM 规范允许懒解析——在真正使用到符号引用时才进行解析。

### 5. 初始化（Initialization）

初始化是类加载的最后一步，也是真正开始执行类中定义的 Java 代码的阶段。在前面的阶段中，除了加载阶段用户可以通过自定义类加载器参与之外，其余动作完全由虚拟机主导。

初始化阶段执行的是类构造器 `<clinit>()` 方法，它由编译器自动收集类中所有静态变量的赋值动作和静态代码块中的语句合并而成。

### 6. 使用（Using）

类加载完成后就可以正常使用了。

### 7. 卸载（Unloading）

当满足特定条件时（如加载该类的 ClassLoader 实例已被 GC 回收），类可以被卸载。由引导类加载器加载的类在 JVM 生命周期内不会被卸载。

## 类加载器的工作步骤

从宏观视角来看，类加载器做了三件事：

1. **在特定目录找到 .class 文件并读取**：不同的类加载器负责不同的路径
2. **将 .class 文件实例化为对应的 Class 对象**：Java 是模板系语言，读取字节码后生成 Class 对象（类模板）
3. **基于 Class 对象创建实例**：只有拥有类模板，才能根据模板创建新的对象实例

---

## 三大内置类加载器

JVM 提供了三种内置的类加载器，它们各自负责不同路径下的类加载工作。

### 启动类加载器（Bootstrap ClassLoader）

启动类加载器是 JVM 自身的一部分，使用 C++ 语言实现，负责加载 `<JAVA_HOME>/lib` 路径下的核心类库或 `-Xbootclasspath` 参数指定的路径下的 JAR 包。

出于安全考虑，Bootstrap 类加载器只加载包名以 `java`、`javax`、`sun` 等开头的类。即使将自定义的 JAR 包放入 `lib` 目录，如果文件名不被虚拟机识别（如 `rt.jar`、`charsets.jar` 等），也不会被加载。

### 扩展类加载器（Extension ClassLoader）

由 `sun.misc.Launcher$ExtClassLoader` 实现，是 Launcher 的静态内部类。负责加载 `<JAVA_HOME>/lib/ext` 目录下或者 `-Djava.ext.dirs` 系统变量指定路径中的类库。

在 Java 9 模块化之后，扩展类加载器被**平台类加载器（Platform ClassLoader）**取代。

### 系统类加载器（System/Application ClassLoader）

由 `sun.misc.Launcher$AppClassLoader` 实现，负责加载 `java -classpath` 或 `-Djava.class.path` 指定路径下的类库，即我们日常开发中 classpath 路径下的代码。

系统类加载器是程序中默认的类加载器，可以通过 `ClassLoader.getSystemClassLoader()` 方法获取。

## 双亲委派模型

双亲委派模型（Parents Delegation Model）是 Java 类加载器协同工作的核心机制。

### 工作原理

当一个类加载器收到类加载请求时，它不会自己先去加载，而是把这个请求委托给父类加载器去执行。如果父类加载器还存在其父类加载器，则进一步向上委托，依次递归，请求最终到达顶层的启动类加载器。如果父类加载器可以完成类加载任务，就成功返回；倘若父类加载器无法完成此加载任务，子加载器才会尝试自己去加载。

```
           Bootstrap ClassLoader
                    ↑
           Extension ClassLoader
                    ↑
           Application ClassLoader
                    ↑
            自定义 ClassLoader
```

### 双亲委派模型的优势

采用双亲委派模式的好处是 Java 类随着它的类加载器一起具备了一种带有**优先级的层次关系**：

1. **避免类的重复加载**：当父加载器已经加载了该类时，子 ClassLoader 不需要再加载一次，保证类在 JVM 中的唯一性

2. **保证核心 API 安全**：Java 核心 API 中定义的类不会被随意替换。假设通过网络传递一个名为 `java.lang.Integer` 的类，通过双亲委托模式传递到启动类加载器后，发现该类已被加载，不会重新加载网络传递的版本，从而防止核心 API 库被篡改

3. **沙箱安全机制**：自定义的 `java.lang` 包下的类无法被加载，因为 `java.lang` 是核心 API 包，需要特殊的访问权限，强制加载将抛出安全异常

### 如何自定义类加载器

- **不打破双亲委派模型**：只需要重写 `findClass()` 方法即可，`loadClass()` 中的委派逻辑会正常执行
- **打破双亲委派模型**：需要重写整个 `loadClass()` 方法，绕过父类委派的逻辑

## 双亲委派模型的破坏场景

### 不同加载器加载同一个类

不同的类加载器加载了同一个 .class 文件，在 JVM 中会被视为两个不同的 Class 对象。这意味着 `instanceof` 判断、类型转换等操作都会失败。这也是双亲委派模型存在的重要原因之一——通过委派机制确保核心类的唯一性。

### 线程上下文类加载器与 SPI

双亲委派模型最著名的破坏者是**线程上下文类加载器（Thread Context ClassLoader）**。

在 Java 应用中存在许多服务提供者接口（Service Provider Interface，SPI），如 JDBC、JNDI 等。这些 SPI 接口属于 Java 核心库，存在于 `rt.jar` 中，由 Bootstrap 类加载器加载。然而 SPI 的第三方实现代码（如 MySQL JDBC 驱动）存放在 classpath 路径下，需要由 Application ClassLoader 加载。

问题在于：SPI 核心接口中的代码需要加载具体的第三方实现类并调用其方法，但 Bootstrap 类加载器无法直接加载 classpath 下的类，双亲委派模型也不允许反向委托。

线程上下文类加载器正是为了解决这个问题而引入的——它允许父类加载器加载的代码通过 `Thread.currentThread().getContextClassLoader()` 获取到子类加载器，从而加载 SPI 的实现类。这虽然打破了双亲委派模型的原则，但却是工程实践中的合理妥协。
