### 引言

当单表数据量达到千万甚至亿级别时，数据库的读写性能会显著下降，单纯的索引优化和读写分离已经无法满足业务需求。此时，**分库分表**（Database Sharding）成为必然的技术选择。然而，分库分表不是简单地拆分数据，它会引入主键生成、跨库 JOIN、分布式事务、分页查询等一系列新问题。本文将从分片策略、中间件选型和常见问题三个维度，系统性地梳理分库分表的实战要点。

---

### 分片策略

分片策略决定了数据如何分布到不同的数据库和表中，直接影响查询效率和系统可维护性。

#### MyCat 分片策略

MyCat 是目前使用广泛的数据库中间件，支持多种分片策略，如范围分片、Hash 取模、一致性 Hash、枚举分片等。

参考：<https://mp.weixin.qq.com/s/LHTpCPDdLK1TfDcPY7c5vQ>

#### MyCat 的 JOIN 支持

**同库 JOIN**：通过全局表和 ER 表实现。

全局表是在所有分片节点上冗余存储的表（如字典表），适合数据量小且变更频率低的公共数据：

```xml
 <!-- 1为开启全局表一致性检测、0为关闭 -->
<property name="useGlobleTableCheck">1</property> 
<table name="company" primaryKey="ID" type="global" dataNode="dn1,dn2,dn3" />
<!-- ER表 -->
<table name="customer" primaryKey="ID" dataNode="dn1,dn2" rule="sharding-by-intfile">
    <childTable name="orders" primaryKey="ID" joinKey="customer_id" parentKey="id">
            <childTable name="order_items" joinKey="order_id" parentKey="id" />
    </childTable>
    <childTable name="customer_addr" primaryKey="ID" joinKey="customer_id" parentKey="id" />
</table>
```

ER 表通过外键关系将关联数据路由到同一个分片节点，保证 JOIN 操作可以在单节点内完成。

参考：<https://blog.csdn.net/Weixiaohuai/article/details/100900026>

**跨库 JOIN**：通过 catlet 实现跨分片的 JOIN 查询：

```sql
/*!mycat:catlet=io.mycat.catlets.ShareJoin*/select * from employee a, employee_detail b where a.id = b.id;
```

参考：<https://blog.csdn.net/Weixiaohuai/article/details/101557503>

### 主键避重问题

分库分表后，传统的自增 ID 无法保证全局唯一性。常见的分布式 ID 生成方案包括：

#### ID 生成器

| 方案 | 特点 |
|------|------|
| **Twitter Snowflake** | 64 位 Long 型，由时间戳 + 机器 ID + 序列号组成，趋势递增，高性能 |
| **美团 Leaf** | 支持号段模式和 Snowflake 模式，兼顾性能与可用性 |
| **百度 UidGenerator** | 基于 Snowflake 优化，支持自定义各段位数 |

#### 散表规则

- **按 ID 范围分片**：要求 ID 自增，按区间路由（如 0~1000万到表1，1000万~2000万到表2），扩容方便但容易产生热点
- **简单 Hash（取模）**：根据 ID 对表数量取模，数据分布均匀但扩容需要数据迁移
- **一致性 Hash**：引入虚拟节点，扩容时只需迁移部分数据，减少数据迁移成本

### 分页查询问题

分库分表后的分页查询是一个经典难题。例如查询"第 100 页，每页 20 条"时，无法直接在各分片上执行 `LIMIT 1980, 20`。

常见解决方案：
- **分表查询 + 应用程序合并**：在每个分片上查询前 N 条数据，由应用程序进行内存排序和合并。数据量较大时性能开销显著。
- **禁止深度分页**：产品层面限制翻页范围，结合"下一页"游标分页模式
- **搜索引擎辅助**：将需要分页检索的数据同步到 Elasticsearch，由 ES 完成分页排序

---

### 关联查询问题

跨库 JOIN 是分库分表后最棘手的问题之一，实践中通常采用以下策略：

- **避免 JOIN 操作**：将 JOIN 改造为多次查询，由应用程序在内存中合并（适合小数据量场景）
- **使用中间件**：MyCat、ShardingSphere 等中间件提供了有限的跨库 JOIN 支持
- **数据异构**：将需要关联查询的数据冗余到 Redis 或 Elasticsearch 中，通过宽表或索引方式满足查询需求

---

### 公共表问题

字典表、配置表等公共数据需要在所有分片中都能访问，解决方案：

- **多库冗余**：在每个分片中维护一份公共表的完整副本
- **多库同步**：通过 MyCat 的全局表机制或 Canal 等 binlog 同步工具保持各副本一致

---

### 事务一致性

分库分表后，原本在单库内通过本地事务保证的一致性，变成了分布式事务问题。

**JTA + XA（两阶段提交）**是标准的强一致性方案，但性能开销大、可用性受限。在实际业务中，更多采用**柔性事务**方案（如 TCC、本地消息表、Saga 等），通过最终一致性来平衡性能和正确性。

### 中间件选型：五大方案对比

| 中间件 | 类型 | 团队 | 现状与特点 |
|--------|------|------|-----------|
| **Cobar** | Proxy 层 | 阿里 B2B | 已停止维护，不支持读写分离、存储过程、跨库 JOIN 和分页 |
| **TDDL** | Client 层 | 淘宝 | 不支持 JOIN 和多表查询，依赖淘宝 Diamond 配置系统，使用较少 |
| **Atlas** | Proxy 层 | 360 | 社区已停止维护 5 年以上 |
| **ShardingSphere（原 Sharding-JDBC）** | Client 层 | 当当/Apache | SQL 语法支持广，支持分库分表、读写分离、分布式 ID、柔性事务，社区活跃 |
| **MyCat** | Proxy 层 | 社区 | 基于 Cobar 改造，功能完善，社区活跃，对各项目透明 |

#### 选型建议

**中小型公司推荐 ShardingSphere**：Client 层方案轻量，运维成本低，不需要额外部署和维护中间件集群。缺点是各业务系统都需要引入 SDK 依赖，升级需要逐一发布。

**中大型公司推荐 MyCat**：Proxy 层方案对业务系统透明，升级只需更新中间件本身。缺点是需要专人维护中间件集群，运维成本较高，且多一层网络转发会有轻微的性能损耗。

### 主从同步

分库分表往往配合主从复制使用，常见的复制优化方案：

#### 半同步复制（Semi-sync）

主库写入 binlog 后，强制将数据同步到从库。从库将日志写入本地 relay log 后返回 ACK，主库收到至少一个从库的 ACK 后才认为写操作完成。在数据安全性和写入性能之间取得平衡。

#### 并行复制

从库开启多个线程，并行读取 relay log 中不同库的日志并重放。这是**库级别的并行**，可以显著降低从库的复制延迟。MySQL 5.7 之后还支持基于组提交的并行复制，进一步提升了并行度。

### 总结

分库分表是应对海量数据的必经之路，但它带来的复杂性也不容忽视。在实施前需要充分评估业务场景，选择合适的分片策略和中间件；在实施中需要妥善解决主键生成、跨库查询、分布式事务等问题；在实施后需要持续关注数据倾斜、热点问题和扩容方案。最重要的一条原则是：**能不分就不分，非分不可时做好充分的技术选型和方案设计**。
