### 一、索引体系

#### 1.1 聚集索引与非聚集索引

- **聚集索引（Clustered Index）**：按照每张表的主键构造一棵 B+ 树，同时叶子节点中存放的即为整张表的行记录数据，也将聚集索引的叶子节点称为数据页。聚集索引的这个特性决定了索引组织表中数据也是索引的一部分。
- **辅助索引（Secondary Index，也称非聚集索引）**：叶子节点并不包含行记录的全部数据。叶子节点除了包含键值以外，每个叶子节点中的索引行中还包含了指向主键的指针。

这意味着通过辅助索引查询数据时，需要先找到主键值，再回到聚集索引中查找完整的行记录，这个过程称为"回表"。

- 深入理解聚集索引与非聚集索引：<https://mp.weixin.qq.com/s/nWiM87G1CBGUkX0_QkUf3g>
- 更多参考：<https://blog.csdn.net/riemann_/article/details/90324846>

#### 1.2 索引原理（B+ Tree）

索引数据结构的演进路线为：

**BST（Binary Search Tree，二叉查找树）→ AVL Tree（平衡二叉树）→ 红黑树 → B-/B+ Tree（平衡多路查找树）→ LSM**

每一步演进都是为了解决上一步的瓶颈问题：

- **为什么 Hash 比 B+ 树更快，MySQL 还用 B+ 树？**
  1. 如果只选一个数据，Hash 确实更快。但数据库中经常选择多条数据，B+ 树索引有序且有链表相连，范围查询效率远高于 Hash
  2. 数据库索引一般在磁盘上，数据量大时可能无法一次装入内存，B+ 树的设计允许数据分批加载，同时树的高度较低，提高查找效率
  3. 与 B 树相比，B+ 树所有数据都在叶子结点，不用跨层访问，且有链表结构，只需找到首尾就能把所有数据取出来

- **各种树结构的适用场景**：
  - AVL Tree 在 Windows NT 内核中广泛使用，适合写少读多的场景
  - 红黑树适合于内存索引查找，比如 Java 中的 Map、Set。参考：<https://mp.weixin.qq.com/s/cnDx8lJ6fXHgLZWsqjWrag>
  - B 树尤其是多路 B 树主要用在文件存储系统，文件系统和数据库的索引都存在硬盘上，用 B 树可以一路一路地查找
  - 红黑树只能有 2 个子节点，B 树子节点大于 2，保证存储相同大小的数据时树的高度更小、数据局部更加紧凑，符合硬盘 IO 局部加载优化的特性

- 更加详细的应用范围：<https://www.cnblogs.com/tiancai/p/9024351.html>
- 完美总结 B+ 树：<https://mp.weixin.qq.com/s/YMbRJwyjutGMD1KpI_fS0A>

### 二、锁机制

MySQL 的锁机制是保证数据一致性和事务控制的核心手段。

#### 2.1 乐观锁与悲观锁

- **悲观锁**：读取数据时就加锁排他，适用于写操作频繁的场景
- **乐观锁**：读取数据不加锁，更新时才做冲突检测（比如版本号机制、CAS 算法），适用于读多写少的场景

参考：
- <https://www.jianshu.com/p/5a081ff5de58>
- <https://www.cnblogs.com/jtlgb/p/10833136.html>

#### 2.2 意向锁

意向共享锁只是表级别的，与行级别的共享锁与排他锁没有直接关系，所以意向锁互相兼容之后交给行级锁进一步控制。意向共享锁与意向排他锁主要是解决表级别"直接"共享锁与"直接"排他锁的冲突检测问题。

添加意向锁的方式：
- 手动：
  - `SELECT column FROM table ... LOCK IN SHARE MODE;`
  - `SELECT column FROM table ... FOR UPDATE;`
- 自动：加行锁的时候，数据库引擎自动添加

#### 2.4 各种锁的关系

所有锁的本质都是为了实现数据一致性及事务控制：
- **共享锁（Share Lock）**：允许多个事务同时读
- **排它锁（Exclusive Lock）**：只允许一个事务读写
- **意向共享锁、意向排它锁**：锁定的是表，用于快速判断表中是否有行级锁

### 三、存储引擎

#### 3.1 InnoDB vs MyISAM

**核心区别在于事务**。InnoDB 基于 MVCC（Multi Version Concurrency Control）实现了一套细粒度的高性能事务控制；MyISAM 仅实现了基于表锁的粗粒度事务（适用于日志后台存储、CMS 后台系统、配置相关、MySQL 系统库里的某些系统表）。

#### 3.2 InnoDB 引擎的 4 大特性

参考：<https://www.cnblogs.com/zhs0/p/10528520.html>

1. **插入缓冲（Insert Buffer）**：只对于非聚集索引（非唯一）的插入和更新有效。对于每一次插入不是直接写到索引页中，而是先判断插入的非聚集索引页是否在缓冲池中，如果在则直接插入；若不在，则先放到 Insert Buffer 中，再按照一定的频率进行合并操作，再写回 Disk。这样通常能将多个插入合并到一个操作中，减少随机 IO 带来的性能损耗。

2. **二次写（Double Write）**：InnoDB 默认 DB Page 为 16KB，而文件系统、磁盘、扇区对应的 Page 小于该数字，因此一次 DB Page 可能被多次写入才能真正写入成功。在写数据时，会在共享表空间写一份数据，之后再同步到磁盘。在应用重做日志前，用户需要一个页的副本，当写入失效发生时，先通过页的副本来还原该页，再进行重做，这就是 Double Write。参考：<https://www.cnblogs.com/chenpingzhao/p/4876282.html>

3. **自适应哈希索引（AHI）**：InnoDB 会监控表上多个索引页的查询，如果观察到建立哈希索引可以带来速度提升，则自动建立哈希索引。主要用于精确等值查找，对范围查找不生效。

4. **预读（Read Ahead）**：数据预加载，将可能需要的数据提前读入缓冲池，减少磁盘 IO 次数。

#### 3.3 共享表空间 vs 独立表空间

- **共享表空间**：InnoDB 的所有数据保存在一个单独的表空间里面，这个表空间可以由很多个文件组成，一个表可以跨多个文件存在。表空间的最大限制为 64TB。
- **独占表空间**：每一个表都将会生成独立的文件方式来进行存储，每一个表都有一个 `.frm` 表描述文件和一个 `.ibd` 文件，包括单独一个表的数据内容以及索引内容。

### 四、WAL 技术（Write-Ahead Logging）

WAL（Write-ahead logging，预写式日志）是数据库系统提供原子性和持久化的一系列技术。在使用 WAL 的系统中，所有的修改都先被写入到日志中，然后再被应用到系统状态中。通常包含 redo 和 undo 两部分信息。

为什么需要 WAL？如果一个系统直接将变更应用到系统状态中，那么在机器掉电重启之后系统需要知道操作是成功了，还是只有部分成功或者是失败了。如果使用了 WAL，那么在重启之后系统可以通过比较日志和系统状态来决定是继续完成操作还是撤销操作。

参考：<https://www.cnblogs.com/hzmark/p/wal.html>

### 五、查询优化

#### 5.1 深分页优化

```sql
# 优化前
select * from t_third_device where third='kuaishou' limit 300000,5;
# 优化后（先通过覆盖索引查出 id，再回表取完整数据）
select * from t_third_device a inner join (select id from t_third_device where third='kuaishou' limit 300000,5) b on a.id=b.id;
```

### 六、常见 FAQ

**Q: INT(M) 中 M 究竟有何用？**

int 类型占四个字节，即 4×8=32 比特位，可以表示 2^32 个数字。声明一个 int 类型的数据，长度便已经固定。M 与数据存储长度没有关系，只是标识数据显示长度，在显示时不足位数会自动填充 0 进行补齐，必须配合 AutoFill 使用。

**Q: InnoDB 行锁是通过什么实现的？**

InnoDB 行锁是通过给索引上的索引项加锁来实现的（MySQL 与 Oracle 不同，后者是通过在数据块中对相应数据行加锁来实现）。这意味着：只有通过索引条件检索数据，InnoDB 才使用行级锁，否则将使用表锁！

**Q: 数据库三范式是什么？**

1. 表中的每个字段不可分割
2. 表有且只有一个主键，即每个表只描述一件事
3. 不存在对非主键列的依赖传递（比如 userId 和 userName 不要同时出现在 order 表中，当然有时为了查询效率也会冗余个别字段）

**Q: 删除记录后自增主键会怎样？**

一张表有 ID 自增主键，insert 了 17 条记录后，删除第 15、16、17 条记录，再重启 MySQL，再 insert 一条记录：
- MyISAM：ID 是 **18**，因为 MyISAM 把自增主键的最大 ID 记录到数据文件里，重启不会丢失
- InnoDB：ID 是 **15**，因为 InnoDB 只把自增主键的最大 ID 记录到内存中，重启数据库会导致最大 ID 丢失

**Q: FLOAT 和 DOUBLE 的区别？**

浮点数以 8 位精度存储在 FLOAT 中，有四个字节；浮点数存储在 DOUBLE 中，精度为 18 位，有八个字节。

**Q: CHAR_LENGTH 和 LENGTH 的区别？**

CHAR_LENGTH 是字符数，LENGTH 是字节数。Latin 字符的这两个值相同，但对于 Unicode 和其他编码，它们是不同的。

**Q: 什么是 federated 表？**

Federated 表允许访问位于其他服务器数据库上的表。参考：<https://blog.csdn.net/qq_33578833/article/details/80177115>

### 八、总结

MySQL 的知识体系庞大而深入，但核心可以归纳为几条主线：**索引决定查询效率**（理解 B+ 树为什么适合磁盘存储）、**锁保证数据一致性**（从悲观/乐观到行锁/表锁/意向锁）、**存储引擎决定功能边界**（InnoDB 的 MVCC 与四大特性）、**WAL 保证持久化**（redo/undo 日志）。掌握这些核心概念后，再结合实际业务进行 SQL 调优和架构设计，就能应对大部分数据库相关的技术挑战。
