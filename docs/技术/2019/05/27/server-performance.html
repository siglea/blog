深入理解服务器性能：从单机评估到容量规划

日期： 2019年05月27日

在系统设计和架构评审中，”这台服务器能扛多少并发？”是一个高频出现的问题。回答这个问题需要对服务器性能有系统性的理解——不仅仅是知道 CPU 和内存的规格，更要理解它们如何影响实际业务的吞吐量和响应时间。

服务器性能的核心指标

评估服务器性能，首先要明确以下几个关键指标：

QPS（Queries Per Second）

每秒查询率，衡量服务器在单位时间内处理请求的能力。对于 Web 服务来说，这是最直观的性能指标。

TPS（Transactions Per Second）

每秒事务数，与 QPS 类似，但一个事务可能包含多次查询。在数据库和支付场景中更常用。

RT（Response Time）

响应时间，从客户端发送请求到收到响应的总耗时。通常关注 平均 RT 和 P99 RT（99% 请求的响应时间上限）。

并发数（Concurrency）

系统同时处理的请求数。它与 QPS 和 RT 的关系可以用 Little’s Law 近似：

并发数 = QPS × 平均RT（秒）

例如，QPS = 1000，平均 RT = 50ms，则并发数 ≈ 1000 × 0.05 = 50。

单服务器性能评估

CPU 密集型 vs IO 密集型

不同业务场景下，性能瓶颈不同：

类型 特点 典型场景 优化方向
CPU 密集型 CPU 使用率高，IO 等待少 加密解密、图像处理、科学计算 提升 CPU 频率、增加核心数
IO 密集型 CPU 空闲多，等待 IO 完成 数据库查询、文件读写、网络调用 使用异步 IO、增加 IO 带宽

大多数 Web 应用属于 IO 密集型，瓶颈在数据库查询和外部服务调用上。

线程数与连接数

对于 Java Web 服务（如 Tomcat），核心配置参数包括：

# Tomcat 线程池配置
server.tomcat.max-threads=200        # 最大工作线程数
server.tomcat.min-spare-threads=10   # 最小空闲线程数
server.tomcat.accept-count=100       # 等待队列长度
server.tomcat.max-connections=8192   # 最大连接数

线程数设置经验公式：

  • CPU 密集型：线程数 = CPU 核心数 + 1
  • IO 密集型：线程数 = CPU 核心数 × (1 + IO 等待时间 / CPU 计算时间)

内存评估

服务器内存需要覆盖以下部分：

  • 操作系统：预留 1-2 GB
  • JVM 堆内存：根据业务数据量和缓存需求设置
  • 堆外内存：NIO Direct Buffer、线程栈等
  • 文件系统缓存：Linux 会利用空闲内存做 Page Cache

单服务器并发连接能力

一个经典的面试题是：”一台服务器最多能支持多少个 TCP 连接？”

理论上限

TCP 连接由四元组标识：(源IP, 源端口, 目标IP, 目标端口)。理论上，一台服务器作为 Server 端监听一个端口，可以接受的连接数取决于客户端的 IP 和端口组合，远超百万级别。

实际限制因素

  1. 文件描述符限制：每个 TCP 连接消耗一个 fd，Linux 默认单进程上限 1024，需要调整
# 查看当前限制
ulimit -n

# 临时修改
ulimit -n 65535

# 永久修改 /etc/security/limits.conf
* soft nofile 65535
* hard nofile 65535

  1. 内存限制：每个 TCP 连接在内核中占用约 3-4 KB（socket buffer），10 万连接约需 300-400 MB

  2. CPU 限制：大量连接的事件处理需要高效的 IO 多路复用（select → poll → epoll）

  3. 端口限制：作为客户端发起连接时，本地端口范围默认 32768-60999

# 查看端口范围
cat /proc/sys/net/ipv4/ip_local_port_range

# 调整端口范围
echo "1024 65535" > /proc/sys/net/ipv4/ip_local_port_range

C10K 与 C10M 问题

  • C10K（单机 1 万并发连接）：在 2000 年代初期是一个挑战，epoll 和 kqueue 的出现使其成为常规操作
  • C10M（单机 1000 万并发连接）：需要绕过内核协议栈，使用 DPDK、Kernel Bypass 等技术

性能测试方法

在做容量规划前，压力测试是必不可少的步骤：

常用压测工具

工具 特点 适用场景
wrk 轻量级、高性能 HTTP 接口压测
JMeter 功能丰富、支持 GUI 复杂场景、多协议
ab (ApacheBench) 简单易用 快速验证
Locust Python 编写，支持分布式 自定义场景
Gatling Scala DSL，报告美观 CI/CD 集成

wrk 快速压测示例

# 使用 12 线程、400 并发连接，压测 30 秒
wrk -t12 -c400 -d30s http://localhost:8080/api/users

# 输出示例：
# Requests/sec:  12345.67  (QPS)
# Transfer/sec:  1.23MB
# Latency Avg:   32.45ms   P99: 120.00ms

容量规划实践

有了性能测试数据后，可以进行容量规划：

  1. 确定目标：预期峰值 QPS 是多少？可接受的 P99 RT 是多少？
  2. 单机基准：通过压测得到单机在目标 RT 下的最大 QPS
  3. 计算实例数：实例数 = 峰值 QPS / 单机 QPS × 冗余系数（通常 1.5-2.0）
  4. 验证：全链路压测验证整体容量

常见性能瓶颈与排查

现象 可能原因 排查命令
CPU 使用率高 死循环、GC 频繁、加密运算 top, jstack, perf
内存持续增长 内存泄漏、缓存无上限 jmap, MAT, free -m
IO Wait 高 慢查询、磁盘IO、日志写入 iostat, iotop
网络延迟高 带宽不足、DNS 解析慢、连接池耗尽 netstat, ss, tcpdump
线程数打满 下游超时、锁竞争 jstack, Thread Dump 分析

总结

服务器性能评估不是简单地堆砌硬件参数，而是要结合业务特点、系统架构和实际压测数据来综合判断。记住这个优化思路链条：明确瓶颈 → 定量测量 → 针对性优化 → 回归验证。在微服务架构下，单节点的性能评估更要结合全链路视角，因为系统的整体吞吐量取决于最慢的那个环节。
