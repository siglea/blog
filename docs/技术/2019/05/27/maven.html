Maven 实战指南：依赖管理、冲突解决与常用配置

日期： 2019年05月27日

Maven 是 Java 生态中最主流的项目构建和依赖管理工具。它通过约定优于配置的理念，统一了项目结构、构建流程和依赖管理方式。本文将从日常开发中最常用的命令出发，重点讲解 Maven 的依赖冲突解决机制、scope 作用域、常见错误排查以及镜像源配置，帮助开发者更高效地使用 Maven。

一、常用 Maven 命令

1.1 清理与构建

    mvn clean package -Dmaven.test.skip=true

这是日常开发中最常用的命令组合：clean 清除上次构建产物，package 编译并打包，-Dmaven.test.skip=true 跳过测试以加快构建速度。

1.2 编译

mvn compile 仅编译 src/main 下的源代码，不执行打包和测试。

1.3 创建项目骨架

创建普通 Java 项目：

   mvn archetype:generate -DgroupId=com.siglea -DartifactId=demo

创建 JavaWeb 项目：

   mvn archetype:generate -DgroupId=com.siglea -DartifactId=demoweb -DarchetypeArtifactId=maven-archetype-webapp

archetype:generate 会交互式地引导你选择项目模板并填写坐标信息，是快速初始化项目的便捷方式。

1.4 运行项目

使用 Jetty 插件运行：

需要先在 pom.xml 中添加 Jetty 依赖：

    <!-- https://mvnrepository.com/artifact/org.mortbay.jetty/jetty -->
    <dependency>
        <groupId>org.mortbay.jetty</groupId>
        <artifactId>jetty</artifactId>
        <version>6.1.26</version>
    </dependency>

然后执行 mvn jetty:run 即可启动 Web 应用。

使用 Spring Boot 插件运行：

mvn spring-boot:run，这是 Spring Boot 项目的标准启动方式。

二、依赖冲突解决机制

依赖冲突是 Maven 项目中最常遇到的问题之一。当项目中通过不同路径引入了同一个依赖的不同版本时，Maven 会按照以下规则来决定使用哪个版本。

2.1 第一声明者优先原则

在 pom.xml 中先声明的依赖优先。例如下面的配置中，spring-context 先声明，它传递依赖的 spring-beans-4.2.4 会被优先采用，而 struts2-spring-plugin 传递的 spring-beans-3.0.5 会被忽略：

<dependencies>
  <!--  第一声明者优先原则 spring-beans-4.2.4 -->
    <dependency>
        <groupId>org.springframework</groupId>
        <artifactId>spring-context</artifactId>
        <version>4.2.4.RELEASE</version>
    </dependency>
   
<!--   spring-beans-3.0.5 -->
    <dependency>
        <groupId>org.apache.struts</groupId>
        <artifactId>struts2-spring-plugin</artifactId>
        <version>2.3.24</version>
    </dependency>
</dependencies>

2.2 路径近者优先原则

依赖传递路径短的优先。如果直接声明了某个依赖的版本，它会覆盖通过传递依赖引入的版本：

<dependencies>   
    <!--   spring-beans-3.0.5 -->
    <dependency>
        <groupId>org.apache.struts</groupId>
        <artifactId>struts2-spring-plugin</artifactId>
        <version>2.3.24</version>
    </dependency>
    <!--  路径近者优先原则 spring-beans-4.2.4 -->
    <dependency>
  		<groupId>org.springframework</groupId>
  		<artifactId>spring-beans</artifactId>
  		<version>4.2.4.RELEASE</version>
  	</dependency>
</dependencies>

2.3 排除依赖原则

使用 <exclusions> 标签主动排除不想要的传递依赖：

<dependency>
  		<groupId>org.apache.struts</groupId>
  		<artifactId>struts2-spring-plugin</artifactId>
  		<version>2.3.24</version>
  		<exclusions>
  		  <exclusion>
  		    <groupId>org.springframework</groupId>
  		    <artifactId>spring-beans</artifactId>
  		  </exclusion>
  		</exclusions>
</dependency>

这是实际开发中最常用的解决冲突手段，可以精确控制传递依赖。

2.4 版本锁定原则

使用 <dependencyManagement> 统一锁定依赖版本，是大型项目中管理依赖版本的最佳实践。在这里声明的版本不会直接引入依赖，但会作为子模块或传递依赖的版本约束：

<!-- 锁定版本，struts2-2.3.24、spring4.2.4、hibernate5.0.7 -->
<properties>
        <spring.version>4.2.4.RELEASE</spring.version>
        <hibernate.version>5.0.7.Final</hibernate.version>
        <struts.version>2.3.24</struts.version>
</properties>
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>org.apache.struts</groupId>
            <artifactId>struts2-core</artifactId>
            <version>${struts.version}</version>
        </dependency>
        <dependency>
            <groupId>org.springframework</groupId>
            <artifactId>spring-context</artifactId>
            <version>${spring.version}</version>
        </dependency>
        <dependency>
            <groupId>org.hibernate</groupId>
            <artifactId>hibernate-core</artifactId>
            <version>${hibernate.version}</version>
        </dependency>
    </dependencies>
</dependencyManagement>

三、依赖作用域（scope）

Maven 的 scope 决定了依赖在不同阶段的可见性，合理使用 scope 可以减小最终产物的体积并避免冲突。

scope 编译 测试 打包 运行时 说明
compile ✔️ ✔️ ✔️ ✔️ 默认值，全阶段可用
provided ✔️ ✔️ ✘ ✔️ 运行时由容器提供，如 servlet-api
runtime ✘ ✔️ ✔️ ✔️ 编译时只需接口，如 JDBC 驱动
test ✔️(测试) ✔️ ✘ ✘ 仅测试阶段可用，如 JUnit
system - - - - 类似 provided，但需配合 systemPath 指定本地 JAR 路径

四、常见错误与解决方案

错误 1：mvn archetype:generate 报 NoClassDefFoundError

    [INFO] --- maven-archetype-plugin:3.0.1:generate (default-cli) @ standalone-pom ---
    [WARNING] Error injecting: org.apache.maven.archetype.common.DefaultPomManager
    java.lang.NoClassDefFoundError: org/jdom/JDOMException
            at java.lang.Class.getDeclaredConstructors0(Native Method)
    解决办法：删除 .m2/repository 对应报错的文件

这通常是本地仓库中的 JAR 文件损坏导致的，删除对应目录后 Maven 会重新下载。

错误 2：mvn archetype:create 已废弃

mvn archetype:create 修改为 mvn archetype:generate

archetype:create 是旧版本的目标，在新版 Maven 中已被移除，统一使用 archetype:generate。

错误 3：mvn site 报 NoClassDefFoundError

当执行 mvn site 出现 java.lang.NoClassDefFoundError: org/apache/maven/doxia/siterenderer/DocumentContent 时，需要在 pom.xml 中显式指定 site 插件版本：

<build>
<pluginManagement>
    <plugins>
        <plugin>
          <groupId>org.apache.maven.plugins</groupId>
          <artifactId>maven-site-plugin</artifactId>
          <version>3.3</version>
        </plugin>
        <plugin>
          <groupId>org.apache.maven.plugins</groupId>
          <artifactId>maven-project-info-reports-plugin</artifactId>
          <version>2.7</version>
        </plugin>
    </plugins>
    </pluginManagement>
</build>

五、镜像源配置

国内开发者访问 Maven 中央仓库速度较慢，配置国内镜像源可以显著提升依赖下载速度。以下配置可以放在 pom.xml 或 settings.xml 中：

<mirrors>
<mirror>
    <id>nexus-aliyun</id>
    <name>Nexus aliyun</name>
    <url>http://maven.aliyun.com/nexus/content/groups/public/</url>
    <mirrorOf>central</mirrorOf>
</mirror>
<mirror>
   <id>maven2-repository.dev.java.net</id>
   <name>Java.net Repository for Maven</name>
   <url>http://download.java.net/maven/2/</url>
   <mirrorOf>central1</mirrorOf>
</mirror>
<mirror>
    <id>java.net2.r1</id>
    <name>java.net Repository hosting the jee6 artifacts</name>
    <url>http://repo1.maven.org/maven2/</url>
    <mirrorOf>central2</mirrorOf>
</mirror>
<mirror>
    <id>java.net2.r3</id>
    <name>java.net Repository hosting the jee6 artifacts</name>
    <url>http://repository.sonatype.org/content/groups/public/</url>
    <mirrorOf>central3</mirrorOf>
</mirror>
<mirror>
<mirror>
    <id>java.net2.r4</id>
    <name>java.net Repository hosting the jee6 artifacts</name>
    <url>http://mirrors.ibiblio.org/pub/mirrors/maven2/</url>
    <mirrorOf>central4</mirrorOf>
</mirror>
<mirror>
    <id>java.net2.r2</id>
    <name>java.net Repository hosting the jee6 artifacts</name>
    <url>http://repository.jboss.com/maven2/</url>
    <mirrorOf>central5</mirrorOf>
</mirror>
<mirror>
    <id>java.net2</id>
    <name>java.net Repository hosting the jee6 artifacts</name>
    <url>http://download.java.net/maven/2</url>
    <mirrorOf>central6</mirrorOf>
</mirror>
</mirrors>

其中阿里云镜像（nexus-aliyun）是国内最推荐的选择，速度快且稳定。

六、总结

Maven 的核心价值在于统一了 Java 项目的构建和依赖管理流程。掌握依赖冲突的四大解决原则（第一声明优先、路径近者优先、排除依赖、版本锁定）和 scope 作用域的区别，就能应对日常开发中绑大部分依赖相关的问题。对于大型多模块项目，建议充分利用 dependencyManagement 进行版本统一管理，并配合 mvn dependency:tree 命令来可视化依赖树，快速定位冲突根源。
