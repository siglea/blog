深入理解TCP/IP协议：从三次握手到性能优化全解析

日期： 2019年05月27日

TCP/IP 协议是互联网通信的基石。无论是网页浏览、即时通讯还是微服务调用，都离不开 TCP 协议提供的可靠传输保障。本文将从 TCP 的核心机制入手，深入剖析连接建立与断开、重传与流控、拥塞处理等关键环节，同时探讨在实际工程中常见的性能优化策略。

推荐阅读：40张图解TCP https://mp.weixin.qq.com/s/LZAAKkrbyu1tNEr0eaRsZQ

TCP 协议核心要素

TCP 协议的核心由三个要素构成：IP+端口（标识连接端点）、序列号（解决乱序问题）、窗口大小（流量控制）。理解这三个要素是掌握 TCP 协议的基础。

TCP 与 UDP 的头部对比

  • TCP 首部较长，未使用「选项」字段时为 24 个字节（6 个 32 位），含「选项」字段时更长。UDP 首部固定 8 个字节（2 个 32 位），开销较小。
  • 为什么 UDP 头部没有「首部长度」字段？ 因为 TCP 有可变长的「选项」字段，而 UDP 头部长度固定不变。
  • 为什么 UDP 头部有「包长度」字段而 TCP 没有？ TCP 数据长度可通过 IP总长度 - IP首部长度 - TCP首部长度 计算得出。实际上 UDP 也可以用同样方式计算，因此 UDP 的包长度字段有点冗余。

TCP 连接的建立：三次握手

TCP 建立连接需要经历 3 次握手和 4 种状态的变迁：LISTEN → SYN_SEND → SYN_RCVD → ESTABLISHED。值得注意的是，第三次握手可以携带数据，因为此时客户端已经明确知道连接建立了。

为什么是三次握手？不是两次、四次？

这是面试中最常见的 TCP 问题之一，答案涉及三个层面：

1. 阻止历史重复连接的初始化（主要原因）

如果是两次握手，就不能判断当前连接是否是历史连接。三次握手允许客户端在准备发送第三次报文时，凭借足够的上下文判断当前连接是否为历史连接：

  • 如果是历史连接（序列号过期或超时），第三次握手发送 RST 报文中止连接
  • 如果不是历史连接，第三次发送 ACK 报文，通信双方成功建立连接

2. 同步双方的初始序列号

三次握手确保双方都知道对方的初始序列号，这是后续可靠传输的基础。

3. 避免资源浪费

如果只有两次握手，当客户端的 SYN 请求在网络中阻塞时，客户端会重新发送 SYN。由于没有第三次握手，服务器每收到一个 SYN 就主动建立一个连接，造成多个冗余的无效连接。

SYN 攻击及防范

SYN 攻击是指攻击者短时间内伪造不同 IP 地址的 SYN 报文，服务端每接收到一个 SYN 就进入 SYN_RCVD 状态，但发出去的 ACK+SYN 报文无法得到回应，久而久之占满 SYN 接收队列，使服务器无法为正常用户服务。

防范措施：

  • net.ipv4.tcp_syncookies = 1：绑定客户端的 cookie，避免生成过多 SYN_RCVD 连接
  • net.ipv4.tcp_max_syn_backlog：设置最大 SYN_RCVD 数
  • net.ipv4.tcp_abort_on_overflow：超出处理能力直接丢弃
  • net.core.netdev_max_backlog：设置最大队列处理数

初始序列号 ISN

客户端和服务端的初始序列号 ISN 是不同的，因为网络中的报文会延迟、复制重发或丢失，不同的 ISN 可以避免不同连接之间产生互相影响。

ISN 的生成算法（RFC1948）：ISN = M + F(localhost, localport, remotehost, remoteport)

  • M 是一个计时器，每隔 4 毫秒加 1（转一圈约 4.55 小时）
  • F 是一个 Hash 算法（推荐 MD5），根据四元组生成随机数值

MSS 与 IP 分片

  • MTU：一个网络包的最大长度，以太网中一般为 1500 字节
  • MSS：除去 IP 和 TCP 头部后，一个网络包所能容纳的 TCP 数据的最大长度

为什么 TCP 层需要 MSS 而不让 IP 层分片？如果由 IP 层分片，当一个分片丢失时整个 IP 报文的所有分片都得重传（因为 IP 层没有超时重传机制）。TCP 在建立连接时协商双方的 MSS 值，数据超过 MSS 时先在 TCP 层分片，这样丢失一个分片只需重传该分片而非所有分片，大大提高了重传效率。

TCP 连接的断开：四次挥手

TCP 断开连接需要 4 次挥手和 6 种状态变迁：客户端经历 FIN_WAIT_1 → FIN_WAIT_2 → TIME_WAIT，服务端经历 CLOSE_WAIT → LAST_ACK → CLOSED。

TIME_WAIT 相关问题

为什么 TIME_WAIT 等待 2MSL？

MSL（Maximum Segment Lifetime）的单位是时间，而 TTL 是经过路由跳数。MSL 应大于等于 TTL 消耗为 0 的时间。等待 2 倍 MSL 是因为网络中可能存在来自发送方的数据包，当这些数据包被接收方处理后又会向对方发送响应，一来一回需要 2 倍时间。

在 LAST-ACK 状态，如果一直没收到 ACK 会发起重发。TIME_WAIT 太短会导致服务端一直处于 LAST-ACK 状态，当客户端发起新的 SYN 请求时，服务端会发送 RST 报文终止连接建立。

为什么需要 TIME_WAIT 状态？

  • 防止具有相同「四元组」的旧数据包被收到
  • 保证被动关闭方能正确关闭，即保证最后的 ACK 能让被动关闭方接收

TIME_WAIT 过多的危害：内存资源占用和端口资源占用（一个 TCP 连接至少消耗一个本地端口）。

优化方案：

  • net.ipv4.tcp_tw_reuse = 1 + net.ipv4.tcp_timestamps = 1：引入时间戳后重复的数据包会因时间戳过期被自然丢弃。注意：tcp_tw_reuse 需慎用，当客户端与服务端主机时间不同步时，客户端的消息会被直接拒绝。
  • net.ipv4.tcp_max_tw_buckets：默认 18000，超过此值时重置所有 TIME_WAIT 连接。方法过于暴力，不推荐使用。
  • SO_LINGER 选项：设置 l_onoff 非 0 且 l_linger 为 0 时，调用 close 后立即发送 RST 标志，跳过四次挥手和 TIME_WAIT。但非常危险，不推荐。

TCP 建立连接的本质含义

  1. 确定通信链路是否接通
  2. 告知彼此第一个发送字节的初始序列号，后续每个字节都以此为原点编号
  3. TCP 是双工的，连接建立后双方互为客户端和服务端
  4. 连接成功表示双方维护了一个可用连接的映射关系
  5. 建连三次握手，断连四次挥手（有时可合并为三次）

应用层保活：为什么有 TCP KeepAlive 还需要心跳？

尽管 TCP 有 KeepAlive 机制，但在实际应用中仍然需要应用层保活，原因如下：

  1. TCP KeepAlive 只检测连接的有无（死活），无法检测连接是否有效。”有效”意味着双方具备发送和接收消息的能力。
  2. 比如服务器负载过高无法响应业务，连接无效但仍然是 ESTABLISHED 状态，KeepAlive 检测不出。
  3. 路由器故障、网线被拔等情况下 KeepAlive 也会失效。
  4. KeepAlive 默认参数时间太长（默认 2 小时），调高频次则服务端压力太大。
  5. KeepAlive 是操作系统底层的被动机制，系统关闭死连接时不会主动通知上层应用。

保活机制相关参数：

  • tcp_keepalive_time=7200：保活时间 7200 秒（2 小时）
  • tcp_keepalive_intvl=75：每次检测间隔 75 秒
  • tcp_keepalive_probes=9：检测 9 次无响应认为不可达

Listen 参数 backlog 的意义

int listen (int socketfd, int backlog)
参数一 socketfd 为 socketfd 文件描述符
参数二 backlog，这参数在历史有一定的变化
在早期 Linux 内核 backlog 是 SYN 队列大小，也就是未完成的队列大小。
在 Linux 内核 2.2 之后，backlog 变成 accept 队列，也就是已完成连接建立的队列长度，所以现在通常认为 backlog 是 accept 队列。

粘包与拆包

TCP 是基于字节流的协议，没有消息边界的概念。当客户端连续发送数据包时，服务端可能收到粘在一起的数据包，这就是粘包/拆包问题。UDP 基于报文发送，首部有 16bit 的长度字段，应用层能很好地区分不同数据报文，所以不存在此问题。

Nagle 算法（SO_TCPNODELAY）：通过将缓冲区内的小封包自动合并为较大封包，减少小包数量以提高网络效率。但对时延敏感的场景需要关闭。

拆包方案：

  • 固定长度：每个包固定长度（如 1024 字节），不足时补空格
  • 分隔符：每个包末尾使用固定分隔符（如 \r\n），接收方按分隔符拆分合并
  • 长度字段：消息分为头部和消息体，头部保存整个消息的长度
  • 自定义协议：根据业务需求设计专用的协议格式

TCP 重传机制

TCP 的可靠性很大程度上依赖于重传机制。参考：https://coolshell.cn/articles/11564.html

经典超时重传

根据超时时间重传，在定时器 Timeout 内收不到 ACK，会按照 1、2、4…64 秒逐渐延迟重传。缺点是无法精确判断应该重传哪些包。

快速重传（Fast Retransmit）

接收方只 ACK 连续的序列号。比如 1 成功、2 丢失，收到 3、4、5 仍然 ACK=2。发送方连续收到 3 个相同的 ACK 即可开始重传，但仍不能解决”重传哪些包”的问题。

SACK（Selective Acknowledgment）

在快速重传基础上，接收方在正常 ACK 基础上追加 SACK（收到的不连续包的序列号范围），发送方据此精确判断需要重传哪些包。

D-SACK（Duplicate SACK）

使用 D-SACK 告诉发送方哪些数据被重复接收了，让发送方知道是发出去的包丢了还是回来的 ACK 丢了。

RTT 与 RTO 计算

TCP 需要精确估算往返时间（RTT）来设置合理的重传超时时间（RTO）。

  1. 经典采样算法：取最近几次 RTT 的加权平均
  2. Karn/Partridge 算法：忽略重传的 RTT，只取正常的 RTT，遇到重传则 RTO 翻倍
  3. Jacobson/Karels 算法（当前使用）：
SRTT = SRTT + α (RTT – SRTT)  —— 计算平滑RTT
DevRTT = (1-β)*DevRTT + β*(|RTT-SRTT|) ——计算平滑RTT和真实的差距（加权移动平均）
RTO= µ * SRTT + ∂ *DevRTT —— 神一样的公式
（其中：在Linux下，α = 0.125，β = 0.25， μ = 1，∂ = 4 ——这就是算法中的"调得一手好参数"，nobody knows why, it just works…） 最后的这个算法在被用在今天的TCP协议中（Linux的源代码在：tcp_rtt_estimator）。

流量控制与拥塞处理

滑动窗口

根据双方发送和接收数据的速度，动态计算出一个”窗口”，窗口为 0 就不传输数据。如果窗口为 0，发送方最多发 3 次 Zero Window Probe（ZWP）包查询接收方的窗口大小，如果 3 次都为 0，就 RST 连接。

糊涂窗口综合症（Silly Window Syndrome）

如果接收方太忙来不及取走数据，窗口会越来越小。为解决此问题：

  • 接收端问题：窗口缩减为 0 时中断连接
  • 发送端问题：采用 Nagle 算法，累积够数据再发送（会产生粘包，需应用层处理）

拥塞处理（Congestion Handling）

拥塞控制的核心思想：慢启动，逐渐加速，预警拥堵主动减速，拥塞发生快速恢复。

慢启动阶段根据回复 ACK 的速度调节窗口大小。参考：http://www.52im.net/thread-515-1-1.html

数据包结构与最大长度

  • UDP 包大小 = 1500 - IP头(20) - UDP头(8) = 1472 Bytes
  • TCP 包大小 = 1500 - IP头(20) - TCP头(20) = 1460 Bytes。MSS（Maximum Segment Size）即 TCP 数据包每次能传输的最大数据分段，通常为 1460。通信双方会取双方 MSS 值的最小值作为此次连接的最大 MSS。
  • IP 层：IP 包大小由 MTU 决定。MTU 值越大封包越大，理论上可增加传送速率，但太大会增加传送出错概率。以太网 MTU 默认 1500，Internet 默认 576 字节。

控制位标志符详解

UDP 头部结构

TCP 头部结构

TCP 的六个控制位含义如下，它们掌控着连接、传输和断开的全部行为：

  1. URG：紧急指针有效位
  2. ACK：只有 ACK=1 时确认序列号字段才有效（除了 SYN=0 时，ACK 必须=1）
  3. PSH：为 1 时要求接收方尽快将数据段送达应用层
  4. RST：为 1 时通知重新建立 TCP 连接
  5. SYN：同步序号位，建立连接时设为 1
  6. FIN：发送端完成发送任务，断开连接时设为 1

Seq 与 ACK 序号机制

序号和确认号是 TCP 可靠传输的核心，要区别于控制位中的 ACK 和 SYN：

  1. 两端分别产生 Seq，Ack = Seq + 1，Sequence Number 缩写为 ISN（Initial Sequence Number）
  2. 乱序、重传、丢包等问题都依赖于 Seq 解决
  3. ISN 与一个假时钟绑定，每 4 微秒加 1，直到超过 2^32 后从 0 开始
  4. 建立连接初始化 Seq 时控制位 SYN=1，否则为 0
  5. 控制位 ACK=1 时 Ack 字段才有效
  6. 重新建立连接后 Seq 重新随机生成，上次残留的数据包会被忽略，缓冲区数据也会清除

TCP 状态详解

  • LISTEN → SYN-SEND → SYN-RECEIVED → ESTABLISHED
  • FIN-WAIT-1 → FIN-WAIT-2 → CLOSING/CLOSE-WAIT → LAST-ACK → TIME_WAIT → CLOSED

Socket 写入成功的含义

在应用层编程时，send/write 的返回值含义需要正确理解：

  1. send/write 只是表示把数据写入 socket 缓冲区，之后操作系统会遵循 TCP 协议进行发送。发送实际上没有”失败状态”而是”收不到 ACK”，收不到 ACK 会触发重传。如果连接有各种错误（缓冲区满、连接状态不对），send/write 时就会直接返回错误。进入缓冲区的数据如果重传多次仍然失败，会强制断连。
  2. 如果缓冲区已满（发送窗口为 0），阻塞 socket 的 send 会一直阻塞直到有空闲空间；非阻塞 socket 会直接返回 -1，errno 设为 EAGAIN。
  3. 理论上连接建立后，拿到合法 Seq 就可以一直传输数据直到有明确错误或一直收不到 ACK。
  4. I/O 缓冲区在每个 TCP 套接字中单独存在，创建时自动生成。关闭套接字仍会继续传送输出缓冲区的遗留数据，但会丢失输入缓冲区的数据。

C10K 问题本质是操作系统问题：传统的同步阻塞 I/O 模型中，创建过多进程/线程导致数据拷贝频繁、上下文切换消耗大，操作系统崩溃。解决关键是减少 CPU 等核心资源消耗。Epoll 成为 C10K killer——FreeBSD 推出 kqueue，Linux 推出 epoll，Windows 推出 IOCP。Nginx、libevent、Node.js 都是 Epoll 时代的产物。

NAT 与 P2P 打洞

  • NAT（Network Address Translator）：IP 地址绑定（内部与公网）
  • NAPT（Network Address/Port Translator）：IP 与端口共同绑定

P2P 打洞原理：通过中间服务器 S “伪装”连接，获取 P1 与 P2 的公网 IP+端口组合，然后 P1 与 P2 分别伪装 S 与彼此建立连接。

常用协议

  • MQTT（Message Queuing Telemetry Transport）：IBM 开发的即时通讯协议，有可能成为物联网的重要组成部分
  • XMPP：可扩展消息与出席协议
  • JSON：轻量级数据交换格式
  • Protobuf：序列化后生成的二进制消息非常紧凑，采用 Varint 编码——用一个或多个字节表示数字，值越小使用越少的字节数

常见术语与缩写

  • MSL：Maximum Segment Lifetime
  • MSS：Maximum Segment Size
  • MTU：Maximum Transmission Unit
  • TCB：Transmission Control Block
  • TTL：Time To Live
