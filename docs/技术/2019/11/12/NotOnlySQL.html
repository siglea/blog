NoSQL数据库全景解析：从存储原理到技术选型实战

日期： 2019年11月12日

在大数据时代，传统的关系型数据库在面对海量数据、高并发读写和灵活的模式变更等需求时逐渐力不从心。NoSQL（Not Only SQL）数据库应运而生，它们以不同的数据模型和存储引擎解决了 RDBMS 难以应对的问题。本文将从存储原理出发，系统分析 Redis、MongoDB、Cassandra、HBase 等主流 NoSQL 数据库的特点和适用场景，帮助你在技术选型时做出明智的决策。

存储的核心原则

谈到存储，效率才是最核心的关注点，容量并非首要问题。无论是消息队列还是其他需要高效率的存储系统，底层利用的核心原理都是一致的：

  1. 随机写转换成顺序写：机械硬盘一次读写时间 = 寻道时间 + 旋转延迟 + 读取数据时间。寻道时间最长，顺序写只需一次寻道，效率远高于随机写。
  2. 集中刷盘：每次刷盘都会进行系统调用，无论机械硬盘还是 SSD，按一定块大小刷盘都比小数据刷盘效率更好。

分布式存储的两个核心问题

对于分布式存储系统，原本存储在一台机器上的数据现在要存放在多台机器上，必须解决两个核心问题：

  • 数据分片（Sharding/Partition）：将数据集合理拆分成多个分片，每台机器负责若干分片，突破单机容量限制，提升整体访问能力，同时降低单个分片故障的影响范围。
  • 数据复制（Replica）：分片无法解决单机故障丢数据的问题，必须通过冗余实现高可用。副本机制也是提升系统吞吐、解决热点问题的重要手段。

分片和副本是正交的，可以只用其中一种或都用，但生产环境通常同时使用——分片解决规模和扩展性问题，副本解决可靠性和可用性问题。

从客户端角度看，分片和副本可以归结为同一个问题：请求路由——请求应该发送给哪台机器来处理。

CAP 理论

CAP 理论是分布式系统设计的基石，任何分布式系统最多只能同时满足一致性（Consistency）、可用性（Availability）和分区容错性（Partition Tolerance）中的两个。不同的 NoSQL 数据库根据自身定位在 CAP 三角中做出了不同的取舍。

NoSQL 的由来

传统关系型数据库（RDBMS）存在以下局限：

  • 高可用 ≠ 分布式：传统 RDB 没有实现真正意义上的分布式，即便是分库分表也仅是”伪分布式”，不具备数据多分区备份等特性
  • 主从同步本质是全量备份：不是分布式要求的可伸缩
  • Schema 固定：数据表结构固定，扩展成本高
  • 存储模型差异：RDB 以行为核心，NoSQL 以列为核心（HBase/Cassandra）或 KV 为核心（Redis）

1946 年第一台通用计算机诞生，直到 1970 年 RDBMS 出现才有了通用数据存储方案。21 世纪 DT 时代让数据容量成为最棘手的问题，Google 于 2006 年提出了 Bigtable，2009 年技术大会上 NoSQL 一词被正式提出，至今已有 225 种解决方案。

NoSQL 与 RDBMS 的核心区别在于：无模式的灵活性（支持灵活的模式变更）和可伸缩性（原生分布式，解决读写和容量扩展问题）。

NoSQL 分类

NoSQL 数据库可分为四大类，各自使用不同的数据结构，适用场景各异：

  • Key-Value Store：Redis、Memcached、Tair
  • Document Store：MongoDB
  • Column Family：Cassandra、HBase
  • Graph Database：Neo4J

关系型 vs 非关系型的应用场景

关系型数据库适合结构化数据（如用户账号、地址）：

  1. 需要结构化查询（如 JOIN）
  2. 数据规模和增长速度通常可预期
  3. 需要事务性和一致性保证

NoSQL 适合非结构化数据（如文章、评论）：

  1. 通常用于模糊处理，如全文搜索、机器学习
  2. 数据量海量，增长速度难以预期
  3. 具有无限（或接近）伸缩性
  4. 按 Key 获取数据效率很高，但 JOIN 等结构化查询支持较差

核心数据结构：从 BST 到 LSM

理解存储引擎必须理解其底层使用的数据结构。以下是常用的树形结构及其特点：

  • AVL 树：要求每个节点的左右子树高度差不超过 1。高度差越小查找效率越高，但调整成本高。
  • 红黑树：自平衡二叉搜索树，从根到叶子的最长路径不超过最短路径的 2 倍。特性包括：根节点是黑色、每个叶子节点是黑色空节点（NIL）、红色节点的子节点都是黑色、任一节点到其叶子的所有路径包含相同数目的黑色节点。
  • 2-3 树：每个节点有 2-3 个子节点
  • B 树：M 阶树，每个节点最多 M 个子节点（操作系统文件目录）
  • B+ 树：内部节点不保存数据，能在内存中存放更多索引，增加缓存命中率。叶子节点相连便于遍历，数据具有顺序性，便于区间查找。
  • LSM 树：
    • 原理类似 JVM 的复制/压缩整理算法——新数据不插入旧位置，而是写入新空间然后与旧数据合并，最老的数据直接清除，保证磁盘上相邻数据都是顺序的
    • 内存中有 memtable（接受实时写）和 immutable memtable（写满了等待向磁盘 flush 或合并），磁盘内有分层的 SSTable
    • LSM 快的原因是写入在内存，之后异步合并写入磁盘（B/B+ 树的写入操作是在磁盘同步进行的）
    • 使用 LSM 的系统：Cassandra、HBase、LevelDB、RocksDB、BigTable
    • 参考：https://cloud.tencent.com/developer/news/340271
    • 算法细节：https://www.cnblogs.com/siegfang/archive/2013/01/12/lsm-tree.html

Redis

Redis 是最广泛使用的内存数据库，以其丰富的数据结构和极高的读写性能著称。

核心特性

  • 主要使用跳表（Skip List）作为有序集合的底层数据结构
  • 持久化机制：
    • RDB（快照）：相当于全量数据备份
    • AOF（写命令日志）：相当于 WAL（Write-Ahead Log）类似的命令日志
    • 参考：https://www.jianshu.com/p/cbe1238f592a

淘汰策略

volatile-lru #从设置了过期时间的数据集中，选择最近最久未使用的数据释放；
allkeys-lru #从数据集中(包括设置过期时间以及未设置过期时间的数据集中)，选择最近最久未使用的数据释放；
volatile-random #从设置了过期时间的数据集中，随机选择一个数据进行释放；
allkeys-random #从数据集中(包括了设置过期时间以及未设置过期时间)随机选择一个数据进行入释放；
volatile-ttl #从设置了过期时间的数据集中，选择马上就要过期的数据进行释放操作；
noeviction #默认值，不删除任意数据(但redis还会根据引用计数器进行释放),这时如果内存不够时，会直接返回错误。

事务支持

watch key1 key2 ... #监视一或多个key,如果在事务执行之前，被监视的key被其他命令改动，则事务被打断 （ 类似乐观锁 ）
multi #标记一个事务块的开始（ queued ）
exec #执行所有事务块的命令 （ 一旦执行exec后，之前加的监控锁都会被取消掉 ）　
discard #取消事务，放弃事务块中的所有命令
unwatch #取消watch对所有key的监控

高可用架构

  • Sentinel 模式（传统主从高可用）：多个 Sentinel 监控集群，Master 故障后选举主 Sentinel，再从 Slave 中选出新 Master，通知客户端节点变更。参考：https://www.cnblogs.com/duanxz/p/4701831.html
  • Twemproxy/nutcraker（Redis 的”分库分表方案”）：能自动分片、方便水平扩展，但移除节点的数据会丢失，动态增加节点时不会对已有数据做重分布。
  • Codis（豌豆荚开源）：ZooKeeper 存放路由表和元数据，Codis-Config 是集成管理工具，Codis-Proxy 是兼容 Redis 协议的无状态代理，Codis-Redis 基于 Redis 2.8 二次开发加入 slot 支持。
  • Redis Cluster（现代分布式集群）：官方原生分布式方案，去中心化架构。

常用客户端

Redisson、Jedis、Lettuce

单线程与多线程

Redis 基于 Reactor 模式开发了文件事件处理器（多个套接字 + IO 多路复用 + 文件事件分派器 + 事件处理器）。因为文件事件分派器队列的消费是单线程的，所以 Redis 被称为单线程模型。

但实际上，Redis 4.0 开始就有多线程概念（后台删除对象等），Redis 6.0 加入了 Threaded IO（网络数据读写和协议解析的多线程），执行命令的核心模块仍然是单线程的。

MongoDB

MongoDB 是最流行的文档型 NoSQL 数据库，以 JSON/BSON 格式存储数据，模式灵活、查询能力强大。

典型应用场景

  • O2O/LBS 应用：使用 MongoDB 的地理位置查询，方便实现查找附近商家、骑手等功能
  • 物联网/车联网：实时采集的车辆轨迹、温湿度等大量机器数据，增长快、单表易超 500 万
  • 电商领域：利用嵌套文档存储不同商品的差异化属性（如上衣的肩宽/胸围 vs 裤子的臀围/裤长）

Map/Reduce

MongoDB 中的 Map/Reduce 主要用于批量处理和聚合操作。Map 函数调用 emit(key, value) 遍历集合中所有记录，将 key 与 value 传给 Reduce 函数处理。Map 和 Reduce 函数使用 JavaScript 编写，可通过 db.runCommand 或 mapreduce 命令执行。

Cassandra

Cassandra 是 Apache 旗下的分布式宽列存储数据库，结合了 Google Bigtable 的数据模型和 Amazon Dynamo 的分布式设计，特别适合写密集型场景。

设计哲学

核心原则是：不要期望 Cassandra 适应你的逻辑，而应该将你的逻辑设计得适合 Cassandra。在 RDBMS 中你得先设计完整的数据模型再考虑查询方式，而在 Cassandra 中可以先思考如何查询数据，然后提供这些数据。一切设计都是为了查询。

数据模型

Cassandra 应该被看作一个有索引的、面向行的存储系统。它的数据结构是一个多维稀疏哈希表——”稀疏”意味着每行可以有不同的列，不必像关系模型那样每行都有相同的列。

数据结构层次：[Keyspace][ColumnFamily][Key][Column]

  • Keyspace：键空间，类似关系型数据库中的 Database，定义了复制策略和复制因子
  • Column Family：列族，类似于表
  • Column：一组键值对

主键设计

Cassandra 的主键设计是最关键的部分，包含 5 种 Key：

  • Primary Key：唯一标识一行数据
  • Partition Key：决定数据在集群中的位置（通过一致性哈希）
  • Composite Key / Compound Key：复合主键
  • Clustering Key：决定同一分区内数据的排序

例如 PRIMARY KEY((col1, col2), col3, col4))，其中 (col1, col2) 是 Partition Key 用于数据定位，col3, col4 是 Clustering Key 用于排序。

CAP 定位

Cassandra 更偏向 AP（可用性 + 分区容错），提供可调一致性级别（HBase 是 CP）。属于宽表家族（HBase、Alibaba TableStore、Google BigTable）。Cassandra 为优异的写吞吐量做了特别优化。

查询约束

  • 第一主键只能用 = 号查询
  • 第二主键支持 = > < >= <=（但需加 ALLOW FILTERING）
  • 索引列只支持 = 号
  • 索引列支持 like，只有主键支持 group by
  • 排序必须有第一主键的 = 号查询，只能根据 Clustering Key 排序

应用场景

  • 产品目录和零售应用
  • 社交媒体分析和推荐引擎
  • Yelp 的广告分析系统、Spotify 的用户信息存储
  • Feeds 场景、滴滴/物联网订单轨迹场景
  • 饿了么的应用：https://zhuanlan.zhihu.com/p/42175864

HBase、Cassandra、MongoDB 横向对比

与 RDBMS 的本质区别

关系数据库磁盘存储是一行接一行，而列式存储是一列接一列。

存储结构对比

数据库 存储结构
MongoDB GridFS、JSON/BSON
HBase HRegionServer → HLog + HRegion → Store(MemStore + StoreFile + HFile) → HDFS
Cassandra LSM、HashNode(一致性Hash虚拟节点)、CommitLog、memtable、SSTable

HBase 和 Cassandra 都基于类似的 LSM 机制。

关键特性对比

特性 MongoDB HBase Cassandra
存储形式 Collection JSON put ‘t_user’,’1001’,’st1:age’,’18’ Column Family, Row, Name/Value/Timestamp
数据类型 丰富，类似SQL 只支持字符串 多种类型
分页查询 skip().limit() scan + STARTROW + LIMIT token() + limit
索引 全索引(BTree) 主要设计二级索引(B+树) Partition Key + Clustering Key
事务 不支持 行级事务 行级原子性和隔离性
一致性 强一致(CP) 强一致(CP) 最终一致/可调(AP)
JOIN 不支持 不支持(需借助工具) 不支持(用数据冗余)

HBase 的数据分片

HBase 的数据分片按表进行，以行为粒度，基于 RowKey 范围拆分，每个分片称为一个 Region。RS（RegionServer）与表是正交的：一张表的 Region 会分布到多台 RS 上，一台 RS 也会服务多张表的 Region。HBase 是水平拆分，一行数据要么属于 A Region 要么属于 B Region。

参考：https://zhuanlan.zhihu.com/p/47074785

TSDB：时间序列数据库

时间序列数据库（Time Series Database）专为时间序列数据优化，如 InfluxDB、OpenTSDB、TimescaleDB、Druid、TableStore、ClickHouse 等。

核心特点：持续高并发写入、无更新；数据压缩存储；低查询延时。广泛应用于物联网和互联网领域，实现对设备及业务服务的实时监控和预测告警。

总结

NoSQL 数据库并非要替代关系型数据库，而是在不同场景下提供更合适的解决方案。选型时需要考虑数据模型、一致性要求、读写比例、扩展性需求等因素。在实际架构中，往往是多种数据库配合使用——用 MySQL 存储核心业务数据，用 Redis 做缓存，用 Elasticsearch 做搜索，用 Cassandra/HBase 存储海量时序数据。
