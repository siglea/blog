分布式事务全解析：从CAP理论到主流框架选型

日期： 2019年11月12日

引言

在单体应用时代，数据库的本地事务就能满足绝大多数一致性需求。但随着微服务架构和分布式数据库的普及，一个业务操作可能横跨多个服务和多个数据库，本地事务力不从心，分布式事务应运而生。本文将从数据库本地事务出发，深入探讨 CAP 定理、BASE 理论，并对比分析主流的分布式事务解决方案。

参考案例：从一笔金币充值去思考分布式事务，五种方案详解！ https://mp.weixin.qq.com/s/dbQsfBdJSs5Ol70hCYCZMw

数据库本地事务简述

理解分布式事务之前，需要先回顾本地事务的 ACID 原则：

  1. 原子性（Atomicity）：事务中的操作要么全部成功，要么全部回滚
  2. 一致性（Consistency）：事务执行前后，数据库从一个一致状态转变为另一个一致状态
  3. 隔离性（Isolation）：并发事务之间互不干扰
  4. 持久性（Durability）：事务提交后，修改永久保存

以 MySQL InnoDB 引擎为例，ACID 的实现依赖日志和锁的配合：

  • 隔离性：通过数据库锁机制实现（行锁、间隙锁、MVCC）
  • 持久性：通过 Redo Log（重做日志）实现。事务提交前只需将 Redo Log 持久化，不必等待数据页落盘
  • 原子性和一致性：通过 Undo Log 实现。Undo Log 记录了数据操作前的备份，支持事务回滚

本地事务的本质是通过单一的事务资源管理器来统一协调调度。当数据和服务分布在多个节点上时，这种单一协调者的模式就不再适用了。

分布式事务产生的原因

分布式事务的产生根源在于数据和服务的分布：

  • Service 多节点部署：公司内部的用户服务、账户服务、订单服务等各个微服务独立部署
  • Resource 多节点：同一个服务的数据可能分布在不同的数据库节点上（分库分表）

当一个业务操作需要同时修改多个服务的数据（如下单扣库存 + 扣余额），就必须保证跨服务/跨库的数据一致性。

CAP 定理

CAP 定理（又称布鲁尔定理）是分布式系统设计的入门理论，指出一个分布式系统不可能同时满足以下三点：

  • C（Consistency，一致性）：所有节点在同一时间看到的数据是一致的。如果在某个节点更新了数据，其他节点立即能读到最新值即为强一致；否则为分布式不一致。
  • A（Availability，可用性）：非故障节点在合理的时间内返回合理的响应（不是错误或超时）。”合理的时间”意味着请求不能被无限期阻塞；”合理的响应”意味着结果是正确的。
  • P（Partition Tolerance，分区容错性）：当网络分区发生时，系统仍能继续工作。

三者不可兼得

在分布式系统中，网络无法 100% 可靠，分区是一个必然现象。因此：

  • 如果选择 CA 放弃 P：当分区发生时，为了保证一致性必须拒绝请求，但这与可用性矛盾。因此分布式系统理论上无法选择 CA 架构。
  • 选择 CP：放弃可用性，追求一致性和分区容错性。ZooKeeper 就是典型的 CP 系统。
  • 选择 AP：放弃强一致性，追求可用性和分区容错性。大多数互联网分布式系统选择这一路线。

需要注意的几点：

  • CAP 理论忽略了网络延迟——实际中节点间的数据同步总需要时间，因此”绝对一致”在现实中是不存在的
  • 选择 CP 不意味着完全放弃 A：分区发生的概率很小，大部分时间仍需保证 CA。即使分区发生，也应为后续的可用性恢复做准备（如通过日志补偿）

BASE 理论

BASE 是 Basically Available（基本可用）、Soft State（软状态）和 Eventually Consistent（最终一致性）的缩写，是对 CAP 中 AP 的进一步扩展。

  • 基本可用：分布式系统在出现故障时，允许损失部分可用功能，保证核心功能可用
  • 软状态：允许系统中存在中间状态（即数据暂时不一致），这个中间状态不影响系统整体可用性
  • 最终一致：经过一段时间后，所有节点的数据最终会达到一致状态

BASE 与 ACID 的对比：ACID 追求强一致性，BASE 通过牺牲强一致性来获得可用性，允许数据在一段时间内不一致但最终达到一致。BASE 解决了 CAP 理论中”无网络延迟”这一理想假设的局限——用软状态和最终一致性来适应现实中不可避免的延迟。

常用分布式事务框架

基于 XA 的 2PC / 3PC

本质是本地事务的加强版，通过引入事务协调者实现强一致性，依赖数据库对 XA 协议的支持，属于数据库资源层方案。优点是一致性保障强，缺点是性能开销大、存在阻塞和单点问题。

Saga

引入补偿机制实现最终一致性。每个参与者定义正向操作和对应的补偿（逆向）操作，当某一步失败时，按逆序执行已完成步骤的补偿操作。适合长事务场景。

TCC（Try-Confirm-Cancel）

将事务分为三个阶段：

  • Try：资源预留（如冻结库存、冻结余额）
  • Confirm：确认提交（扣减冻结的资源）
  • Cancel：取消释放（解冻资源）

TCC 对业务侵入性较大，需要业务方实现三个接口，但灵活性最高，性能也优于 XA。

本地消息表

在业务操作的同一个本地事务中写入消息表，由后台任务定期扫描消息表并发送消息给下游。下游处理成功后更新消息状态。这是一种经典的最终一致性方案。

RocketMQ 事务消息

阿里开源的 RocketMQ 提供了原生的事务消息支持，其核心是”半消息”机制——消息先发送到 MQ 但对消费者不可见，本地事务提交后再确认消息，MQ 将消息投递给消费者。

Seata

阿里开源的分布式事务解决方案，支持 AT、TCC、Saga 和 XA 四种模式。AT 模式对业务代码几乎零侵入，通过代理数据源自动生成补偿日志，是目前社区中最流行的分布式事务框架之一。

总结

分布式事务是分布式系统中最具挑战性的问题之一。理解 CAP 定理和 BASE 理论是做出正确架构决策的前提——在绝大多数互联网业务场景中，最终一致性是比强一致性更务实的选择。在具体方案选型上，需要根据业务对一致性的要求、性能预期和技术团队能力来权衡：对一致性要求极高且能接受性能损耗的场景选择 XA/2PC；需要灵活控制且团队能力强的场景选择 TCC；追求简单可靠的场景选择本地消息表或 RocketMQ 事务消息；想要开箱即用且低侵入的场景选择 Seata AT 模式。
