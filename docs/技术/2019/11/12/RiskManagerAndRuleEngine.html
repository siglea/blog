风控系统与规则引擎：架构设计与技术选型全解析

日期： 2019年11月12日

引言

互联网业务的快速发展带来了日益严峻的风险挑战——薅羊毛、恶意刷单、欺诈交易、账户盗用等问题层出不穷。风控系统（Risk Management System）作为业务安全的核心防线，需要在毫秒级的时间内完成风险判断并做出决策。而规则引擎（Rule Engine）则是风控系统中最核心的决策组件，它将业务规则从代码中解耦出来，使风控策略能够灵活配置、快速迭代。

风控系统的整体架构

一个完整的风控系统通常分为以下几层：

1. 数据采集层

风控的基础是数据。采集层负责从各个业务系统收集用户行为数据，包括：

  • 设备指纹：设备型号、操作系统、IP 地址、GPS 位置等
  • 行为数据：点击轨迹、停留时间、操作频率等
  • 业务数据：交易金额、收货地址、支付方式等
  • 外部数据：第三方征信、黑名单库、运营商数据等

采集方式通常包括客户端 SDK 埋点、服务端日志、消息队列实时推送等。

2. 特征工程层

原始数据需要经过加工才能用于风控决策。特征工程将原始数据转化为有业务含义的特征变量：

  • 统计特征：最近1小时交易次数、7天内登录设备数、30天交易总额等
  • 衍生特征：交易金额偏离历史均值的程度、IP 地理位置与常用地的距离等
  • 关联特征：同一设备关联的账户数、同一收货地址的订单数等

特征的实时计算通常依赖 Flink、Spark Streaming 等流处理框架，离线特征则通过 Hive、Spark 等批处理引擎计算。

3. 决策引擎层

这是风控系统的核心，负责根据特征数据执行风控策略并输出决策结果（通过、拒绝、人工审核等）。决策引擎通常包含：

  • 规则引擎：基于 if-then 规则的判断逻辑
  • 模型引擎：基于机器学习模型的评分系统
  • 名单系统：黑白灰名单的匹配

4. 处置与反馈层

根据决策结果执行相应的处置动作（拦截、限流、二次验证等），并将处置结果反馈回系统，形成数据闭环。

规则引擎详解

什么是规则引擎

规则引擎的核心思想是将业务规则从应用程序代码中分离出来，用特定的规则语言来定义和管理。当业务条件发生变化时，只需修改规则配置，无需重新部署应用。

一条典型的风控规则如下：

IF 用户注册时间 < 7天
   AND 单笔交易金额 > 5000元
   AND 收货地址与常用地址不同
THEN 标记为高风险，触发人工审核

主流规则引擎技术选型

引擎 特点 适用场景
Drools 功能最强大，支持 Rete 算法，有完整的规则管理平台 规则复杂、需要规则管理 UI 的企业级场景
Aviator 轻量高性能的表达式求值引擎，阿里开源 简单条件判断、高性能要求场景
Groovy 基于 JVM 的动态语言，灵活性极高 规则逻辑复杂、需要动态编程能力的场景
IKExpression 超轻量，适合简单表达式 简单数学和逻辑运算
QLExpress 阿里开源，语法简洁，性能优秀 电商风控、营销规则

性能对比（基于简单表达式求值的 benchmark）：

Aviator > QLExpress > Groovy > IKExpression > Drools

Drools 虽然性能不是最优，但胜在规则管理能力最强。在实际选型中，需要在性能和管理能力之间做权衡。

Drools 规则引擎实践

Drools 是最广泛使用的 Java 规则引擎，基于 Rete 算法进行高效的模式匹配。一个简单的 Drools 规则示例：

rule "高风险交易识别"
    when
        $tx : Transaction(amount > 5000, userAge < 7)
        $addr : Address(city != $tx.userCity)
    then
        $tx.setRiskLevel("HIGH");
        insert(new Alert($tx.getId(), "高风险交易"));
end

有赞团队在实践中采用 Drools 作为底层引擎，并在此基础上构建了可视化的规则配置平台，使运营人员能够直接配置风控规则，大大缩短了策略上线周期。

实时风控的技术架构

现代风控系统对实时性要求极高，典型的实时风控架构基于 Flink 构建：

事件流 → Kafka → Flink CEP → 规则引擎 → 决策结果 → 业务系统
                    ↕
               特征存储（Redis/HBase）

关键技术点：

  1. Flink CEP（Complex Event Processing）：用于检测复杂事件模式，例如”同一用户5分钟内在3个不同城市发起交易”
  2. Redis 特征缓存：将实时计算的特征存入 Redis，保证毫秒级读取
  3. Lambda 架构：实时层处理最新数据，离线层进行全量特征计算和模型训练

美团酒旅团队的实践表明，基于 Flink 的实时风控系统能够将决策延迟控制在 50ms 以内，同时支持每秒数万笔交易的吞吐量。

风控策略的层次设计

成熟的风控系统通常采用分层策略：

  1. 名单层：黑白名单快速过滤，命中即直接决策，耗时最短
  2. 规则层：基于业务经验的规则判断，可快速上线和调整
  3. 模型层：机器学习模型评分，捕获规则难以覆盖的复杂模式
  4. 人工审核层：对可疑案例进行人工复核，同时为模型提供标注数据
请求 → 名单匹配 → 规则引擎 → 模型评分 → 综合决策
         ↓           ↓           ↓
       直接通过/拒绝  风险标记     风险评分

总结

风控系统是技术与业务深度融合的领域。规则引擎让风控策略的管理变得灵活高效，而 Flink 等流处理框架则提供了实时计算的能力。在技术选型上，没有银弹——Drools 适合规则复杂且需要可视化管理的场景，Aviator/QLExpress 适合追求极致性能的场景。关键是根据业务特点，设计分层的、可演进的风控架构，在安全性和用户体验之间找到平衡。
