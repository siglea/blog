深入理解 JVM 垃圾回收机制：算法、收集器与调优实践

日期： 2019年11月04日

Java 虚拟机（JVM）的垃圾回收（GC）机制是 Java 程序员必须掌握的核心知识之一。理解 GC 的工作原理不仅能帮助我们写出更高效的代码，更是排查线上内存问题、进行性能调优的基础。本文将从垃圾回收算法、分代模型、各种收集器的特性，到 JVM 内存模型进行系统梳理。

一、GC 收集算法

在实际 JVM 实现中，通常是多种算法组合使用，针对不同内存区域选择最合适的算法。

1.1 标记-复制（Mark-Copy）

将存活对象复制到新的内存区域，然后整体回收老区域。优点是没有碎片，缺点是需要额外的内存空间。

适用收集器：Serial New、Parallel New、Parallel Scavenge New、G1

1.2 标记-清除（Mark-Sweep）

标记所有存活对象，然后清除未标记的对象。实现简单，但会产生内存碎片。

适用收集器：CMS（产生碎片后默认选择 Serial Old 配合进行最终整理）

1.3 标记-整理（Mark-Compact）

标记存活对象后，将它们向一端移动，然后清理边界之外的所有空间。兼顾了无碎片和内存利用率，但移动对象的开销较大。

适用收集器：Serial Old、Parallel Old、G1

二、分代模型

为了更高效地进行垃圾回收，JVM 将堆内存划分为不同的代。G1 收集器中的分区（Region）设计也是为了充分发挥多核并发优势。

2.1 内存分区

  • Young Generation（年轻代）
    • Eden（伊甸园）：新对象首先在此分配
    • Survivor（幸存者区）：分为 From 和 To 两个区（也称 S0/S1）
  • Tenured/Old Generation（老年代）：长期存活的对象晋升至此
  • Permanent Generation（永久代）：Java 8 之前存放类元数据
  • Metaspace（元空间）：Java 8 之后取代永久代，使用本地内存

2.2 分代回收的工作流程

分代回收器将堆分为老年代和年轻代，默认比例为 2:1。年轻代内部 Eden、To Survivor、From Survivor 的默认比例是 8:1:1。

执行流程如下：

  1. 新对象在 Eden 区分配
  2. Eden 区满时触发 Minor GC，将 Eden + From Survivor 中存活的对象复制到 To Survivor 区
  3. 清空 Eden 和 From Survivor 分区
  4. From Survivor 和 To Survivor 交换角色
  5. 每次在 Survivor 区间移动时存活的对象，年龄 +1
  6. 当年龄达到阈值（默认 15）时，晋升到老年代；大对象也会直接进入老年代
  7. 老年代空间占用到达阈值后触发 Major GC / Full GC，一般使用标记-整理算法

以上过程循环往复，构成了整个分代垃圾回收的执行流程。

三、垃圾收集器详解

JVM 提供了多种垃圾收集器，它们在吞吐量、停顿时间、内存开销等方面各有侧重。

3.1 Serial 收集器（复制算法）

新生代单线程收集器，标记和清理都是单线程。优点是简单高效，适合单核或小内存的客户端应用。

3.2 ParNew 收集器（复制算法）

新生代并行收集器，实际上是 Serial 收集器的多线程版本。在多核 CPU 环境下有着比 Serial 更好的表现，常与 CMS 配合使用。

3.3 Parallel Scavenge 收集器（复制算法）

新生代并行收集器，追求高吞吐量。吞吐量 = 用户线程时间 /（用户线程时间 + GC 线程时间）。具有 GC 自适应调节策略（GC Ergonomics），通过调整新生代大小等参数，在合适的时间和耗时之间达到平衡。适合后台计算等对交互响应要求不高的场景。

3.4 Serial Old 收集器（标记-整理算法）

老年代单线程收集器，是 Serial 收集器的老年代版本。

3.5 Parallel Old 收集器（标记-整理算法）

老年代并行收集器，吞吐量优先，是 Parallel Scavenge 收集器的老年代版本。

3.6 CMS 收集器（标记-清除算法）

全称 Concurrent Mark Sweep，老年代并发收集器，以获取最短回收停顿时间为目标。具有高并发、低停顿的特点，是追求最短 GC 停顿时间场景的首选。需要注意的是，CMS 与 Parallel Scavenge 不可组合使用。

3.7 G1 收集器（标记-整理算法）

G1（Garbage First）是 JDK 1.7 引入的新一代收集器，基于标记-整理算法实现，不会产生内存碎片。与前六种收集器不同，G1 回收的范围是整个 Java 堆（包括新生代和老年代）。

G1 的关键特性：

  • 可以通过 -XX:MaxGCPauseMillis 设置可控的停顿时间目标
  • 通过 -XX:G1HeapRegionSize 设置 Region 大小（2 的幂次，尽可能分为 2048 份）
  • 根据算法判断哪个 Region 回收价值最高，优先回收（这也是”Garbage First”名称的由来）
  • Mixed GC 不是 Full GC，它只能回收部分老年代 Region。如果 Mixed GC 无法跟上内存分配速度，导致老年代填满，就会退化为 Serial Old 进行 Full GC

3.8 ZGC

ZGC 是在 64 位 JVM 中使用的新一代低延迟收集器，其核心创新在于：

  • 使用 Colored Pointer（染色指针） 标记 Finalizable、Remapped、Marked1、Marked0 等状态，节省遍历 RememberSet 的时间，并可执行不同的处理逻辑
  • Region 大小不固定，不同大小的对象使用不同的 Page，方便整理回收
  • 大部分对象标记和对象转移都可以与应用线程并发执行

3.9 收集器组合（共 7 种）

组合 特点
Parallel Old + Parallel Scavenge 高吞吐量
G1 新生代+老年代统一管理
CMS + Serial / CMS + ParNew（2 种） CMS 短停顿 + Serial Old 兜底
Serial Old + Serial 单线程
Serial Old + ParNew 混合
Serial Old + Parallel Scavenge 混合

四、安全点与安全区域

GC 不能在程序运行的任意位置暂停线程，而是需要线程运行到”安全点（Safepoint）”才能进行 STW（Stop The World）操作。安全区域（Safe Region）则是指在一段代码中，引用关系不会发生变化的区域，线程在安全区域内可以安全地进行 GC。

五、空间分配担保

在发生 Minor GC 之前，JVM 会检查老年代最大可用的连续空间是否大于新生代所有对象总空间。如果不满足条件，则需要进行空间分配担保判断，决定是否直接进行 Full GC。

六、JVM 内存模型

理解 GC 机制的前提是清楚 JVM 的内存布局：

  • 堆内存（Heap）：存放对象实例（new 的对象），是 GC 管理的主要区域，几乎所有的对象实例都在这里分配内存。
  • 方法区（Method Area）：也称”永久代”（Java 8 前），用于存储虚拟机加载的类信息、常量、静态变量，是各个线程共享的内存区域。
  • 虚拟机栈（JVM Stack）：线程私有，生命周期与线程相同。每个方法会创建一个栈帧，包含局部变量表、操作数栈、动态连接、方法出口等信息。
  • 本地方法栈（Native Stack）：与虚拟机栈类似，但为 Native 方法服务。
  • 程序计数器（PC Register）：记录当前线程执行程序的位置，用于循环、分支、方法跳转、异常处理和线程恢复。线程私有，保证切换后能恢复到正确位置。
  • 直接内存（Direct Memory）：并非虚拟机内存的一部分。JDK 1.4 引入的 NIO 可以调用 Native 方法直接分配堆外内存，不受堆大小限制。
