JVM 常用监控与诊断工具详解：从 jps 到 VisualVM

日期： 2020年03月27日

在 Java 应用的开发和运维过程中，JVM 监控与诊断工具是排查性能问题、内存泄漏和线程死锁的必备利器。JDK 自带了一套功能强大的命令行工具，包括 jps、jstat、jinfo、jmap、jhat、jstack 等，配合 JConsole 和 VisualVM 等图形化工具，可以覆盖绝大多数 JVM 诊断场景。本文将逐一介绍这些工具的用法和实战示例。

一、jps：JVM 进程查看工具

jps（JVM Process Status Tool）用于列出当前系统中所有正在运行的 Java 进程，是其他所有 JVM 诊断工具的起点——你需要先知道目标进程的 PID。

jps
31856 xhs-push.jar
22640 Jps

jps -q
31856
22640

[root@VM_0_17_centos ~]# jps -m
31856 xhs-push.jar --push.queue=message_queue_low
22825 Jps -m

[root@VM_0_17_centos ~]# jps -l
31856 /home/web/xhs-push/xhs-push.jar
22886 sun.tools.jps.Jps

[root@VM_0_17_centos ~]# jps -v
31856 xhs-push.jar -Xms256m -Xmx512m -XX:+PrintGCDateStamps -XX:+PrintGC -Xloggc:...
22932 Jps -Dapplication.home=/usr/java/jdk1.8.0_171-amd64 -Xms8m

常用参数说明：

  • -q：只输出 PID，省略主类名
  • -m：输出传给 main 方法的参数
  • -l：输出主类的完整包名或 JAR 路径
  • -v：输出 JVM 启动参数

二、jstat：JVM 统计监控工具

jstat（JVM Statistics Monitoring Tool）可以实时监测系统资源占用与 JVM 运行情况，支持类加载、GC、编译等多维度的统计信息。

2.1 命令语法

Usage: jstat -help|-options
       jstat -<option> [-t] [-h<lines>] <vmid> [<interval> [<count>]]

参数说明：

  • option：统计选项，如 -gcutil、-class 等
  • -t：输出时间戳
  • vmid：目标 Java 进程号
  • interval：采样间隔（秒或毫秒）
  • count：打印次数，缺省则无限打印

2.2 GC 输出字段含义

S0  — Heap上的 Survivor space 0 区已使用空间的百分比
S1  — Heap上的 Survivor space 1 区已使用空间的百分比
E   — Heap上的 Eden space 区已使用空间的百分比
O   — Heap上的 Old space 区已使用空间的百分比
P   — Perm space 区已使用空间的百分比
YGC — 从应用程序启动到采样时发生 Young GC 的次数
YGCT– 从应用程序启动到采样时 Young GC 所用的时间(单位秒)
FGC — 从应用程序启动到采样时发生 Full GC 的次数
FGCT– 从应用程序启动到采样时 Full GC 所用的时间(单位秒)
GCT — 从应用程序启动到采样时用于垃圾回收的总时间(单位秒)

2.3 可用选项一览

## jstat -options
    -class 类加载统计
    -compiler 编译统计 
    -gc
    -gccapacity
    -gccause Allocation Failure
    -gcmetacapacity 元数据空间统计
    -gcnew
    -gcnewcapacity
    -gcold
    -gcoldcapacity
    -gcutil 总结垃圾回收统计
    -printcompilation JVM编译方法统计

三、jinfo：Java 配置信息查看

jinfo 用于查看和动态修改 JVM 配置参数，在不重启应用的情况下调整某些 JVM 参数时非常有用。

 -flags               to print VM flags
 -sysprops            to print Java system properties
 <no option>          to print both of the above

四、jmap：内存映射工具

4.2 对象统计与堆转储

## 示例2
jmap -F -histo 16464 > 16464.log
-histo[:live] 打印每个class的实例数目,内存占用,类全名信息. VM的内部类名字开头会加上前缀"*". 如果live子参数加上后,只统计活的对象数量. 

## 示例2
jmap -dump:format=b,file=outfile 3024可以将3024进程的内存heap输出出来到outfile文件里，再配合MAT（内存分析工具）。

-histo 可以快速定位哪些类的实例占用了最多内存；-dump 则生成完整的堆转储文件，用于离线分析。

五、jhat：堆转储分析工具

jhat（JVM Heap Analysis Tool）用于分析 jmap 生成的堆转储文件，并启动一个 HTTP 服务器供浏览器查看分析结果。

jhat outfile
http://localhost:7000

在实际工作中，更推荐使用 Eclipse MAT（Memory Analyzer Tool）来分析大型堆转储文件，因为 MAT 支持 OQL 查询、支配树分析等高级功能，比 jhat 更加强大。

六、jstack：线程堆栈分析工具

jstack 是排查线程问题（死锁、CPU 飙高、线程阻塞）的核心工具，它打印出指定 Java 进程中所有线程的堆栈信息。

# 示例
jstach 5940

排查 CPU 飙高的标准流程

这是线上环境中最常用的排查套路：

# 查看该进程中哪个线程占用大量 CPU，执行 top -H -p 11281 
# 找到执行时间最长的线程id：19279
# printf "%x\n" 19279 #输出十六进制 4b4f
# jstack 19268 |grep 4b4f
# 接下来我们将进程中的所有线程输出到一个文件中，执行：jstack  11281> jstack.txt
# 在进程中查找对应的线程 ID，执行：cat jstack.txt | grep -i 2c6d
# jstack -l 14716 |grep deadlock # 查看死锁

流程总结：top -Hp 找到高 CPU 线程 → 将线程 ID 转为十六进制 → 在 jstack 输出中定位该线程的堆栈 → 分析具体代码位置。

七、图形化工具

JConsole

JConsole 是 JDK 自带的 JMX 监控工具，提供了内存、线程、类加载、MBean 等多维度的实时监控。适合开发环境使用。

VisualVM

VisualVM 是功能更全面的 JVM 监控工具，集成了 jstat、jmap、jstack 等多个命令行工具的功能，还支持插件扩展（如 Visual GC 插件），是本地开发和测试环境的首选工具。

八、其他工具

javap：字节码查看工具

javap 可以反编译 .class 文件，查看 Java 字节码指令。这对于理解 JVM 执行机制、分析编译器优化非常有帮助。

HSDIS：JIT 反汇编插件

HSDIS 是 HotSpot VM 的反汇编插件，可以将 JIT 编译器生成的本地机器码反汇编为可读的汇编指令，用于深度性能分析。

九、总结

JVM 的监控和诊断工具链非常完善，从进程查看（jps）、统计监控（jstat）、配置查看（jinfo）、内存分析（jmap + jhat）、线程分析（jstack），到图形化的 JConsole 和 VisualVM，形成了一个完整的诊断闭环。掌握这些工具的核心用法，能够在生产环境中快速定位和解决大部分 JVM 相关的性能问题。

建议在日常开发中养成使用这些工具的习惯，而不是等到线上出现故障时才临时查阅文档。
