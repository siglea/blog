Docker 容器技术入门：从原理到实践，再到容器编排生态

日期： 2020年03月12日

Docker 是当今最流行的容器化技术，它彻底改变了应用的开发、交付和运行方式。本文将从 Docker 的底层原理出发，通过常用命令实战，逐步深入到容器编排和云平台生态的全景概览。

一、Docker 底层原理

Docker 的核心技术基础来自 Linux 内核的两个关键特性：

  • cgroups（Control Groups，2006 年 Linux 推出）：用于限制、记录和隔离进程组使用的物理资源（CPU、内存、磁盘 IO 等）。cgroups 使得 Docker 能够为每个容器分配有限的系统资源。
  • namespace（命名空间）：提供了进程级别的资源隔离，包括 PID、Network、Mount、User 等命名空间。namespace 使得容器内的进程”看到”的是一个独立的操作系统环境。

这两项技术的结合，使得 Docker 容器既能实现进程级别的隔离，又几乎没有虚拟机的性能开销。

二、Docker 基础命令实战

2.1 Hello World

Docker 的第一步，总是从运行一个简单的容器开始：

docker run hello-world
docker run ubuntu:15.10 /bin/echo "Hello world"

2.2 拉取镜像

从 Docker Hub 拉取官方镜像：

docker pull library/hello-world

2.3 运行 Nginx

docker run --name my_nginx_v1 -p 8080:80 -d nginx:v1

参数说明：

  • --name my_nginx_v1：指定容器名称
  • -p 8080:80：将本地 8080 端口映射到容器内部的 80 端口
  • -d：在后台运行容器

参考：https://www.runoob.com/docker/docker-install-nginx.html

2.4 运行 Spring Boot 应用

docker run -p 8080:8080 -t springboot/spring-boot-docker

参考：http://www.ityouknow.com/springboot/2018/03/19/spring-boot-docker.html

2.5 查看运行中的容器

docker ps
docker ps -a

输出字段说明：

字段 含义
CONTAINER ID 容器 ID
IMAGE 使用的镜像
COMMAND 启动容器时运行的命令
CREATED 容器的创建时间
STATUS 容器状态
PORTS 端口信息和连接类型（tcp/udp）
NAMES 容器名称

容器的 7 种状态：created（已创建）、restarting（重启中）、running（运行中）、removing（迁移中）、paused（暂停）、exited（停止）、dead（死亡）。

2.6 查看容器日志

docker logs 2b1b7a428627

2.7 启动 / 重启 / 停止容器

docker start amazing_cori
docker restart amazing_cori
docker stop 2b1b7a428627

2.8 删除容器和镜像

docker rm 20ca2a70ee13     # 删除容器
docker rmi ec49dbd13cf5    # 删除镜像

2.9 run 与 start 的区别

这是初学者容易混淆的概念：

  • docker run 后面指定的是一个镜像，它会基于镜像创建一个新容器并启动
  • docker start 后面指定的是一个容器，它启动的是一个之前已经创建过的容器

简言之：run = 创建 + 启动，start = 启动已有容器。

三、Dockerfile

Dockerfile 是构建 Docker 镜像的蓝图，通过一系列指令定义了镜像的构建过程。掌握 Dockerfile 的编写是容器化应用的基本功。

参考：

  • Dockerfile 命令详解 http://www.ityouknow.com/docker/2018/03/15/docker-dockerfile-command-introduction.html
  • Docker 系列文章 http://www.ityouknow.com/docker.html

四、Compose、Machine 与 Swarm

Docker 生态中有三个重要的编排工具：

  • Docker Compose：用于定义和运行多容器应用，通过 docker-compose.yml 文件声明式地管理服务、网络和卷
  • Docker Machine：用于在本地或远程主机上自动化安装 Docker 引擎
  • Docker Swarm：Docker 原生的集群管理和编排工具，提供服务发现、负载均衡和滚动更新等功能

参考：http://www.ityouknow.com/docker/2018/03/22/docker-compose.html

五、容器编排与云平台生态

容器技术的发展催生了丰富的编排和云平台生态。理解这些平台之间的关系和差异，有助于在实际项目中做出合理的技术选型。

5.1 Cloud Foundry 与 Kubernetes

Cloud Foundry 是一个开源的 PaaS 平台，专注于应用的部署和运维。Kubernetes 则是容器编排的事实标准，专注于容器的调度和管理。两者可以结合使用——Cloud Foundry 提供应用抽象，Kubernetes 提供基础设施编排。

参考：

  • Cloud Foundry和Kubernetes的区别 https://blog.csdn.net/qq_30154571/article/details/84955097
  • CF/K8s结合简史 https://blog.csdn.net/zhanggongsw/article/details/79948420

5.2 阿里飞天云 Apsara Stack 与 OpenStack

  • Apsara Stack：阿里云的企业级混合云平台，底层基于飞天操作系统
  • OpenStack：开源的云计算基础设施平台，提供 IaaS 层的虚拟化资源管理

5.3 Kubernetes 与 OpenStack

Kubernetes 和 OpenStack 并不是竞争关系，而是互补的：OpenStack 管理虚拟机级别的基础设施，Kubernetes 管理容器级别的应用编排。在很多企业中，Kubernetes 运行在 OpenStack 提供的虚拟机之上。

5.4 容器编排三巨头：Kubernetes vs Docker Swarm vs Mesos Marathon

平台 定位 特点
Kubernetes 容器编排标准 功能最丰富，生态最完善，学习曲线较陡
Docker Swarm Docker 原生编排 简单易用，与 Docker 集成度高，功能相对基础
Mesos Marathon 通用资源调度 适合超大规模集群，可同时管理容器和非容器负载

目前 Kubernetes 已经成为容器编排的事实标准，Docker Swarm 适合小规模或入门场景，Mesos 在大规模数据中心场景中仍有优势。

5.5 国内外 PaaS 服务一览

PaaS（Platform as a Service）层为开发者提供了应用运行时环境，屏蔽了底层基础设施的复杂性。

参考：https://www.jianshu.com/p/a4b9be650fa4

总结

Docker 从底层的 cgroups + namespace 实现了轻量级的进程隔离，通过镜像机制解决了环境一致性问题，再通过 Compose/Swarm/Kubernetes 等工具实现了从单机到集群的编排管理。理解 Docker 的原理和生态，是现代后端工程师的必备技能。

在技术选型时，建议遵循”由简到繁”的原则：单机开发用 Docker CLI，多容器编排用 Docker Compose，集群化部署用 Kubernetes。
