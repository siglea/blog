Netty网络编程实战：从Reactor模型到高性能服务端开发

日期： 2020年05月18日

Netty 是当前 Java 生态中最流行的异步事件驱动网络框架，广泛应用于 RPC 框架（Dubbo）、消息中间件（RocketMQ）、即时通讯系统、游戏服务器等高性能网络场景。它对 JDK 原生 NIO 进行了封装和增强，提供了更简洁的 API、更完善的异常处理和更优秀的性能。本文将从 Netty 的核心概念出发，深入剖析 Reactor 模型、零拷贝机制、编解码器设计等关键技术，并通过完整的代码示例展示如何使用 Netty 构建高性能服务端。

Netty 核心要点

线程模型

NioEventLoopGroup 和 EventLoop 都继承或实现了 ScheduledExecutorService、ExecutorService，因此本质上都是线程池。参考：https://www.jianshu.com/p/da4398743b5a

NioEventLoopGroup 还提供了两个重要方法：

  • setIoRatio()：设置 I/O 任务和非 I/O 任务的执行时间比
  • rebuildSelectors()：重建线程中的 Selector，用于规避 JDK 的 epoll 100% CPU Bug

ByteBuffer 选择

Java NIO 支持 HeapByteBuffer 与 DirectByteBuffer，Netty 使用 DirectByteBuffer 实现零拷贝：

ByteBuffer buffer = ByteBuffer.allocate(int capacity);
ByteBuffer buffer = ByteBuffer.allocateDirect(int capacity);

Channel 类型

  • OioServerSocketChannel：同步阻塞 IO 的服务端实现，接受新的客户端连接，并为其创建 OioSocketChannel
  • NioServerSocketChannel：异步非阻塞 IO 的服务端实现（最常用）
  • BIO 的 Stream 是单向的，NIO 的 Channel 是双向的
  • 入站处理器：ChannelInboundHandler、SimpleChannelInboundHandler

Reactor 多种模式

Reactor 模式又有别名 “Dispatcher” 或 “Notifier”。Reactor 是反应堆的意思，它是一种通过一个或多个输入同时传递给服务处理器的事件驱动处理模式。服务端程序处理传入的多路请求，并将它们同步分派给对应的处理线程。

Reactor 模式是编写高性能网络服务器的必备技术之一。Netty 支持灵活配置不同的 Reactor 模式：

package com.crossoverjie.cim.server.server;

import com.crossoverjie.cim.common.constant.Constants;
import com.crossoverjie.cim.common.protocol.CIMRequestProto;
import com.crossoverjie.cim.server.init.CIMServerInitializer;
import com.crossoverjie.cim.server.util.SessionSocketHolder;
import com.crossoverjie.cim.server.vo.req.SendMsgReqVO;
import io.netty.bootstrap.ServerBootstrap;
import io.netty.channel.ChannelFuture;
import io.netty.channel.ChannelFutureListener;
import io.netty.channel.ChannelOption;
import io.netty.channel.EventLoopGroup;
import io.netty.channel.nio.NioEventLoopGroup;
import io.netty.channel.socket.nio.NioServerSocketChannel;
import io.netty.channel.socket.nio.NioSocketChannel;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import org.springframework.beans.factory.annotation.Value;
import org.springframework.stereotype.Component;

import javax.annotation.PostConstruct;
import javax.annotation.PreDestroy;
import java.net.InetSocketAddress;

@Component
public class CIMServer {

    public void start() throws InterruptedException {
        // 单线程Reactor配置
        EventLoopGroup singleEventLoopGroup = new NioEventLoopGroup();
        EventLoopGroup singleEventLoop = singleEventLoopGroup.next();
        ServerBootstrap bootstrapSingle = new ServerBootstrap()
                .group(singleEventLoop, singleEventLoop)
                .channel(NioServerSocketChannel.class)
                .localAddress(new InetSocketAddress(nettyPort))
                //保持长连接
                .childOption(ChannelOption.SO_KEEPALIVE, true)
                // Nagle拆包算法禁用
                .childOption(ChannelOption.TCP_NODELAY, true)
                // sync-recv个数，防止sync-blood攻击
                .childOption(ChannelOption.SO_BACKLOG,1024)
                .childHandler(new CIMServerInitializer());

        // 单Boss线程，多Worker线程，Reactor配置
        EventLoopGroup eventLoopGroup = new NioEventLoopGroup();
        EventLoopGroup bossEventLoop = eventLoopGroup.next();
        EventLoopGroup workLoopGroup = new NioEventLoopGroup();
        ServerBootstrap bootstrapSingle2 = new ServerBootstrap()
                .group(bossEventLoop, workLoopGroup)
                .channel(NioServerSocketChannel.class)
                .localAddress(new InetSocketAddress(nettyPort))
                .childOption(ChannelOption.SO_KEEPALIVE, true)
                .childOption(ChannelOption.SO_BACKLOG,1024)
                .childHandler(new CIMServerInitializer());

        // 多Boss线程，多Worker线程，Reactor配置
        EventLoopGroup boss = new NioEventLoopGroup();
        EventLoopGroup work = new NioEventLoopGroup();
        ServerBootstrap bootstrap = new ServerBootstrap()
                .group(boss, work)
                .channel(NioServerSocketChannel.class)
                .localAddress(new InetSocketAddress(nettyPort))
                .childOption(ChannelOption.SO_KEEPALIVE, true)
                .childOption(ChannelOption.SO_BACKLOG,1024)
                .childHandler(new CIMServerInitializer());

        ChannelFuture future = bootstrap.bind().sync();
        if (future.isSuccess()) {
            LOGGER.info("启动 cim server 成功");
        }
    }
}

三种 Reactor 配置的适用场景不同：单线程适合低并发测试环境，单 Boss 多 Worker 适合大多数生产场景，多 Boss 多 Worker 适合极高并发的场景。

操作系统层面的 Zero-Copy

零拷贝是高性能网络编程的核心技术之一。传统 I/O 操作需要四次数据拷贝和四次上下文切换：

1. ReadBuffer copyTo AppBuffer
2. AppBuffer  copyTo SocketBuffer
3. SocketBuffer  copyTo  NIC Buffer
通过transferTo()，减少了从内核态copyTo用户态，演变为
1. ReadBuffer copyTo SocketBuffer
3. SocketBuffer  copyTo  NIC Buffer
通过sendfile，去除了SocketBuffer，演变为 (Linux 2.1)
1. ReadBuffer copyTo  NIC Buffer
根据socket buffer中的位置和偏移量直接将kernel buffer的数据copy到网卡设备（protocol engine）中，演变为 (Linux 2.4)
1. 根据位置和偏移量，   NIC Buffer，直接从ReadBuffer读取

参考：https://www.jianshu.com/p/017f193663a0

NIO 本身支持 DMA 方式的 Zero-Copy

DMA（Direct Memory Access，直接存储器访问）允许外设直接访问内存而不需要 CPU 参与，进一步减少了数据拷贝的 CPU 开销。

参考：https://mp.weixin.qq.com/s/A8vc7exPDko1k1WaXuO0Og

Netty 零拷贝的三层含义

Netty 的零拷贝主要体现在三个方面：

  1. DirectByteBuffer：发送和接收消息使用堆外内存（DirectMemory），直接通过 DMA 发送到网卡接口，避免了传统堆内存先拷贝到直接内存再写入 Socket 的额外拷贝
  2. CompositeByteBuf：需要组合两个 ByteBuffer 时，传统方式需要创建新数组并拷贝数据。CompositeByteBuf 并没有真正合并 Buffer，而是保存了它们的引用，避免了数据拷贝
  3. FileChannel.transferTo：依赖操作系统实现零拷贝，直接将文件缓冲区的数据发送到目标 Channel

Protocol Buffer 序列化

速度快的原因

  1. 编码/解码方式简单（只需要简单的位移等数学运算）
  2. 采用 Protocol Buffer 自身的框架代码和编译器共同完成

数据压缩效果好的原因

  1. 采用独特的编码方式，如 Varint、Zigzag 编码
  2. 采用 T-L-V（Tag-Length-Value）的数据存储方式，减少了分隔符的使用，数据存储更紧凑

TCP 粘包/拆包问题

产生原因

  • 应用程序写入的字节大小大于套接字发送缓冲区大小，会发生拆包
  • 应用程序写入的数据小于缓冲区大小，网卡将多次写入的数据一起发送，会发生粘包
  • 进行 MSS 大小的 TCP 分段，当 TCP 报文长度减去头部长度大于 MSS 时发生拆包
  • 以太网帧的 payload 大于 MTU（1500 字节）时进行 IP 分片

Netty 提供的解决方案

Netty 提供了四种开箱即用的解码器来处理粘包/拆包问题：

  • FixedLengthFrameDecoder：固定长度，每个包固定长度，不足时补空格
  • LineBasedFrameDecoder / DelimiterBasedFrameDecoder：分隔符方式，在每个包末尾使用固定分隔符（如 \r\n）
  • LengthFieldBasedFrameDecoder：长度字段方式，消息分为头部和消息体，头部保存整个消息的长度

BIO、NIO、AIO 的区别

理解三种 I/O 模型对于选择合适的网络编程方案至关重要：

  • BIO（Blocking I/O）：同步阻塞 I/O，数据读写必须阻塞在一个线程内等待完成。适合活动连接数不高（小于 1000）的场景，编程模型简单。线程池可作为天然漏斗缓冲请求。但面对十万级连接时无能为力。

  • NIO（Non-blocking I/O）：同步非阻塞 I/O，Java 1.4 引入。提供了 Channel、Selector、Buffer 等抽象，支持面向缓冲的、基于通道的 I/O 操作。适合高负载、高并发的网络应用。

  • AIO（Asynchronous I/O）：异步非阻塞 I/O，Java 7 引入（NIO 2）。基于事件和回调机制，操作完成后由操作系统通知相应线程。Windows 下有 IOCP 支持性能优异，Linux 下 JDK 采用自建线程池方式。Netty 曾尝试使用 AIO 但后来放弃。

关键区别：NIO 需要循环 select，当 IO 准备好时由业务线程自行执行 IO 操作（IO 操作本身是同步的）；AIO 通过线程池真正异步化了。

参考：https://www.cnblogs.com/itxiaok/p/10357822.html

为什么选择 Netty 而不是 JDK 原生 NIO？

JDK 原生 NIO 存在以下问题：

  1. API 繁杂：需要熟练掌握 Selector、ServerSocketChannel、SocketChannel、ByteBuffer 等
  2. 额外技能要求高：必须精通 Java 多线程和 Reactor 模式
  3. 可靠性补齐工作量大：断连重连、网络闪断、半包读写、失败缓存、网络拥塞和异常码流等问题都需要自行处理
  4. JDK NIO 的 Epoll Bug：Selector 空轮询导致 CPU 100%，至 JDK 1.7 仍未根本解决

参考：

  • http://www.52im.net/forum.php?mod=viewthread&tid=163&highlight=netty
  • Epoll Bug：http://songkun.me/2019/07/26/2019-07-26-java-nio-epoll-bug-and-netty-solution/

Netty 中的 Epoll 实现

虽然 JDK 自身在 Linux 下提供了基于 epoll 的 Selector 实现，Netty 仍然实现了自己的 epoll 版本，主要原因有两个：

  1. 支持更多 Socket Option：如 TCP_CORK 和 SO_REUSEPORT
  2. 使用边缘触发（ET）模式：ET 只在状态发生变化时触发，比水平触发（LT）理论效率更高，但对使用者要求也更高

Netty 编码实战

EchoServer

下面是一个完整的 Netty 服务端示例，展示了如何配置 Boss/Worker 线程组、绑定端口、设置 Channel 初始化器：

package com.siglea.bobo.imserver.service;

import io.netty.bootstrap.ServerBootstrap;
import io.netty.channel.ChannelFuture;
import io.netty.channel.ChannelInitializer;
import io.netty.channel.EventLoopGroup;
import io.netty.channel.nio.NioEventLoopGroup;
import io.netty.channel.socket.SocketChannel;
import io.netty.channel.socket.nio.NioServerSocketChannel;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.beans.factory.annotation.Value;
import org.springframework.stereotype.Service;

import java.net.InetSocketAddress;

@Service("EchoServer")
public class EchoServer
{
    // 服务器端口
    @Value("${server.port}")
    private int port;
    // 通过nio方式来接收连接和处理连接
    private static EventLoopGroup boss = new NioEventLoopGroup();
    private static EventLoopGroup work = new NioEventLoopGroup();

    // 启动引导器
    private static ServerBootstrap b = new ServerBootstrap();
    @Autowired
    private EchoServerHandler echoServerHandler;

    public void run()
    {
        try
        {
            b.group(boss, work);
            // 设置nio类型的channel
            b.channel(NioServerSocketChannel.class);
            // 设置监听端口
            b.localAddress(new InetSocketAddress(port));
            // 设置通道初始化
            b.childHandler(new ChannelInitializer<SocketChannel>()
            {
                //有连接到达时会创建一个channel
                protected void initChannel(SocketChannel ch) throws Exception
                {
                    // pipeline管理channel中的Handler
                    // 在channel队列中添加一个handler来处理业务
                    ch.pipeline().addLast("echoServerHandler",echoServerHandler);
                }
            });
            // 配置完成，开始绑定server
            // 通过调用sync同步方法阻塞直到绑定成功

            ChannelFuture f = b.bind().sync();
            System.out.println(EchoServer.class.getName() +
                    " started and listen on " + f.channel().localAddress());

            // 监听服务器关闭事件
            // 应用程序会一直等待，直到channel关闭
            f.channel().closeFuture().sync();
        } catch (Exception e)
        {
            e.printStackTrace();
        } finally
        {
            // 关闭EventLoopGroup，释放掉所有资源包括创建的线程
            work.shutdownGracefully();
            boss.shutdownGracefully();
        }

    }
}

EchoServerHandler

服务端的消息处理器，负责接收和回复客户端消息：

package com.siglea.bobo.imserver.service;


import io.netty.buffer.Unpooled;
import io.netty.channel.ChannelFutureListener;
import io.netty.channel.ChannelHandlerContext;
import io.netty.channel.ChannelInboundHandlerAdapter;
import io.netty.util.CharsetUtil;
import io.netty.util.ReferenceCountUtil;
import org.springframework.stereotype.Service;

@Service("echoServerHandler")
public class EchoServerHandler extends ChannelInboundHandlerAdapter
{

    /**
     * 建立连接时，发送一条消息
     */
    @Override
    public void channelActive(ChannelHandlerContext ctx) throws Exception
    {
        System.out.println("连接的客户端地址:" + ctx.channel().remoteAddress());
        super.channelActive(ctx);
    }

    public void channelRead(ChannelHandlerContext ctx, Object msg)
    {
        try
        {
            System.out.println("server received data :" + msg);
//            ctx.write(msg);//写回数据，
            ctx.write(Unpooled.copiedBuffer("I am Server", CharsetUtil.UTF_8));
        } catch (Exception e){
            e.printStackTrace();
        } finally {
            ReferenceCountUtil.release(msg);
        }
    }

    public void channelReadComplete(ChannelHandlerContext ctx)
    {
        //flush掉所有写回的数据
        ctx.writeAndFlush(Unpooled.EMPTY_BUFFER)
                .addListener(ChannelFutureListener.CLOSE); //当flush完成后关闭channel
    }

    public void exceptionCaught(ChannelHandlerContext ctx, Throwable cause)
    {
        //捕捉异常信息
        cause.printStackTrace();
        //出现异常时关闭channel
        ctx.close();
    }
}

EchoClientHandler

客户端的消息处理器，连接建立后发送消息并处理服务端的响应：

package com.siglea.bobo.imclient.service;


import io.netty.buffer.ByteBuf;
import io.netty.buffer.ByteBufUtil;
import io.netty.buffer.Unpooled;
import io.netty.channel.ChannelHandlerContext;
import io.netty.channel.ChannelInboundHandlerAdapter;
import io.netty.util.CharsetUtil;
import io.netty.util.ReferenceCountUtil;
import org.springframework.stereotype.Service;

@Service("echoClientHandler")
public class EchoClientHandler extends ChannelInboundHandlerAdapter
{
    /**
     * 此方法会在连接到服务器后被调用
     */
    public void channelActive(ChannelHandlerContext ctx)
    {

        System.out.println("channelActive");
        ctx.write(Unpooled.copiedBuffer("Netty rocks!", CharsetUtil.UTF_8));
        ctx.flush();
    }

    /**
     * 业务逻辑处理
     */
    @Override
    public void channelRead(ChannelHandlerContext ctx, Object msg) throws Exception
    {
        System.out.println("channelRead");
        // 如果不是protobuf类型的数据
        if (!(msg instanceof ByteBuf))
        {
            System.out.println("未知数据!" + msg);
            return;
        }
        try
        {
            ByteBuf in = (ByteBuf) msg;
            System.out.println("Client received: " +
                    ByteBufUtil.hexDump(in.readBytes(in.readableBytes())));
        } catch (Exception e)
        {
            e.printStackTrace();
        } finally
        {
            ReferenceCountUtil.release(msg);
        }
    }
    /**
     * 捕捉到异常
     */
    public void exceptionCaught(ChannelHandlerContext ctx, Throwable cause)
    {
        System.out.println("exceptionCaught");
        cause.printStackTrace();
        ctx.close();
    }
}

Netty 分布式部署

在分布式场景下，Netty 的部署架构需要解决跨节点消息路由的问题：

零拷贝深入解析

传统拷贝方式

发送数据时，传统实现需要 File.read(bytes) + Socket.send(bytes)，涉及四次数据拷贝和四次上下文切换：

  1. 数据从磁盘读取到内核的 Read Buffer
  2. 数据从内核缓冲区拷贝到用户缓冲区
  3. 数据从用户缓冲区拷贝到内核的 Socket Buffer
  4. 数据从内核的 Socket Buffer 拷贝到网卡接口缓冲区

零拷贝实现

通过 Java 的 FileChannel.transferTo 方法，可以避免第 2、3 步多余的拷贝（需底层操作系统支持）：

  1. 调用 transferTo，数据从文件由 DMA 引擎拷贝到内核 Read Buffer
  2. DMA 从内核 Read Buffer 将数据拷贝到网卡接口 Buffer

以上两次操作都不需要 CPU 参与，达到了零拷贝。

Netty 中的零拷贝

ByteBuffer：使用堆外内存（DirectMemory）进行 Socket 读写。传统堆内存需要先拷贝到直接内存再写入 Socket，DirectMemory 可以直接通过 DMA 发送到网卡。

Composite Buffers：CompositeByteBuf 没有真正将多个 Buffer 组合起来，而是保存了它们的引用，避免了数据拷贝。

FileChannel.transferTo：依赖操作系统实现零拷贝。

在实际生产环境中，TCP 网关通常使用 HAProxy、Nginx 或 LVS 进行负载均衡。但大部分场景还是采用通过 REST 方式获取 IP List，然后由客户端直接发起长连接的方式。

总结

Netty 之所以成为 Java 网络编程的事实标准，在于它在 JDK NIO 基础上做了大量的工程化优化：完善的线程模型、优雅的零拷贝实现、灵活的编解码器架构、健壮的异常处理机制。在实际项目中，合理选择 Reactor 模式、正确处理粘包/拆包、充分利用零拷贝特性，是构建高性能网络服务的关键。
