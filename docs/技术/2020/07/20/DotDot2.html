分布式系统核心问题：副本控制、组件对比与常见面试题精讲

日期： 2020年07月20日

分布式系统的设计本质上是在高可用、一致性、扩展性和性能之间寻找平衡。本文将从分布式副本控制方案出发，对比主流分布式组件（Redis Cluster、HBase、Kafka、Elasticsearch 等）的架构差异，并涵盖 Dubbo、RabbitMQ、Redis、MySQL、Java 并发等方向的高频面试题，是一篇面向有一定基础的技术人员的综合知识梳理。

一、分布式副本控制方案

分布式系统中数据副本的管理方式，决定了系统的可用性和一致性特征。常见的副本控制协议可归纳为三类：

  • 中心化的副本控制协议：即主从模式，所有写请求都经过一个中心节点
  • Primary-Secondary 协议：即 Leader/Follower 模式，数据按切片分段存储，每个分片有自己的主从
  • 去中心化的副本控制协议：如 Gossip 协议或 ZK 选举阶段，不依赖固定的主节点

二、高可用、一致性、扩展性与性能

分布式系统通过存储分片来实现高可用和水平扩展，具体有两种常见模式：

  1. 物理机 Broker 分主从
  2. 物理机 Broker 不分主从，但数据切片的存储分主从

一致性则依赖不同的协议来保证：

  1. 主从同步协议
  2. 去中心化 Gossip 协议

2.1 主流分布式组件对比

评估一个分布式组件的架构，可以从以下维度入手：节点是否对等？写入是否同步副本？写入是代理还是直连？读取是读主还是读副？读取是代理还是直连？扩容是否自动数据迁移？

组件 节点模型 写入方式 读取方式 扩容方式 一致性模型
ZooKeeper Leader/Follower 代理给 Leader 读任意节点，直连 自动数据迁移 选举 CP，广播 AP
Redis Cluster 主备集群，主节点对等 直连主节点（或被 MOVED） 直连主节点 redis-trib.rb CP
HBase 主备集群，RegionServer 对等 通过 ZK 找到目标 RegionServer 通过 ZK 定位 自动 CP
Kafka Leader/Follower（Partition 级） 直连 Partition Leader 读 Leader，直连 kafka-reassign-partitions.sh -
RocketMQ 主/次（物理 Broker） 直连 Master 任意 updateTopic -
Elasticsearch Primary/Replica（Shard 级） 协调节点转发到 Primary Shard 读任意，协调节点协调 自动数据迁移 CA
Cassandra 节点对等 写任意节点（Quorum） 读多节点返回最新数据 自动数据迁移 AP

一些补充说明：

  • ZooKeeper：Follower 有可能数据不是最新的，如果需要最新的可以调用 sync() 方法。ZK 的数据结构是 DataTree，也就是以 path 为 key 的 Map。
  • Kafka：本质是 Partition 数据的 Leader 与 Follower 会分布在不同的 Broker 机器上，不会造成机器资源的不平衡。Kafka Client 通过向任意 Broker 发送 metadata request 来获取 Partition 的 Leader。
  • RocketMQ：物理机 Broker 有主从，物理机是主就是分片 Message 的主，支持异步复制/同步双写。
  • Elasticsearch：此处的 Primary 是指根据 Document 路由而得到的 Primary Shard，不是指单独的机器。
  • Cassandra：一致性 Hash 确定分区，依靠协调节点。

Cassandra 与 Gossip 协议

Cassandra 采用去中心化架构，节点间通过 Gossip 协议进行状态同步。Gossip 协议的核心思想是每个节点随机选择其他节点交换信息，最终所有节点会收敛到一致的状态。

在单机层面，RAID 提供了磁盘级别的冗余和容错能力，是分布式存储的基础设施之一。

三、Dubbo 相关

3.1 Dubbo Monitor 实现原理

Consumer 端在发起调用之前会先走 filter 链；Provider 端在接收到请求时也是先走 filter 链，然后才进行真正的业务逻辑处理。默认情况下，在 Consumer 和 Provider 的 filter 链中都会有 MonitorFilter。其工作流程如下：

  1. MonitorFilter 向 DubboMonitor 发送数据
  2. DubboMonitor 将数据进行聚合后（默认聚合 1 分钟的统计数据）暂存到 ConcurrentMap<Statistics, AtomicReference> statisticsMap，然后使用一个含有 3 个线程（线程名：DubboMonitorSendTimer）的线程池每隔 1 分钟调用 SimpleMonitorService 遍历发送 statisticsMap 中的统计数据，每发送完毕一个，就重置当前的 Statistics 的 AtomicReference
  3. SimpleMonitorService 将这些聚合数据塞入 BlockingQueue queue 中（队列大小为 100000）
  4. SimpleMonitorService 使用一个后台线程（线程名：DubboMonitorAsyncWriteLogThread）将 queue 中的数据写入文件（该线程以死循环的形式来写）
  5. SimpleMonitorService 还会使用一个含有 1 个线程（线程名：DubboMonitorTimer）的线程池每隔 5 分钟将文件中的统计数据画成图表

3.2 Dubbo SPI 和 Java SPI 的区别

JDK SPI 的问题在于会一次性加载所有的扩展实现，如果有的扩展初始化很耗时但又没用上，会造成资源浪费，且无法只加载某个特定实现。

Dubbo SPI 的改进：

  1. 对 Dubbo 进行扩展，不需要改动 Dubbo 的源码
  2. 延迟加载，可以一次只加载自己想要加载的扩展实现
  3. 增加了对扩展点 IOC 和 AOP 的支持，一个扩展点可以直接 setter 注入其它扩展点
  4. Dubbo 的扩展机制能很好的支持第三方 IoC 容器，默认支持 Spring Bean

3.3 Dubbo 如何优雅停机？

Dubbo 是通过 JDK 的 ShutdownHook 来完成优雅停机的，所以如果使用 kill -9 PID 等强制关闭指令，是不会执行优雅停机的，只有通过 kill PID 时，才会执行。

四、消息队列相关

RabbitMQ 消息基于什么传输？

由于 TCP 连接的创建和销毁开销较大，且并发数受系统资源限制，会造成性能瓶颈。RabbitMQ 使用信道（Channel）的方式来传输数据。信道是建立在真实的 TCP 连接内的虚拟连接，且每条 TCP 连接上的信道数量没有限制。

Zookeeper 对于 Kafka 的作用

ZooKeeper 是一个高性能的协调服务，在 Kafka 中主要用于在集群中不同节点之间进行通信。具体来说：

  • 提交偏移量（offset），使得节点失败后能从之前提交的偏移量中恢复
  • Leader 检测
  • 分布式同步
  • 配置管理
  • 识别新节点何时离开或连接
  • 节点实时状态监控

Kafka 消费者手动提交偏移量

将 auto.commit.offset 设为 false，然后在处理一批消息后调用 commitSync()（同步提交）或 commitAsync()（异步提交）。

五、Redis 相关

Pipeline 的好处

可以将多次 IO 往返的时间缩减为一次，前提是 pipeline 执行的指令之间没有因果相关性。使用 redis-benchmark 进行压测时可以发现，影响 Redis QPS 峰值的一个重要因素是 pipeline 批次指令的数目。

Jedis 与 Redisson 对比

Jedis 是 Redis 的 Java 客户端实现，API 提供了比较全面的 Redis 命令支持；Redisson 实现了分布式和可扩展的 Java 数据结构，功能较为简单，不支持字符串操作、排序、事务、管道、分区等 Redis 特性。Redisson 的宗旨是促进使用者对 Redis 的关注分离，从而让使用者能够将精力更集中地放在处理业务逻辑上。

Redis 哈希槽

Redis 集群没有使用一致性 Hash，而是引入了哈希槽的概念。Redis 集群有 16384 个哈希槽，每个 Key 通过 CRC16 校验后对 16384 取模来决定放置哪个槽，集群的每个节点负责一部分 Hash 槽。

如何从 1 亿个 Key 中找出特定前缀的 10 万个 Key？

使用 keys 指令可以扫出指定模式的 Key 列表。但由于 Redis 是单线程的，keys 指令会导致线程阻塞一段时间，线上服务会停顿，直到指令执行完毕。

更好的方案是使用 scan 指令，scan 可以无阻塞地提取出指定模式的 Key 列表，但会有一定的重复概率，在客户端做一次去重即可。整体所花费的时间会比直接用 keys 指令长。

六、MySQL 相关

BLOB 和 TEXT 的区别

BLOB 是一个二进制对象，可以容纳可变数量的数据。TEXT 是一个不区分大小写的 BLOB。两者的唯一区别在于对 BLOB 值进行排序和比较时区分大小写，对 TEXT 值不区分大小写。

最多可以创建多少列索引？

任何标准表最多可以创建 16 个索引列。

NOW() 和 CURRENT_DATE() 的区别

NOW() 显示当前年份、月份、日期、小时、分钟和秒。CURRENT_DATE() 仅显示当前年份、月份和日期。

完整性约束

数据完整性（Data Integrity）分为以下四类：

  1. 实体完整性：规定表的每一行在表中是惟一的实体
  2. 域完整性：表中的列必须满足某种特定的数据类型约束，包括取值范围、精度等
  3. 参照完整性：两个表的主关键字和外关键字的数据应一致，保证表之间数据的一致性
  4. 用户定义的完整性：针对某个特定关系数据库的约束条件，反映某一具体应用必须满足的语义要求

与表有关的约束包括：列约束（NOT NULL）和表约束（PRIMARY KEY、FOREIGN KEY、CHECK、UNIQUE）。

SQL 语句优化方法

  1. Where 子句中，表之间的连接必须写在其他 Where 条件之前，过滤最多记录的条件写在 Where 子句末尾，HAVING 最后
  2. 用 EXISTS 替代 IN、用 NOT EXISTS 替代 NOT IN
  3. 避免在索引列上使用计算
  4. 避免在索引列上使用 IS NULL 和 IS NOT NULL
  5. 对查询进行优化，应尽量避免全表扫描，首先应考虑在 WHERE 及 ORDER BY 涉及的列上建立索引
  6. 应尽量避免在 WHERE 子句中对字段进行 null 值判断
  7. 应尽量避免在 WHERE 子句中对字段进行表达式操作

七、Java 并发相关

活锁与死锁

活锁：任务或者执行者没有被阻塞，由于某些条件没有满足，导致一直重复尝试、失败、尝试、失败。活锁和死锁的区别在于，处于活锁的实体是在不断的改变状态（所谓的”活”），而处于死锁的实体表现为等待；活锁有可能自行解开，死锁则不能。

常见场景：

  • 消息重试：某个消息处理失败后一直重试，但由于消息格式不对导致永远解析失败。解决方案：不可修复的错误不要重试，或限定重试次数。
  • 相互礼让：两个线程彼此响应从而修改自己状态，导致无法执行下去，类似两人在走廊里互相让路。解决方案：选择随机退让，引入随机性。

饥饿

一个或者多个线程因为种种原因无法获得所需要的资源，导致一直无法执行。Java 中导致饥饿的原因：

  1. 高优先级线程吞噬所有的低优先级线程的 CPU 时间
  2. 线程被永久堵塞在一个等待进入同步块的状态
  3. 线程在等待一个本身也处于永久等待完成的对象

解决方法：引入公平锁。

Lock 接口 vs synchronized

Lock 接口比同步方法和同步块提供了更具扩展性的锁操作，其优势包括：

  • 可以使锁更公平
  • 可以使线程在等待锁的时候响应中断
  • 可以让线程尝试获取锁，并在无法获取锁的时候立即返回或者等待一段时间
  • 可以在不同的范围、以不同的顺序获取和释放锁

整体上来说 Lock 是 synchronized 的扩展版，Lock 提供了无条件的、可轮询的（tryLock）、定时的（tryLock 带参）、可中断的（lockInterruptibly）、可多条件队列的（newCondition）锁操作。Lock 的实现类基本都支持非公平锁（默认）和公平锁，synchronized 只支持非公平锁。在大部分情况下，非公平锁是高效的选择。

SynchronizedMap vs ConcurrentHashMap

  • SynchronizedMap 一次锁住整张表来保证线程安全，每次只能有一个线程访问 Map
  • ConcurrentHashMap 使用分段锁来保证多线程下的性能。一次锁住一个桶，默认将 Hash 表分为 16 个桶，原来只能一个线程进入，现在可以同时有 16 个写线程执行
  • ConcurrentHashMap 使用了一种不同的迭代方式：当 iterator 被创建后集合再发生改变不再抛出 ConcurrentModificationException，而是在改变时 new 新的数据，iterator 完成后再将头指针替换为新的数据

为什么 wait()/notify()/notifyAll() 必须在同步块中调用？

当一个线程需要调用对象的 wait() 方法时，必须拥有该对象的锁，接着它会释放这个对象锁并进入等待状态。同样，调用 notify() 时也会释放这个对象的锁。由于所有这些方法都需要线程持有对象的锁，所以只能通过同步来实现。

为什么 sleep() 和 yield() 是静态方法？

这两个方法将在当前正在执行的线程上运行，在其他处于等待状态的线程上调用这些方法是没有意义的。设为静态方法可以避免程序员错误地认为可以在其他非运行线程上调用。

如何确保 main() 方法所在线程最后结束？

使用 Thread 类的 join() 方法，确保所有程序创建的线程在 main() 方法退出前结束。

九、总结

分布式系统的架构设计没有银弹，每种组件都在 CAP 三角中做了不同的取舍。理解这些取舍背后的原理，才能在实际项目中做出合理的技术选型。同时，Java 并发编程的基础知识（锁机制、线程协作、并发容器）是理解分布式系统运行机制的底层支撑，值得反复深入学习。
