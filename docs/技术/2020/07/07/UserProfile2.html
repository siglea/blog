用户画像系统中的标签存储方案详解

日期： 2020年07月07日

在互联网产品日趋精细化运营的今天，用户画像（User Profile）已成为支撑个性化推荐、精准营销和智能风控的核心基础设施。而用户画像的本质，就是将海量用户行为和属性数据抽象为一套结构化的标签体系。标签的生产只是第一步，如何高效地存储、查询和更新这些标签，才是真正考验工程能力的地方。

本文将围绕用户画像标签的存储架构，从数据仓库基础出发，逐步展开标签的分区存储、冗余存储、ID 映射以及多种存储引擎的选型与实践。

数据仓库基础

“数据仓库之父” W.H.Inmon 在《Building the Data Warehouse》一书中定义数据仓库是”一个面向主题的、集成的、非易失的、随时变化的、用来支持管理人员决策的数据集合”。理解这四大特征，是理解标签存储架构设计的前提。

  • 面向主题：业务数据库中的数据主要针对事务处理，各个业务系统之间是相互分离的，而数据仓库中的数据是按照一定主题进行组织的。在用户画像场景下，”用户”本身就是一个核心主题，所有标签都围绕用户维度进行组织。

  • 集成：数据仓库中存储的数据是从业务数据库中提取出来的，但并不是对原有数据的简单复制，而是经过了抽取、清理、转换（ETL）等工作。业务数据库记录的是每一项业务处理的流水账，这些数据不适合直接用于分析处理，进入数据仓库前需要经过一系列计算，同时抛弃一些无关分析处理的数据。

  • 非易失：业务数据库中一般存储短期数据，因此其数据是不稳定的，记录的是数据变化的瞬态。数据仓库中的数据大多表示过去某一时刻的数据，主要用于查询、分析，不像业务系统中的数据库一样经常修改，一般数据仓库构建完成后主要用于访问，不进行修改和删除。

  • 随时变化：数据仓库关注的是历史数据，按时间顺序定期从业务库和日志库里面载入新的数据进行追加，带有时间属性。

对于用户画像系统而言，数据仓库承担着标签计算的底层支撑，原始行为日志经过 ETL 清洗后沉淀到数据仓库，再通过各类规则和模型生产出用户标签。

标签分区存储

在标签存储的物理设计上，通常采用分区表的方式来组织数据。每个标签或标签组作为一个分区，用户作为行维度，形成一张宽表。

在这种存储方案中，标签值通常使用数字来表示：

  • 0：用户没被打上该标签
  • 1：用户被打上该标签，但是没有具体数值
  • 具体数字：用户被打上该标签，有具体数值（如消费金额、活跃天数等）

这种设计的优势在于查询某个用户的所有标签时只需要一次读取，但弊端在于标签维度增多时宽表会变得非常宽，写入性能和存储效率可能下降。实际工程中，通常会按照标签的业务域（如基础属性、行为偏好、消费能力）进行分区，在查询效率和存储效率之间取得平衡。

标签以 UserID 进行冗余存储

为了提升按用户维度查询的效率，标签数据通常会以 UserID 为 key 进行冗余存储。每个用户对应一行记录，将该用户的所有标签聚合在一起。

这种冗余存储是典型的空间换时间策略。在需要获取某个用户完整画像的场景（比如实时推荐、风控决策），这种设计能够将查询从多次 IO 降低到一次，极大提升响应速度。

ID-MAP：多端身份统一

现实世界中，用户的身份是碎片化的：

  • 用户使用不同设备登录系统（手机、PC、平板）
  • 同一设备上可能登录多个用户
  • 可以通过 event_log、page_view_log 等来获取登录时长、登录天数
  • UserID 与 CookieID 通常是多对多的关系

ID-MAP 的核心目标是将同一个自然人在不同平台、不同设备上的 ID 关联起来，形成统一的用户标识。这个过程通常采用图计算或并查集算法，将具有关联关系的 ID 归并到同一个 OneID 下。ID-MAP 的准确性直接影响用户画像的质量——如果 ID 合并出错，标签就会被错误地打到其他人身上。

MySQL 存储元数据管理

标签体系本身也需要管理，包括标签的名称、类型、计算规则、更新频率、所属业务域等。这些元数据通常存储在 MySQL 中，方便通过管理后台进行 CRUD 操作。

元数据管理是标签平台化运营的基础。有了完善的元数据，标签的生产、消费和运维才能形成闭环。

MySQL 存储结果集

对于离线计算产出的标签结果，可以用脚本通过 Sqoop 把 Hive 结果导入 MySQL 中，主要服务离线报表和后台查询场景。但如果需要在线实时访问标签数据，建议使用 HBase 等分布式存储引擎，MySQL 在高并发大数据量场景下的读写性能难以满足需求。

HBase 存储

对于线上实时查询场景，HBase 是用户画像标签存储的常见选择。使用 Spark 将用户标签计算结果同步到 HBase，以 UserID 作为 RowKey，标签作为列族中的列，能够实现毫秒级的随机读写。

HBase 的列族设计非常适合标签这种半结构化数据——不同用户可能拥有不同的标签集合，使用列族存储可以避免传统关系型数据库中大量 NULL 值的问题。同时 HBase 天然支持水平扩展，随着用户量和标签维度的增长，只需增加 RegionServer 即可。

Elasticsearch 作为 HBase 的二级索引

HBase 的 RowKey 查询非常高效，但如果需要按标签值反查用户（如”查询所有消费金额大于 1000 的用户”），HBase 就力不从心了。此时引入 Elasticsearch 作为 HBase 的二级索引是一种成熟的实践方案。

ES 负责索引标签维度的数据，支持复杂的组合查询和聚合分析；查询到目标用户 ID 后，再回 HBase 获取完整的画像信息。这种 ES + HBase 的组合架构兼顾了灵活查询和海量存储的需求。

全链路数据监控

从 ETL 到 Hive、再到 HBase 和 Elasticsearch，整个数据链路涉及多个环节，每个环节都可能出现数据质量问题。脏数据和旧数据的危害不容小觑——一条错误的标签可能导致推荐结果偏差、营销资源浪费甚至用户投诉。

完善的数据监控体系应该覆盖：

  • ETL 阶段：数据量监控、字段空值率、格式校验
  • Hive 阶段：标签覆盖率、标签分布变化、数据新鲜度
  • HBase/ES 同步阶段：数据一致性校验、同步延迟监控
  • 应用层：标签命中率、查询超时率、异常值告警

总结

用户画像的标签存储并非简单的”写入数据库”，而是一个涉及数据仓库设计、多种存储引擎选型、ID 映射、索引优化和数据质量监控的系统工程。核心要点包括：

  1. 以数据仓库为基础，通过 ETL 清洗和加工原始数据
  2. 采用分区存储和 UserID 冗余存储平衡查询效率
  3. 通过 ID-MAP 实现多端身份统一
  4. MySQL 管理元数据，HBase 服务线上实时查询
  5. Elasticsearch 作为二级索引支持灵活的标签组合查询
  6. 全链路数据监控保障标签质量

选择合适的存储方案没有银弹，关键在于根据业务场景的实时性要求、数据规模和查询模式来做权衡。
