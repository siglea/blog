Java核心知识点精讲：并发编程、集合框架与JVM底层机制

日期： 2020年07月08日

Java 作为企业级开发的主力语言，其核心知识体系覆盖了并发编程、集合框架、JVM 内存管理、Spring 生态等多个领域。本文将这些高频且重要的知识点进行系统梳理和深入解析，包括 ArrayList 的扩容机制、volatile 与 CAS 的并发原语、JVM 锁优化策略、HashMap 底层设计、NIO 编程要点等，适合有一定 Java 基础的开发者巩固和提升核心能力。

集合框架

ArrayList 的扩容机制

ArrayList 底层是一个动态数组，其扩容行为是面试中的高频话题。

初始化阶段：ArrayList 首先判断传入的初始容量参数 initialCapacity——如果等于 0，则初始化为空数组；如果不等于 0，则初始化为指定容量的数组（默认首次添加元素时容量为 10）。

扩容时机：当数组大小超过当前容量时（比如初始容量为 10，添加第 11 个元素时），触发扩容。新容量为旧容量的 1.5 倍（Vector 则是 2 倍）。

扩容方式：以新容量创建原数组的拷贝，修改原数组指向新数组，旧数组被 GC 回收。

ArrayList 与 Vector 的详细对比

  • ArrayList 创建时默认大小为 0，首次添加元素时扩容到 10；Vector 创建时默认大小为 10
  • ArrayList 每次扩容 1.5 倍；Vector 每次扩容 2 倍（指定 capacityIncrement 后，每次增加 capacityIncrement 个单位）
  • add、get、size 方法复杂度都为 O(1)，remove 方法复杂度为 O(n)
  • ArrayList 是非线程安全的，Vector 是线程安全的

HashMap 为什么初始容量是 2 的幂次方？

这是一个精妙的设计决策，涉及三个层面：

  1. 位运算优化：HashMap 确定元素落在数组哪个位置时，计算方式是 hash % n（类似数据库分表）。但取模运算消耗较大，可以用 (n - 1) & hash 的位运算替代，前提是 n 必须是 2 的幂。
  2. 减少哈希冲突：如果初始容量是奇数，(n-1) 就为偶数，偶数的二进制末尾都是 0，经过 & 运算后末尾固定为 0，导致大量位置永远无法存放元素，空间浪费且增加哈希冲突。
  3. 扩容效率：2 的幂次方乘以 2 在二进制中只是高位前移一位（如 4=00000100 和 8=00001000），扩容时只需判断高位 hash，移动到之前位置的倍数即可，免去重新计算位置的运算。

并发编程

volatile 关键字

volatile 提供三个保证：保证可见性、禁止指令重排、但无法实现原子性。

指令重排只保证串行语义的执行一致性（单线程），并不关心多线程间的语义一致性。volatile 通过内存屏障（Memory Barrier）禁止了特定类型的指令重排，确保多线程环境下的可见性。

CAS 与 Unsafe

CAS（Compare-And-Swap） 是一条 CPU 并发原语，判断内存某个位置的值是否为预期值，如果是则更改为新值，整个过程是原子的。

CAS 在 Java 中体现为 sun.misc.Unsafe 类中的各个方法，JVM 会将其编译为 CAS 汇编指令。这是一种完全依赖于硬件的功能，原语的执行必须是连续的，在执行过程中不允许被中断。

CAS 的自旋机制：

# 我们可以看到getAndAddInt方法执行时，有个do while
# unsafe.getAndAddInt
public final int getAndAddInt(Object var1, long var2, int var4){
    int var5;
    do {
        var5 = this.getIntVolatile(var1, var2);
    } while(!this.compareAndSwapInt(var1, var2, var5, var5 + var4));
    return var5;
}

# 如果CAS失败，会一直进行尝试。如果CAS长时间一直不成功，可能会能CPU带来很大的开锁。

CAS 的局限性：只能保证一个共享变量的原子性。

Exchanger：线程间数据交换

Exchanger 允许在并发任务之间交换数据。当两个线程都到达同步点时，它们交换数据结构——第一个线程的数据进入第二个线程，反之亦然。

典型使用场景：

  • 线程 A 往队列中存入数据，线程 B 从队列中消费数据
  • 线程 A 存满时才交换给线程 B，线程 B 消费完成后才交换给线程 A
  • 双方生产和消费速率可能不同
  • 对方线程调用 exchange 之前，当前线程执行到 exchange 会阻塞

JVM 锁优化

JVM 对原生锁做了哪些优化？

自旋锁：在将线程阻塞前先让其自旋等待一段时间，如果在等待期间其他线程已解锁，就无需阻塞操作，避免了用户态到内核态的切换。

三种 Monitor 实现（锁的升级路径）：

  • 偏向锁（Biased Locking）：默认情况。JVM 利用 CAS 操作在对象头 Mark Word 部分设置线程 ID，表示该对象偏向当前线程。不涉及真正的互斥锁，因为大部分对象生命周期中最多被一个线程锁定。
  • 轻量级锁：如果有另一线程试图锁定被偏向过的对象，JVM 撤销偏向锁，升级为轻量级锁。依赖 CAS 操作 Mark Word 试图获取锁。
  • 重量级锁：如果轻量级锁 CAS 重试失败，进一步升级为重量级锁。

JVM 检测到不同的竞争状况时会自动切换到适合的锁实现，这就是锁的升级、降级。

ReadWriteLock 源码分析

读写锁的核心思想是读读共享、读写互斥、写写互斥，通过将 state 变量的高 16 位记录读锁、低 16 位记录写锁来实现：

final boolean tryWriteLock() {
            Thread current = Thread.currentThread();
            int c = getState();
            if (c != 0) {
                int w = exclusiveCount(c);
                if (w == 0 || current != getExclusiveOwnerThread())
                    return false;
                if (w == MAX_COUNT)
                    throw new Error("Maximum lock count exceeded");
            }
            if (!compareAndSetState(c, c + 1))
                return false;
            setExclusiveOwnerThread(current);
            return true;
        }

                    }

面向对象设计原则（SOLID）

这五大原则是面向对象设计的基石，遵循这些原则可以写出更加灵活、可维护的代码：

  • S — 单一职责原则（SRP）：一个类最好只做一件事，只有一个引起它变化的原因。可以看作低耦合、高内聚在面向对象中的引申。
  • O — 开放封闭原则（OCP）：软件实体应该是可扩展的，而不是可修改的。对扩展开放，对修改封闭。
  • L — 里氏替换原则（LSP）：子类必须能够替换其基类。这是保证继承复用的基础。
  • I — 接口隔离原则（ISP）：使用多个小的接口，而不是一个大的总接口。
  • D — 依赖倒置原则（DIP）：依赖于抽象。高层模块不依赖于底层模块，二者共同依赖于抽象。

NIO 的空轮询 Bug

这是 Java NIO 中一个著名的 Bug：selector.select() 应该一直阻塞直到有就绪事件到达，但由于实现上的缺陷，select() 可能在没有任何就绪事件的情况下返回，导致 while(true) 被不断执行，最终某个 CPU 核心利用率飙升到 100%。

问题代码：

while (true) {
    selector.select();  // select 可能在无就绪事件时异常返回！主要发生在linux系统

    Set<SelectionKey> readyKeys = selector.selectedKeys();
    Iterator<SelectionKey> it = readyKeys.iterator();

    while (it.hasNext()) {
        SelectionKey key = it.next();
        ...  // 处理事件
        it.remove();
    }
}

Netty 的解决方案——通过计数检测空轮询并重建 Selector：

selectCnt = 0;  // epoll 空轮询场景下 select 调用次数
long currentTimeNanos = System.nanoTime();  // 每个 for 循环开始时的绝对时间

for (;;) {
    timeoutMillis = ...  // 初始化超时参数

    int selectedKeys = selector.select(timeoutMillis);
    selectCnt++;

    long time = System.nanoTime();  // 记录执行到此处的绝对时间：

    // 检测逻辑
    if (time - currentTimeNanos > timeoutMillis) {
        selectCnt = 1;  // 未发生 epoll 空轮询，所以把 selectCnt 重置为 1
    } else if (selectCnt >= 重试次数阈值（默认 512）) {
        selector = selectRebuildSelector(selectCnt);  // 解决 epoll bug 的实际逻辑
        selectCnt = 1;  // 解决本次 epoll bug，重置 selectCnt
        break;
    }

    currentTimeNanos = time;  // 重置下次 for 循环开始时间
}

零拷贝（Zero Copy）

传统 I/O 的拷贝问题

在传统 I/O 操作中，每次都需要把内核空间的数据拷贝到用户空间，效率低下。零拷贝就是为了解决这个问题。

两种实现方式

mmap + write：将文件映射到进程的地址空间，省去了内核 Read 缓冲区到用户缓冲区的拷贝，但仍需要内核 Read 缓冲区到内核 Socket 缓冲区的拷贝。

sendfile：不仅减少了数据拷贝，还减少了上下文切换次数。数据传送只发生在内核空间里。Linux 2.4 后进一步优化，将 Kernel Buffer 中的位置和偏移量信息记录到 Socket Buffer，NIC Buffer 直接从 Read Buffer 读取，彻底消除了内核空间的 CPU Copy。

Netty 的零拷贝体现

  1. DirectByteBuffer：使用堆外直接内存进行 Socket 读写，不需要字节缓冲区的二次拷贝
  2. CompositeChannelBuffer：操作多个 Channel 时不开辟新内存，直接保存所有 ChannelBuffer 的引用，在子 ChannelBuffer 里进行读写
  3. transferTo 方法：直接将文件缓冲区的数据发送到目标 Channel

内存管理

内存泄漏的常见原因

  1. 内部类持有外部类引用
  2. 静态集合类
  3. 各种连接的打开与关闭（数据库、IO、网络连接）
  4. 变量作用域不合理（类似逃逸）
  5. 修改了 Hash 算法导致之前的 hash key 对应的 value 不能被释放
  6. ThreadLocal 内存泄漏：内部 Map 的 key 是通过虚引用关联的 ThreadLocal，GC 时 key 消失导致 Value 泄漏。使用时记得显式调用 remove()。参考：https://www.jianshu.com/p/1a5d288bdaee
  7. 监听器的回调绑定与取消

Direct Memory 回收机制

DirectByteBuffer 通过虚引用（Phantom Reference）实现堆外内存的释放。Cleaner 是 PhantomReference 的子类，通过双向链表维护。当 DirectByteBuffer 对象从 pending 状态到 enqueue 状态时，触发 Cleaner 的 clean() 方法，通过 Unsafe 释放堆外内存。

        try {
            base = unsafe.allocateMemory(size);
        } catch (OutOfMemoryError x) {
            Bits.unreserveMemory(size, cap);
            throw x;
        }
        unsafe.setMemory(base, size, (byte) 0);
        if (pa && (base % ps != 0)) {
            // Round up to page boundary
            address = base + ps - (base & (ps - 1));
        } else {
            address = base;
        }
        // DirectByteBuffer 内 构建Cleaner对象用于跟踪DirectByteBuffer对象的垃圾回收，以实现当DirectByteBuffer被垃圾回收时，堆外内存也会被释放
        cleaner = Cleaner.create(this, new Deallocator(base, size, cap));
        // Cleaner 内 clean
        cleaner.clean()
        // Deallocator 继承 Runnable
        public void run() {
            if (address == 0) {
                // Paranoia
                return;
            }
            unsafe.freeMemory(address);
            address = 0;
            Bits.unreserveMemory(size, capacity);
        }

Spring 生态

SpringBoot Starter 自动装配原理

SpringBoot 的自动装配机制分三步：

  1. 引入该 Starter 的相关依赖
  2. 查找 resources/META-INF/spring.factories 下对应的 XXXAutoConfiguration
  3. 根据 Condition 注解自动装配相关 Bean

常用的条件注解：

  • @ConditionalOnBean：容器中有指定 Bean 时
  • @ConditionalOnClass：类路径下有指定类时
  • @ConditionalOnExpression：基于 SpEL 表达式判断
  • @ConditionalOnJava：基于 JVM 版本判断
  • @ConditionalOnMissingBean：容器中没有指定 Bean 时
  • @ConditionalOnMissingClass：类路径下没有指定类时
  • @ConditionalOnProperty：指定属性是否有指定值
  • @ConditionalOnResource：类路径下是否有指定资源
  • @ConditionalOnWebApplication：当前项目是 Web 项目时
  • @ConditionalOnNotWebApplication：当前项目不是 Web 项目时

Spring 循环依赖与三级缓存

Spring 使用三级缓存解决循环依赖问题。Bean 创建分三步：

  1. 初始化 init：分配内存空间，此时在 singletonFactories（三级缓存）完成
  2. 注入属性：此时在 earlySingletonObjects（二级缓存）中
  3. 完成创建：最终会在 singletonObjects（一级缓存）中

循环依赖解决过程（以 A 和 B 互相依赖为例）：

  1. A 初始化经过第 1、2 步，在第 2 步发现依赖 B
  2. 触发 B 的初始化进入第 1 步，B 在第 2 步发现依赖 A，此时 B 只能在第 1 步
  3. 把 B 第 1 步的不完美引用注入给 A，让 A 先完成注入进入第 3 步
  4. 最后完成 B 的完美初始化

编码与字符集

UTF-8 变长编码规则

  • 单字节符号：第一位设为 0，后面 7 位为 Unicode 码。英语字母的 UTF-8 与 ASCII 码相同。
  • n 字节符号（n>1）：第一个字节的前 n 位都设为 1，第 n+1 位设为 0，后面字节的前两位一律设为 10。例如：1110xxxx 10xxxxxx 10xxxxxx（3 字节，xxx 放入 Unicode）

字节序

Unicode 规范中每个文件最前面加入 BOM（Byte Order Mark），即”零宽度非换行空格”（ZERO WIDTH NO-BREAK SPACE，FEFF）：头两个字节为 FE FF 表示大端，FF FE 表示小端。

Redis 与 Memcached 的区别

  1. Redis 不仅支持 k/v，还支持 list、set、hash 等数据结构
  2. Redis 支持 master-slave 模式的数据备份
  3. Redis 支持数据持久化（RDB/AOF），可将内存数据保持到磁盘
  4. Redis 并非所有数据都一直存储在内存中（这是与 Memcached 最大的区别）
  5. Memcached 可以存储大文件
